# 그린 필드 프로젝트 (Greenfield Project)

**한 줄 요약:** 아무것도 없는 빈 땅에서 새로 짓는 프로젝트.

## 비유로 이해하기

12살도 이해할 수 있게 비유하면:

- **그린필드** = 아무것도 안 지어진 **초록색 빈 잔디밭**에 새 건물을 짓는 것
- **브라운필드** = 이미 낡은 건물이 있는 **갈색 공터**를 리모델링하거나 그 옆에 증축하는 것

## 핵심 특징

| 구분        | 그린필드 (Greenfield)        | 브라운필드 (Brownfield)            |
| ----------- | ---------------------------- | ---------------------------------- |
| **시작점**  | 제로에서 시작                | 기존 시스템 위에서 시작            |
| **레거시**  | 없음                         | 있음 (호환성 고려 필요)            |
| **기술 선택** | 자유롭게 선택 가능          | 기존 스택에 제약받음               |
| **설계 자유도** | 높음                      | 낮음                               |
| **리스크**  | 검증 안 된 설계              | 기술 부채 상속                     |
| **초기 속도** | 빠름 (제약 없으니까)        | 느림 (이해부터 해야 하니까)        |

## 그린필드의 장점

- 최신 기술 스택을 자유롭게 채택 가능
- 클린 아키텍처를 처음부터 적용 가능
- 기술 부채 없이 깨끗한 출발
- 팀이 원하는 패턴/컨벤션을 정할 수 있음

## 그린필드의 함정

> "자유가 많다 = 결정할 게 많다"

- **오버엔지니어링 위험**: "나중에 필요할 것 같으니까" 하고 과도하게 설계
- **검증 안 된 기술 도입**: 새 거니까 새 기술 쓰자 → 삽질
- **과소평가**: 레거시가 없으니 쉬울 거라 착각 → 도메인 복잡성은 여전함
- **Second System Effect**: 기존 시스템의 모든 문제를 한번에 해결하려다 더 복잡해짐

## 실무에서의 판단 기준

```
그린필드를 선택해야 할 때:
├── 기존 시스템이 없거나 완전히 다른 도메인
├── 레거시 시스템의 기술 부채가 리팩토링 비용 > 재구축 비용
├── 비즈니스 요구사항이 기존 아키텍처로 수용 불가
└── 팀이 새로운 기술 스택에 충분한 역량 보유

브라운필드를 선택해야 할 때:
├── 기존 시스템이 동작하고 있고 점진적 개선 가능
├── 기존 데이터/사용자 마이그레이션 비용이 큼
├── 리스크를 최소화해야 하는 상황
└── Strangler Fig 패턴으로 점진 교체 가능
```

## 아키텍처 관점에서의 의미

그린필드에서는 **모든 아키텍처 결정이 열려 있기 때문에**, 오히려 더 신중한 설계가 필요합니다. 이때 이 프로젝트(Demiurge)의 아키텍트 오케스트레이션 같은 체계적인 리뷰 프로세스가 특히 효과적입니다 — 잘못된 초기 설계는 프로젝트 전체의 방향을 결정하니까요.
