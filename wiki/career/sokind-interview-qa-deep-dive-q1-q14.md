FastAPI에서 `exception_handler` 데코레이터로 등록하는 것과, Spring에서 `@RestControllerAdvice`로 등록하는 것은 **같은 패턴의 다른 문법**이야.\n\n\"전역 예외\"라는 표현이 좀 오해를 줄 수 있는데:\n- ❌ \"비즈니스 로직이 아닌 예외만\" 이라는 뜻이 아니야\n- ✅ **\"어떤 컨트롤러에서든 발생하는 예외를 한 곳에서 처리\"** 라는 뜻이야\n\n비즈니스 예외(`InsufficientBalanceException` 같은)도 전역 핸들러에서 처리하는 게 일반적이야.\n\n---\n\n### Q3-2: @ControllerAdvice vs @RestControllerAdvice\n\n#### 현실 세계 비유\n\n- `@ControllerAdvice` = 학교 **교무실** (문제를 받아서 처리하고, 처리 결과를 **편지**(View)로 줄 수도 있고 **말**(JSON)로 할 수도 있어)\n- `@RestControllerAdvice` = 학교 교무실인데 **항상 말(JSON)로만 대답**하는 규칙\n\n#### 단계별 설명\n\n**Step 1: @ControllerAdvice**\n\n```java\n@ControllerAdvice\npublic class GlobalExceptionHandler {\n    \n    @ExceptionHandler(NotFoundException.class)\n    public String handleNotFound(Model model) {\n        model.addAttribute(\"message\", \"페이지를 찾을 수 없습니다\");\n        return \"error/404\";  // → View(HTML)를 반환\n    }\n    \n    @ExceptionHandler(BusinessException.class)\n    @ResponseBody  // 이걸 붙여야 JSON 반환\n    public ErrorResponse handleBusiness(BusinessException e) {\n        return new ErrorResponse(e.getCode(), e.getMessage());\n    }\n}\n```\n\n- 기본적으로 **View 이름을 반환**하는 것으로 간주\n- JSON을 반환하려면 메서드마다 `@ResponseBody`를 붙여야 해\n\n**Step 2: @RestControllerAdvice**\n\n```java\n@RestControllerAdvice  // = @ControllerAdvice + @ResponseBody\npublic class GlobalExceptionHandler {\n    \n    @ExceptionHandler(NotFoundException.class)\n    public ErrorResponse handleNotFound(NotFoundException e) {\n        return new ErrorResponse(\"NOT_FOUND\", e.getMessage());\n        // 자동으로 JSON 변환! @ResponseBody 불필요\n    }\n}\n```\n\n- **모든 메서드에 `@ResponseBody`가 자동 적용**\n- REST API 서버라면 거의 100% 이걸 씀\n\n| 비교 | @ControllerAdvice | @RestControllerAdvice |\n|------|:---:|:---:|\n| 반환 기본 동작 | View 이름 | JSON/XML (직렬화) |\n| @ResponseBody 필요? | ✅ 메서드마다 | ❌ 자동 적용 |\n| 용도 | MVC (HTML 반환) | REST API (JSON 반환) |\n| 관계 | 부모 | `@ControllerAdvice` + `@ResponseBody` |\n\n💡 이건 `@Controller` vs `@RestController` 관계와 **완전히 동일**해.\n\n---\n\n### Q3-3: @ExceptionHandler의 역할\n\n✅ **맞아, 정확한 이해야!**\n\n`@ExceptionHandler`는 **\"이 메서드가 어떤 예외를 처리하는 녀석인지\"를 표시하는 annotation**이야.\n\n```java\n@RestControllerAdvice\npublic class GlobalExceptionHandler {\n\n    @ExceptionHandler(BusinessException.class)      // ← \"나는 BusinessException 담당이야\"\n    public ResponseEntity<ErrorResponse> handleBusiness(BusinessException e) {\n        // ...\n    }\n\n    @ExceptionHandler(NotFoundException.class)       // ← \"나는 NotFoundException 담당이야\"\n    public ResponseEntity<ErrorResponse> handleNotFound(NotFoundException e) {\n        // ...\n    }\n    \n    @ExceptionHandler({IOException.class, TimeoutException.class})  // ← 여러 개도 가능\n    public ResponseEntity<ErrorResponse> handleIO(Exception e) {\n        // ...\n    }\n}\n```\n\n비유하면:\n- `@RestControllerAdvice` = **교무실** (예외를 모아서 처리하는 장소)\n- `@ExceptionHandler` = 교무실 안의 **담당 선생님 이름표** (\"나는 싸움 담당\", \"나는 지각 담당\")\n\n참고: `@ExceptionHandler`는 `@RestControllerAdvice` 없이도 **개별 Controller 안에서** 쓸 수 있어. 그러면 해당 Controller에서 발생한 예외만 처리해.\n\n```java\n@RestController\npublic class UserController {\n    \n    @ExceptionHandler(UserNotFoundException.class)  // 이 Controller 안에서만 동작\n    public ErrorResponse handleUserNotFound(UserNotFoundException e) {\n        // ...\n    }\n}\n```\n\n---\n\n### Q3-4: HTTP Status Code만으로 충분하지 않은가?\n\n좋은 질문이야! 이건 **규모에 따라 답이 달라져.**\n\n#### 현실 세계 비유\n\n병원에 갔는데:\n- **HTTP Status Code만** = 의사가 \"아파요\" 라고만 적어줌 (400 = 잘못 요청, 404 = 못 찾음)\n- **도메인 에러 코드** = 의사가 \"위염 (A01)\", \"식도역류 (A02)\" 처럼 구체적 병명을 적어줌\n\n둘 다 \"배가 아프다(400)\"는 같은데, **뭐가 문제인지** 세부 정보가 다르지.\n\n#### 왜 HTTP Status Code만으로는 부족한가?\n\n```\n상황 1: 잔액 부족으로 결제 실패\n상황 2: 일일 한도 초과로 결제 실패\n상황 3: 카드가 만료되어 결제 실패\n```\n\n셋 다 HTTP Status Code는 **400 Bad Request** 또는 **422 Unprocessable Entity**야. 클라이언트가 이걸 구분할 방법이 없어.\n\n```json\n// HTTP Status Code만\n{ \"status\": 400, \"message\": \"결제 실패\" }  // 뭐가 문제인데??\n\n// 도메인 에러 코드 포함\n{ \n  \"status\": 400, \n  \"code\": \"PAYMENT_001\",       // 잔액 부족\n  \"message\": \"잔액이 부족합니다\",\n  \"detail\": \"현재 잔액: 1,000원, 필요 금액: 5,000원\"\n}\n```\n\n#### 판단 기준 테이블\n\n| 상황 | HTTP Status만으로 충분? | 도메인 코드 필요? |\n|------|:---:|:---:|\n| 단순 CRUD API | ✅ | ❌ |\n| 내부 마이크로서비스 통신 | ✅ | ⚠️ 선택 |\n| 결제/금융 API | ❌ | ✅ |\n| Public API (외부 개발자용) | ❌ | ✅ |\n| 클라이언트가 에러별 분기 처리 필요 | ❌ | ✅ |\n| 에러 모니터링/대시보드 필요 | ❌ | ✅ |\n\n💡 **네 말도 맞아.** 간단한 서비스에서는 HTTP Status Code만으로 충분할 수 있어. 하지만 서비스가 커지면:\n\n1. **같은 400이라도 원인이 다양** → 클라이언트가 에러별로 다른 UI를 보여줘야 함\n2. **에러 추적** → \"PAYMENT_001이 급증했다\"가 \"400이 많다\"보다 유용\n3. **다국어 지원** → 에러 코드로 클라이언트가 적절한 메시지를 매핑\n\n---\n\n### Q3-5: 4xx가 WARNING인 이유, ERROR가 될 수도 있나?\n\n#### 현실 세계 비유\n\n식당에서:\n- **4xx (WARNING)** = 손님이 메뉴에 없는 걸 주문함 → 서버(웨이터) 잘못이 아님, \"없는 메뉴입니다\" 하면 됨\n- **5xx (ERROR)** = 주방에서 불이 남 → 식당 자체 문제, 즉시 대응 필요\n\n#### 왜 기본적으로 4xx = WARNING인가?\n\n```\n4xx = 클라이언트의 실수 → 서버는 정상 동작 중\n5xx = 서버의 문제 → 서버에 장애가 있음\n```\n\n로깅의 핵심 목적은 **\"우리 시스템에 문제가 있는가?\"**를 파악하는 거야. 4xx는 클라이언트가 잘못한 거니까, 서버 운영팀이 긴급 대응할 필요가 없어.\n\n#### ✅ 하지만 네 말도 맞아! 4xx가 ERROR가 되어야 하는 경우\n\n| 상황 | 로그 레벨 | 이유 |\n|------|:---:|------|\n| 일반적인 400 (잘못된 입력) | ⚠️ WARN | 클라이언트 실수, 서버 정상 |\n| 일반적인 404 (없는 페이지) | ⚠️ WARN | 클라이언트가 잘못된 URL 요청 |\n| 401/403이 **갑자기 급증** | ❌ ERROR | 보안 공격 가능성 → 즉시 확인 필요 |\n| 내부 서비스 간 통신에서 4xx | ❌ ERROR | 내부 서비스는 잘못된 요청을 보내면 안 됨 → 버그 |\n| 429 (Rate Limit) 급증 | ❌ ERROR | DDoS 공격이거나 클라이언트 구현 버그 |\n| 404인데 **있어야 하는 리소스** | ❌ ERROR | 데이터 정합성 문제일 수 있음 |\n\n```java\n@ExceptionHandler(BusinessException.class)\npublic ResponseEntity<ErrorResponse> handle(BusinessException e) {\n    if (e instanceof SecurityRelatedException) {\n        log.error(\"보안 관련 예외 발생: {}\", e.getMessage());  // ERROR!\n    } else {\n        log.warn(\"비즈니스 예외: {}\", e.getMessage());         // WARN\n    }\n    return ResponseEntity.status(e.getStatus()).body(...);\n}\n```\n\n💡 **결론**: \"4xx = WARN\"은 **기본값(default)**이지 **절대 규칙**이 아니야. 상황에 따라 동적으로 로그 레벨을 조절하는 게 좋은 설계야. 특히 **보안 관련 4xx**와 **내부 서비스 간 4xx**는 ERROR로 격상하는 게 맞아.\n\n---\n\n### Q3-6: BusinessException, MethodArgumentNotValidException — 기본 제공?\n\n| 예외 | Spring 기본 제공? | 설명 |\n|------|:---:|------|\n| `MethodArgumentNotValidException` | ✅ 기본 제공 | `@Valid` 검증 실패 시 Spring이 자동으로 던짐 |\n| `BusinessException` | ❌ 직접 만들어야 함 | 관례적 이름이지, Spring에 없음 |\n\n#### 상세 설명\n\n**MethodArgumentNotValidException** — Spring 기본 제공 ✅\n\n```java\n// Controller에서 @Valid를 쓰면\n@PostMapping(\"/users\")\npublic User createUser(@Valid @RequestBody CreateUserRequest req) { ... }\n\n// CreateUserRequest의 유효성 검증 실패 시\n// Spring이 자동으로 MethodArgumentNotValidException을 던짐\npublic class CreateUserRequest {\n    @NotBlank String name;      // null이면 → 예외 발생\n    @Email String email;         // 이메일 형식 아니면 → 예외 발생\n    @Min(0) Integer age;         // 음수면 → 예외 발생\n}\n```\n\n이건 `spring-boot-starter-validation` (Hibernate Validator) 의존성에서 나와. 패키지는 `org.springframework.web.bind`.\n\n**BusinessException** — 직접 만들어야 함 ❌\n\n```java\n// 이건 개발자가 직접 정의하는 커스텀 예외\npublic class BusinessException extends RuntimeException {\n    private final ErrorCode errorCode;\n    \n    public BusinessException(ErrorCode errorCode) {\n        super(errorCode.getMessage());\n        this.errorCode = errorCode;\n    }\n}\n\n// 이런 식으로 확장\npublic class InsufficientBalanceException extends BusinessException {\n    public InsufficientBalanceException() {\n        super(ErrorCode.INSUFFICIENT_BALANCE);\n    }\n}\n```\n\nSpring에서 **기본 제공하는 주요 예외들**을 정리하면:\n\n| 예외 | 발생 상황 |\n|------|-----------|\n| `MethodArgumentNotValidException` | `@Valid` 검증 실패 |\n| `HttpRequestMethodNotSupportedException` | 허용되지 않은 HTTP 메서드 (405) |\n| `HttpMediaTypeNotSupportedException` | 지원하지 않는 Content-Type (415) |\n| `MissingServletRequestParameterException` | 필수 파라미터 누락 (400) |\n| `NoHandlerFoundException` | 매핑된 핸들러 없음 (404) |\n| `ResponseStatusException` | 개발자가 상태코드 지정 (다양) |\n| `AccessDeniedException` | Spring Security 권한 부족 (403) |\n\nFastAPI 비교:\n\n| Spring Boot | FastAPI |\n|---|---|\n| `MethodArgumentNotValidException` | `RequestValidationError` (Pydantic) |\n| `BusinessException` (커스텀) | 커스텀 Exception class |\n| `ResponseStatusException` | `HTTPException` |\n\n---\n\n## Q4. API 버전 관리 전략\n\n### Q4-1: URL 오염과 라우팅 복잡성\n\n#### URL 오염 (URL Pollution)\n\n현실 비유: 집 주소에 **불필요한 정보가 계속 추가**되는 것.\n\n```\n정상: 서울시 강남구 테헤란로 123\n오염: 서울시 강남구 테헤란로 123 (2024년도버전) (우편번호변경이력있음)\n```\n\nAPI에서:\n\n```\n/v1/users/123\n/v2/users/123\n/v3/users/123\n```\n\n**리소스 자체는 같은 User인데**, URL이 달라져. REST 원칙에서 **하나의 리소스는 하나의 URI**로 식별되어야 하는데, 같은 리소스에 여러 URI가 생기는 거야. 이걸 \"URL이 오염됐다\"고 표현해.\n\n추가 문제:\n- 문서화할 때 모든 버전의 URL을 관리해야 함\n- 클라이언트가 \"어떤 버전 쓰지?\" 혼란\n- 버전이 쌓이면 URL이 지저분해짐\n\n#### 라우팅 복잡성\n\n```java\n// 버전이 3개면 같은 로직에 라우팅이 3벌\n@GetMapping(\"/v1/users/{id}\")  // v1 로직\n@GetMapping(\"/v2/users/{id}\")  // v2 로직 (v1과 약간 다름)\n@GetMapping(\"/v3/users/{id}\")  // v3 로직 (v2와 약간 다름)\n```\n\n- 라우팅 테이블이 버전 수 x 엔드포인트 수 만큼 증가\n- v1과 v2의 차이가 \"필드 하나 추가\"뿐인데도 별도 라우팅 필요\n- API Gateway에서 라우팅 규칙 관리가 복잡해짐\n\n⚠️ 단, 이건 **이론적 단점**이고, 실무에서는 URI 방식이 **가장 널리 쓰이는 방식**이야. 단점이 있지만 단순하고 명확하다는 장점이 더 크기 때문이야.\n\n---\n\n### Q4-2: CDN에서 API 응답을 캐싱하는가?\n\n#### 짧은 답변: ✅ 해. 하지만 **모든 API를 하는 건 아니야.**\n\n#### 현실 비유\n\nCDN 캐싱은 **편의점 진열대**와 같아:\n- 자주 팔리는 인기 상품(자주 조회되는 읽기 전용 API) → 진열대에 진열 ✅\n- 주문 제작 상품(개인화된 API 응답) → 진열 안 함 ❌\n\n#### CDN에서 캐싱하기 좋은 API vs 안 하는 API\n\n| 캐싱 적합 | 캐싱 부적합 |\n|-----------|-------------|\n| `GET /v1/products` (상품 목록) | `POST /v1/orders` (주문 생성) |\n| `GET /v1/categories` (카테고리) | `GET /v1/users/me` (개인 정보) |\n| `GET /v1/config` (앱 설정) | `GET /v1/cart` (장바구니) |\n| Public API 응답 | 인증 필요한 API |\n\n#### ✅ 네 말이 맞아 — 세밀한 제어 가능\n\n```\n# CDN 설정 예시 (Cloudflare/AWS CloudFront)\n/v1/products/*     → Cache TTL: 5분\n/v1/categories/*   → Cache TTL: 1시간\n/v1/users/*        → No Cache (Bypass)\n/v1/orders/*       → No Cache (Bypass)\n```\n\n- **어떤 API를 캐싱할지**: 경로(path) 기반 규칙으로 지정\n- **Invalidation 기준**: TTL(Time to Live) 설정, 또는 수동 purge API 호출\n- **Cache-Control 헤더**: 서버가 `Cache-Control: public, max-age=300` 같이 헤더로 지시\n\nURI 방식(`/v1/`, `/v2/`)이 CDN 캐싱에 유리한 이유는, CDN이 **URL 기반으로 캐시 키를 만들기 때문**이야. `/v1/products`와 `/v2/products`가 자동으로 다른 캐시 엔트리가 돼. Header 방식은 CDN이 `Vary` 헤더를 해석해야 해서 설정이 복잡해져.\n\n---\n\n### Q4-3: URI vs URL\n\n#### 현실 비유\n\n- **URI** = 사람의 **이름** (식별만 하면 됨). \"김철수\"\n- **URL** = 사람의 **집 주소** (찾아갈 수 있음). \"서울시 강남구 테헤란로 123에 사는 김철수\"\n\n#### 관계\n\n```\nURI (Uniform Resource Identifier) — 리소스를 \"식별\"\n ├── URL (Uniform Resource Locator) — 위치로 식별 (접근 방법 포함)\n └── URN (Uniform Resource Name) — 이름으로 식별\n```\n\n| | URI | URL | URN |\n|--|-----|-----|-----|\n| 의미 | 리소스 식별자 | 리소스 위치 | 리소스 이름 |\n| 예시 | `/users/123` | `https://api.com/users/123` | `urn:isbn:0451450523` |\n| 접근 방법 포함? | 아닐 수도 있음 | ✅ 항상 (https, ftp 등) | ❌ 없음 |\n| 관계 | 상위 개념 | URI의 하위 | URI의 하위 |\n\n📌 **실무에서는?** 거의 구분 없이 혼용해. 대부분의 웹 API 맥락에서 URI라고 하면 URL을 의미해. 면접에서는 \"URI는 식별자의 상위 개념이고, URL은 위치를 포함하는 URI의 하위 개념입니다\" 정도로 답하면 충분해.\n\n---\n\n### Q4-4: 쿼리 파라미터 & 쿼리\nsession.query(Product).filter(\n    Product.data['category'].astext == 'electronics'\n)\n```\n\n---\n\n### Q6-3: 풀텍스트 검색\n\n#### `LIKE '%keyword%'`는 풀텍스트 검색인가?\n\n**❌ 아니야.** `LIKE '%keyword%'`는 풀텍스트 검색이 아니라 **패턴 매칭(pattern matching)**이야. 풀텍스트 검색과는 근본적으로 다른 메커니즘이야.\n\n> 🧒 **비유**: `LIKE '%keyword%'`는 **도서관의 모든 책을 하나하나 펼쳐서, 모든 페이지의 모든 줄을 눈으로 읽으면서 \"keyword\"라는 단어가 있는지 찾는 것**이야. 풀텍스트 검색은 **도서관 뒤에 있는 \"색인(index)\" 책을 펴서, \"keyword\"를 찾으면 \"이 단어는 3번 책 42쪽, 7번 책 15쪽에 있습니다\"라고 바로 알려주는 것**이야.\n\n**왜 `LIKE '%keyword%'`가 slow query를 일으키는지:**\n\n1. 앞에 `%`가 붙으면 **B-Tree 인덱스를 사용할 수 없음** (인덱스는 접두사 기반이라 \"이 글자로 시작하는 것\"만 빠르게 찾음)\n2. 따라서 **Full Table Scan** 발생 → 테이블의 모든 행을 하나하나 확인\n3. 100만 행이면 100만 번 문자열 비교 → 당연히 느림\n\n---\n\n#### PostgreSQL의 GIN 인덱스 + tsvector가 이 문제를 어떻게 해결하는가\n\n**Step 1 — tsvector란?**\n\ntsvector는 **텍스트를 검색에 최적화된 형태로 변환한 것**이야.\n\n```\n원본 텍스트: \"The quick brown fox jumps over the lazy dog\"\ntsvector:    'brown':3 'dog':9 'fox':4 'jump':5 'lazi':8 'quick':2\n```\n\n- 불용어(the, over) 제거\n- 어근(stemming) 추출 (jumps → jump, lazy → lazi)\n- 각 단어의 **위치 정보** 저장\n\n**Step 2 — GIN 인덱스란?**\n\nGIN(Generalized Inverted Index)은 **역색인(inverted index)**이야. 구글 검색 엔진이 웹페이지를 찾는 것과 같은 원리야.\n\n```\n[GIN 인덱스 내부 구조]\n\n\"brown\"  → {row_id: 1, 5, 42, 100}\n\"fox\"    → {row_id: 1, 33}\n\"jump\"   → {row_id: 1, 5, 7, 88, 200}\n\"quick\"  → {row_id: 1, 99}\n...\n\n\"brown\"을 검색하면 → 바로 row 1, 5, 42, 100을 반환 (테이블 풀 스캔 없음!)\n```\n\n**Step 3 — 실제 사용:**\n```sql\n-- 1. tsvector 컬럼 추가 + GIN 인덱스 생성\nALTER TABLE articles ADD COLUMN search_vector tsvector;\nUPDATE articles SET search_vector = to_tsvector('english', title || ' ' || body);\nCREATE INDEX idx_search ON articles USING GIN (search_vector);\n\n-- 2. 풀텍스트 검색 (인덱스를 탐)\nSELECT * FROM articles \nWHERE search_vector @@ to_tsquery('english', 'quick & fox');\n\n-- 3. 순위 매기기까지 가능\nSELECT *, ts_rank(search_vector, to_tsquery('english', 'database')) AS rank\nFROM articles \nWHERE search_vector @@ to_tsquery('english', 'database')\nORDER BY rank DESC;\n```\n\n**⚠️ PostgreSQL에서도 slow query가 발생할 수 있는 경우:**\n- GIN 인덱스를 만들지 않고 `to_tsvector()` 함수를 직접 WHERE에 쓰면 → 매번 변환 + 풀 스캔\n- 매우 일반적인 단어(stopword가 아닌데 거의 모든 행에 있는 단어)를 검색하면 → 인덱스를 타도 결과가 너무 많아서 느림\n- GIN 인덱스 업데이트 비용: INSERT/UPDATE가 매우 빈번하면 GIN 인덱스 유지 비용이 높음\n- 하지만 **`LIKE '%keyword%'`에 비하면 차원이 다른 성능**\n\n---\n\n#### MySQL에서 풀텍스트 검색 문제를 어떻게 해결하나\n\n**방법 1: MySQL FULLTEXT 인덱스 사용 (내장)**\n\n```sql\n-- FULLTEXT 인덱스 생성\nALTER TABLE articles ADD FULLTEXT INDEX ft_idx (title, body);\n\n-- 검색 (LIKE 대신 MATCH...AGAINST 사용)\nSELECT * FROM articles \nWHERE MATCH(title, body) AGAINST('keyword' IN BOOLEAN MODE);\n```\n\n| 항목 | MySQL FULLTEXT | PostgreSQL tsvector + GIN |\n|------|---------------|--------------------------|\n| 언어 지원 | 영어 위주 (한국어/CJK는 ngram parser 필요) | 다국어 사전 지원, 커스텀 사전 가능 |\n| 연산자 | `+`, `-`, `*`, `\"\"` (기본적) | `&`, `|`, `!`, `<->` (근접 검색), 가중치 |\n| 순위 매기기 | 기본적인 relevance | `ts_rank`, `ts_rank_cd` (커버 밀도) |\n| 인덱스 타입 | InnoDB FULLTEXT (inverted index) | GIN (더 범용적) |\n| CJK 한국어 | ngram parser (MySQL 5.7.6+) 필요 | 별도 사전 설정 필요 |\n\n**방법 2: 외부 검색 엔진 (가장 흔한 실무 해결책)**\n\n실무에서 MySQL로 풀텍스트 검색이 필요하면, 대부분 **외부 검색 엔진**을 도입해:\n\n| 솔루션 | 특징 | 적합 |\n|--------|------|------|\n| **Elasticsearch** | 분산 검색 엔진, 강력한 분석/집계 | 대규모 로그, 복잡한 검색 요구 |\n| **Meilisearch** | 가볍고 빠름, 타이핑 중 검색 | 소규모~중규모, 빠른 구축 |\n| **OpenSearch** | Elasticsearch fork, AWS 관리형 | AWS 인프라 |\n\n**방법 3: 가상 컬럼 + B-Tree (간단한 경우)**\n\n특정 패턴만 검색한다면:\n```sql\n-- 접두사 검색으로 바꿀 수 있다면\nWHERE name LIKE 'keyword%'  -- B-Tree 인덱스 사용 가능!\n\n-- 또는 역순 인덱스\nALTER TABLE t ADD COLUMN name_reversed VARCHAR(255) \nGENERATED ALWAYS AS (REVERSE(name)) STORED;\nCREATE INDEX idx_rev ON t(name_reversed);\n-- 접미사 검색: WHERE name_reversed LIKE REVERSE('%keyword')\n```\n\n💡 **네 경험과 연결**: MySQL 8.x에서 `LIKE '%keyword%'` slow query를 겪었다면, 가장 현실적인 해결책은:\n1. 먼저 `FULLTEXT` 인덱스 + `MATCH...AGAINST`로 전환 시도\n2. 한국어가 포함되면 `ngram_token_size` 설정과 함께 ngram parser 사용\n3. 검색 요구사항이 복잡하면 Elasticsearch 같은 전용 엔진 도입\n\n---\n\n### Q6-4: 확장성\n\n#### 어떤 확장성을 이야기하는 거야?\n\n여기서 말하는 확장성(Extensibility)은 **하드웨어 확장(스케일링)이 아니라, 데이터베이스 기능 자체를 확장할 수 있는 능력**이야.\n\n> 🧒 **비유**: \n> - MySQL은 **조립식 장난감 세트**야. 설명서에 있는 모델만 만들 수 있어. 부품을 바꾸거나 새 부품을 추가하는 건 제한적.\n> - PostgreSQL은 **레고**야. 기본 블록으로 원하는 거 다 만들 수 있고, 다른 회사의 특수 블록(확장 모듈)도 끼울 수 있어.\n\n#### MySQL의 확장 — 무엇이 가능하고 왜 제한적인가\n\n**MySQL이 할 수 있는 것:**\n- 스토리지 엔진 교체 (InnoDB, MyISAM, MEMORY 등) — 이건 MySQL의 독특한 장점\n- UDF(User Defined Function)를 C/C++로 작성\n- 플러그인 시스템 (인증, 감사, 암호화 등)\n\n**왜 제한적인가:**\n\n| 제한 | 설명 |\n|------|------|\n| **커스텀 데이터 타입 불가** | INT, VARCHAR, JSON 등 내장 타입만 사용. \"좌표\", \"화폐\", \"IP주소\" 같은 새 타입을 만들 수 없음 |\n| **커스텀 인덱스 타입 불가** | B-Tree, Hash, FULLTEXT, R-Tree만 제공. 새로운 인덱스 알고리즘을 추가할 수 없음 |\n| **커스텀 연산자 불가** | `+`, `-`, `=` 같은 연산자를 새로 정의하거나 오버로딩 불가 |\n| **프로시저 언어 제한** | SQL과 (제한적인) 내장 프로시저 언어만 사용. Python이나 JavaScript로 함수 작성 불가 |\n| **외부 데이터 접근 제한** | 다른 DB나 파일 시스템을 \"외부 테이블\"로 연결하는 표준 방법 없음 |\n\n---\n\n#### PostgreSQL의 확장 — 커스텀 타입, 함수, 확장 모듈\n\n**1) 커스텀 데이터 타입 (Custom Types)**\n\nPostgreSQL은 새로운 데이터 타입을 만들 수 있어:\n```sql\n-- 복합 타입 (Composite Type)\nCREATE TYPE address AS (\n    street TEXT,\n    city TEXT,\n    zip_code VARCHAR(10)\n);\n\n-- 열거 타입 (Enum)\nCREATE TYPE mood AS ENUM ('sad', 'ok', 'happy');\n\n-- 범위 타입 (Range Type)\nCREATE TYPE salary_range AS RANGE (subtype = numeric);\n```\n\n**2) 커스텀 함수 — 여러 언어로 작성 가능**\n\n```sql\n-- PL/pgSQL (기본)\nCREATE FUNCTION calculate_tax(price NUMERIC) RETURNS NUMERIC AS $$\nBEGIN\n    RETURN price * 0.1;\nEND;\n$$ LANGUAGE plpgsql;\n\n-- PL/Python (Python으로 DB 함수 작성!)\nCREATE FUNCTION sentiment_analysis(text_input TEXT) RETURNS TEXT AS $$\n    import nltk  -- Python 라이브러리 사용 가능!\n    return analyze(text_input)\n$$ LANGUAGE plpython3u;\n\n-- PL/V8 (JavaScript로도 가능)\n```\n\n**3) 확장 모듈(Extension) — PostgreSQL의 핵심 강점**\n\n`CREATE EXTENSION` 한 줄로 강력한 기능을 추가:\n\n| 확장 모듈 | 기능 | MySQL에서의 대안 |\n|-----------|------|-----------------|\n| **PostGIS** | 지리/공간 데이터 처리 (위도/경도 검색, 거리 계산) | MySQL Spatial (매우 제한적) |\n| **pg_trgm** | 유사 문자열 검색 (오타 허용, fuzzy match) | 없음 (앱 레벨에서 처리) |\n| **hstore** | 키-값 저장 (JSONB보다 가벼움) | 없음 |\n| **uuid-ossp** | UUID 생성 | MySQL 8.0+ UUID() 함수 (제한적) |\n| **pgcrypto** | 암호화 함수 | AES_ENCRYPT 등 (제한적) |\n| **timescaledb** | 시계열 데이터 최적화 | 없음 (InfluxDB 등 별도 DB 필요) |\n| **pg_stat_statements** | 쿼리 성능 분석 | Performance Schema (내장) |\n| **FDW (Foreign Data Wrapper)** | 다른 DB/CSV/API를 외부 테이블로 연결 | FEDERATED 엔진 (매우 제한적) |\n\n💡 **핵심 차이의 근본 이유**: PostgreSQL은 설계 초기부터 **\"사용자가 타입, 연산자, 인덱스, 함수를 확장할 수 있는 카탈로그 기반 아키텍처\"**로 만들어졌어. 시스템 카탈로그(pg_type, pg_operator, pg_proc 등)에 새 항목을 등록하면 엔진이 자연스럽게 인식함. MySQL은 **애플리케이션 중심 설계**로, 빠르고 간단한 사용에 초점을 맞추다 보니 내부 확장 포인트가 적어.\n\n---\n\n### Q6-5: 쓰기 성능\n\n#### PostgreSQL INSERT가 더 빠르고 MySQL이 느린 것 아닌가?\n\n**좋은 질문이야. 상황에 따라 다르고, 단순 비교가 어려워.** 하나씩 풀어보자.\n\n> 🧒 **비유**:\n> - MySQL INSERT: 책장에 새 책을 꽂을 때, **목차(인덱스)도 같이 업데이트**하고, **수정 이력 메모장(Undo Log)도 준비**해야 함\n> - PostgreSQL INSERT: 새 책을 그냥 **서가 빈 자리에 꽂기만** 하면 됨. 목차는 나중에 업데이트\n\n**단순 INSERT 성능 비교:**\n\n| 요소 | MySQL/InnoDB | PostgreSQL |\n|------|-------------|------------|\n| **INSERT 시 해야 할 일** | 데이터 페이지에 쓰기 + Undo Log 준비 + redo log(WAL) | 데이터 페이지에 쓰기 + WAL |\n| **클러스터드 인덱스** | ✅ InnoDB는 PK 순서로 데이터 저장 → PK 순서 INSERT 빠름, 랜덤 PK INSERT 느림 | ❌ 힙(heap) 테이블 → 빈 곳에 아무데나 넣음. PK 순서 무관 |\n| **UPDATE의 본질** | in-place 수정 + Undo Log 체인 생성 | 새 튜플 INSERT + 이전 튜플에 xmax 표시 |\n\n**실제로는:**\n- **단순 대량 INSERT (벌크)**: 비슷하거나 MySQL이 약간 빠를 수 있음 (클러스터드 인덱스로 순차 쓰기 시)\n- **UPDATE가 섞인 혼합 워크로드**: PostgreSQL의 UPDATE는 사실상 INSERT+DELETE이므로, UPDATE 자체는 I/O가 더 많지만 잠금 경합은 적음\n- **동시 쓰기(concurrent writes)**: PostgreSQL이 유리할 수 있음. MySQL InnoDB는 Undo Log에 대한 잠금과 gap lock 등이 동시 쓰기를 제한\n\n📌 **원래 표의 \"MySQL = 단순 INSERT 빠름\"은 주로 InnoDB의 클러스터드 인덱스가 순차 PK INSERT에 최적화되어 있다는 맥락**이고, PostgreSQL이 INSERT 자체가 느리다는 뜻은 아니야.\n\n---\n\n#### \"복잡 트랜잭션에 강하다\"는 어떤 의미인가?\n\n**트랜잭션의 단순/복잡 구분:**\n\n| 구분 | 단순 트랜잭션 | 복잡 트랜잭션 |\n|------|-------------|-------------|\n| **예시** | 단일 행 INSERT, 단일 행 UPDATE, 포인트 조회 | 여러 테이블 JOIN + 집계 + 서브쿼리 + CTE + 조건부 UPDATE |\n| **특징** | 빠르게 시작-종료, 잠금 범위 좁음 | 오래 걸림, 여러 테이블/행에 걸쳐 잠금 |\n| **비유** | 편의점에서 물 하나 사기 | 대형마트에서 장바구니 가득 채워서 한번에 결제하기 |\n\n**PostgreSQL이 복잡 트랜잭션에 강한 이유:**\n\n| 이유 | 설명 |\n|------|------|\n| **1. 쿼리 옵티마이저가 더 정교** | PostgreSQL의 옵티마이저는 CTE, Window Function, Lateral Join, Recursive Query 등 복잡한 쿼리 패턴에 대해 더 다양한 실행 계획을 고려함. MySQL 옵티마이저는 역사적으로 단순 쿼리에 최적화 |\n| **2. MVCC 잠금 모델** | PostgreSQL의 SSI(Serializable Snapshot Isolation)는 낙관적 — 읽기가 쓰기를 블로킹하지 않음. MySQL의 Serializable은 비관적(모든 SELECT가 LOCK IN SHARE MODE) → 복잡 트랜잭션에서 교착(deadlock) 가능성 높음 |\n| **3. 풍부한 SQL 기능** | 복잡한 비즈니스 로직을 SQL 수준에서 처리 가능 (RETURNING 절, UPSERT, 배열 연산, JSON 연산 등) → 트랜잭션 내 라운드트립 감소 |\n| **4. B-Tree Index (MySQL, PostgreSQL 공통 기본)\n\n**🏠 비유: 백과사전의 \"ㄱ-ㄴ-ㄷ\" 목차 구조**\n\n백과사전은 \"ㄱ\"을 펼치면 그 안에 \"가-갸-거-겨...\"로 나뉘고, \"가\" 안에서 \"가격-가구-가방...\"으로 더 세분화된다. B-Tree도 이런 계층적 분류와 같다.\n\n**구현 원리:**\n```\n                    [50]                    ← Root (Level 0)\n                  /      \\\n          [20, 35]        [70, 85]          ← Branch (Level 1)\n         /   |   \\       /   |   \\\n    [10,15][25,30][40,45][60,65][75,80][90,95]  ← Leaf (Level 2)\n    \n    Leaf 노드끼리 양방향 연결 리스트 (PostgreSQL)\n    Leaf 노드끼리 단방향 연결 리스트 (MySQL InnoDB)\n```\n\n- **Balanced**: 루트에서 모든 Leaf까지 깊이가 동일 → 어떤 값을 찾든 동일한 시간\n- 일반적으로 3~4 레벨이면 수백만~수천만 행 커버 가능 (한 노드에 수백 개 키)\n- Leaf 노드가 연결되어 있으므로 **범위 검색**(BETWEEN, >, <)에 효율적\n\n**효과적인 경우:**\n- 등호(`=`), 범위(`>`, `<`, `BETWEEN`), 정렬(`ORDER BY`), `LIKE 'abc%'` (앞부분 일치)\n- **대부분의 경우에 사용** — \"뭘 쓸지 모르겠으면 B-Tree\"\n\n**비효과적인 경우:**\n- `LIKE '%abc'` (앞에 와일드카드) → 인덱스 풀스캔\n- 매우 낮은 카디널리티 (boolean 같은 것) → 풀스캔이 더 나을 수 있음\n\n---\n\n#### 2. Hash Index\n\n**🏠 비유: 학교 사물함 번호**\n\n학생 이름 \"김철수\" → 해시 함수 → 42번 사물함. 이름을 넣으면 바로 사물함 번호가 나오는 것. 중간이 없다. 42번이냐 아니냐만 있다.\n\n**구현 원리:**\n```\nkey \"alice\" → hash(\"alice\") = 0x3A2B → bucket 42 → row pointer\nkey \"bob\"   → hash(\"bob\")   = 0x7F1C → bucket 15 → row pointer\n```\n\n**MySQL에서의 상태:**\n- InnoDB에서 명시적 Hash Index 생성은 지원하지만, 내부적으로 B-Tree로 처리됨\n- 대신 **Adaptive Hash Index**가 있음: InnoDB가 자주 접근되는 B-Tree 페이지를 자동으로 해시 인덱스로 캐싱\n\n**PostgreSQL에서의 상태:**\n- PostgreSQL 10 이전에는 WAL(Write-Ahead Log) 미지원으로 크래시 시 손상 위험 → 사실상 사용 금지\n- PostgreSQL 10 이후 WAL 지원으로 안전해짐, 하지만 여전히 B-Tree 대비 장점이 제한적\n\n**효과적인 경우:**\n- **정확한 등호 비교만** 할 때 (`WHERE session_id = 'abc123'`)\n- 이론적으로 O(1) 접근\n\n**❌ 절대 안 되는 경우:**\n- 범위 검색, 정렬, 부분 일치 — 해시는 순서 개념이 없음\n\n---\n\n#### 3. GIN Index (PostgreSQL 전용)\n\n**🏠 비유: 책 맨 뒤의 \"찾아보기(색인)\" 페이지**\n\n책 뒤에 \"인공지능: p.15, p.42, p.103\" 이렇게 적혀있는 것. 하나의 키워드가 여러 페이지에 등장하는 것을 역으로 추적하는 구조.\n\n**구현 원리:**\n```\nGIN = Generalized Inverted Index (일반화된 역인덱스)\n\n원본 데이터:\n  row1: tags = ['python', 'fastapi', 'async']\n  row2: tags = ['python', 'django']\n  row3: tags = ['java', 'spring']\n\nGIN 인덱스 내부:\n  'async'   → {row1}\n  'django'  → {row2}\n  'fastapi' → {row1}\n  'java'    → {row3}\n  'python'  → {row1, row2}\n  'spring'  → {row3}\n```\n\n**효과적인 경우:**\n- **JSONB 컬럼 검색**: `WHERE data @> '{\"status\": \"active\"}'`\n- **배열 포함 검색**: `WHERE tags @> ARRAY['python']`\n- **Full-Text Search**: `WHERE to_tsvector('english', content) @@ to_tsquery('database & index')`\n- 하나의 행이 **여러 값을 가지는** 구조에 최적\n\n**트레이드오프:**\n- INSERT/UPDATE 시 인덱스 갱신 비용이 B-Tree보다 높음\n- `GIN_PENDING_LIST_LIMIT`으로 fastupdate 조절 가능\n\n---\n\n#### 4. GiST Index (PostgreSQL 전용)\n\n**🏠 비유: 지도에서 \"이 근처 음식점\" 찾기**\n\n네이버 지도에서 \"현재 위치 반경 1km 음식점\"을 검색하면, 지도가 영역을 잘게 나눠서 빠르게 찾아준다.\n\n**구현 원리:**\n```\nGiST = Generalized Search Tree\n\n공간을 계층적으로 분할:\n  전체 서울 → [강남구, 서초구, 송파구, ...]\n    강남구 → [역삼동, 대치동, 삼성동, ...]\n      역삼동 → [블록1, 블록2, ...]\n        블록1 → 개별 좌표점들\n```\n\n**효과적인 경우:**\n- **공간 데이터**: PostGIS와 함께 `ST_DWithin(geom, point, 1000)` (1km 이내)\n- **범위 타입**: `WHERE daterange @> '2024-01-15'` (겹침, 포함 연산)\n- **Full-Text Search**: GIN보다 빌드 빠르고 크기 작지만, 검색은 느림 (write가 많으면 GiST, read가 많으면 GIN)\n\n---\n\n#### 5. BRIN Index (PostgreSQL 전용)\n\n**🏠 비유: 아파트 층별 안내판**\n\n\"1~3층: 상가, 4~10층: 사무실, 11~30층: 주거\" 이런 안내판. 정확한 호수는 모르지만, \"주거 공간을 찾으려면 11층 이상만 보면 돼\"라고 범위를 빠르게 좁혀줌.\n\n**구현 원리:**\n```\nBRIN = Block Range INdex\n\n테이블의 물리적 블록을 그룹으로 묶어 요약 정보만 저장:\n\n  Block 0~127:   created_at min=2024-01-01, max=2024-01-31\n  Block 128~255: created_at min=2024-02-01, max=2024-02-28\n  Block 256~383: created_at min=2024-03-01, max=2024-03-31\n  \nWHERE created_at = '2024-02-15' → Block 128~255만 스캔\n```\n\n**효과적인 경우:**\n- **시계열 데이터**: 로그, 이벤트, 센서 데이터처럼 삽입 순서와 값의 순서가 상관관계가 높은 경우\n- **매우 큰 테이블**: 인덱스 크기가 B-Tree의 1/100~1/1000 수준\n- `pages_per_range` 파라미터로 정밀도 조절 가능\n\n**⚠️ 주의:**\n- 데이터가 물리적으로 정렬되어 있어야 효과적. 랜덤 삽입이면 모든 블록 범위가 겹쳐서 의미 없음\n- PostgreSQL은 Clustered Index가 없으므로, 주기적으로 `CLUSTER` 또는 처음부터 시간순 삽입되는 데이터에 적합\n\n---\n\n#### 6. Trie, Quadtree, k-d tree 등의 자료구조를 일반화한 것.\n\n**효과적인 경우:**\n- IP 주소 범위 검색 (`inet` 타입)\n- 전화번호, 우편번호처럼 계층적 구조의 문자열\n- 2D 공간에서 GiST보다 나은 경우가 있음 (데이터 분포에 따라)\n\n---\n\n#### 7. Full-Text Index (구현 방식 차이)\n\n| 측면 | MySQL | PostgreSQL |\n|---|---|---|\n| 엔진 | InnoDB 자체 구현 | `tsvector`/`tsquery` 타입 + GIN/GiST |\n| 한국어 | 기본 미지원 (ngram parser 필요) | 기본 미지원 (외부 사전 필요) |\n| 유연성 | 제한적 | 가중치, 랭킹, 사전 커스텀 가능 |\n| 성능 | 소규모 텍스트에 적합 | 대규모 텍스트에 적합 |\n\n---\n\n#### 8. Partial Index (PostgreSQL 전용)\n\n```sql\nCREATE INDEX idx_active_users ON users (email) WHERE is_active = true;\n```\n\n**비유:** 전교생 명단에서 \"현재 재학 중인 학생만\" 따로 정리한 명단. 졸업생은 포함하지 않아서 목록이 훨씬 작고 빠름.\n\n**효과적인 경우:**\n- `is_deleted = false`인 행만 자주 검색할 때\n- 특정 상태값을 가진 행이 전체의 소수일 때\n- 인덱스 크기를 줄여 메모리 효율 증가\n\n---\n\n### Q7-3: 인덱스 설계 원칙\n\n#### 카디널리티(Cardinality) 기반 인덱스 설계\n\n**🏠 비유: 분류 기준의 \"구별 능력\"**\n\n반 친구 30명을 찾을 때:\n- \"성별\"로 분류 → 남/여 2그룹 → 한 그룹에 15명 → **구별 능력 낮음** (Low Cardinality)\n- \"이름\"으로 분류 → 거의 30그룹 → 한 그룹에 1명 → **구별 능력 높음** (High Cardinality)\n- \"생년월일\"로 분류 → 비슷한 친구가 있을 수 있지만 꽤 잘 구별됨 → **중간 Cardinality**\n\n**원칙: 카디널리티가 높을수록 인덱스 효과가 크다.**\n\n**단계별 설명:**\n\n**1단계: 왜 높은 카디널리티가 좋은가?**\n```\ngender 인덱스로 WHERE gender = 'M' 검색:\n→ 전체 100만 행 중 50만 행이 해당 → 인덱스가 50%를 걸러냄 → 별 효과 없음\n→ 옵티마이저가 \"차라리 Full Table Scan이 낫겠다\"고 판단할 수 있음\n\nemail 인덱스로 WHERE email = 'alice@ex.com' 검색:\n→ 전체 100만 행 중 1행이 해당 → 인덱스가 99.9999%를 걸러냄 → 매우 효과적\n```\n\n**2단계: 카디널리티 확인 방법**\n```sql\n-- MySQL\nSHOW INDEX FROM users;  -- Cardinality 컬럼 확인\n\n-- PostgreSQL\nSELECT n_distinct FROM pg_stats WHERE tablename = 'users';\n-- n_distinct > 0: 고유값 개수 추정\n-- n_distinct < 0: 비율 (-0.5 = 전체의 50%가 고유)\n```\n\n**3단계: 복합 인덱스에서의 카디널리티**\n```sql\n-- 복합 인덱스에서는 카디널리티가 높은 컬럼을 앞에 배치\n-- ❌ (gender, email) → gender='M'으로 먼저 50%를 걸러낸 후 email 검색\n-- ✅ (email, gender) → email로 먼저 거의 1개로 좁힘\n\n-- 단, 쿼리 패턴이 더 중요! (아래에서 설명)\n```\n\n**⚠️ 예외 상황:**\n- Boolean 컬럼이라도 `WHERE is_deleted = false`를 거의 항상 검색한다면 → **PostgreSQL의 Partial Index**가 정답\n- 낮은 카디널리티라도 다른 컬럼과 복합 인덱스로 묶으면 효과적일 수 있음\n\n---\n\n#### 복합 인덱스 순서: WHERE → ORDER BY → SELECT\n\n**🏠 비유: 도서관에서 책 찾는 순서**\n\n\"파이썬 프로그래밍 책 중에서 최신순으로 정렬해서, 제목과 저자만 알고 싶어\"\n\n1. **WHERE** = \"파이썬 프로그래밍 책 찾기\" (조건 필터)\n2. **ORDER BY** = \"최신순 정렬\" (정렬)\n3. **SELECT** = \"제목과 저자만\" (필요한 정보)\n\n이 순서가 복합 인덱스의 컬럼 순서와 일치해야 한다.\n\n**단계별 설명:**\n\n**1단계: 쿼리 예시**\n```sql\nSELECT name, email           -- 3️⃣ 가져올 컬럼\nFROM users\nWHERE status = 'active'      -- 1️⃣ 필터 조건\n  AND department_id = 5      -- 1️⃣ 필터 조건\nORDER BY created_at DESC     -- 2️⃣ 정렬 조건\nLIMIT 20;\n```\n\n**2단계: 이상적인 복합 인덱스**\n```sql\nCREATE INDEX idx_users_optimal \nON users (status, department_id, created_at DESC, name, email);\n--        ^^^^^^^^^^^^^^^^^^^   ^^^^^^^^^^^^^^^   ^^^^^^^^^^\n--        WHERE 조건 (필터)      ORDER BY (정렬)   SELECT (커버링)\n```\n\n**3단계: 왜 이 순서인가?**\n\n```\n인덱스 B-Tree 구조:\n\nLevel 1: status = 'active' (필터 → 범위 축소)\n  Level 2: department_id = 5 (추가 필터 → 더 축소)\n    Level 3: created_at DESC 순서로 정렬되어 있음\n             → 별도 정렬(filesort) 불필요! \n             → Leaf 노드에 name, email 포함\n             → 테이블 접근(더블 룩업) 불필요!\n```\n\n만약 순서가 `(created_at, status, department_id)`라면?\n```\nLevel 1: created_at = ??? → 범위 전체를 스캔해야 함\n  → 그 중에서 status='active', department_id=5를 필터\n  → 인덱스는 있지만 비효율적\n```\n\n**4단계: 등호(=) 조건은 앞에, 범위 조건은 뒤에**\n```sql\n-- 이 쿼리에서:\nWHERE status = 'active'        -- 등호 (=)\n  AND department_id = 5        -- 등호 (=)\n  AND created_at > '2024-01-01' -- 범위 (>)\n\n-- 최적 인덱스: (status, department_id, created_at)\n-- 범위 조건 이후의 컬럼은 인덱스 검색에 활용 안 됨 (스캔만 가능)\n```\n\n#### ✅ MySQL과 PostgreSQL 공통인가?\n\n**예, 공통 원칙이다.** B-Tree 인덱스의 구조적 특성에서 나오는 원칙이므로, B-Tree를 사용하는 모든 RDBMS에 적용된다.\n\n다만 세부 차이:\n| 측면 | MySQL (InnoDB) | PostgreSQL |\n|---|---|---|\n| 옵티마이저 | 비용 기반, 인덱스 힌트 가능 | 비용 기반, 더 정교한 통계 |\n| DESC 인덱스 | 8.0+ 지원 | 오래전부터 지원 |\n| Covering 활용 | Clustered Index에 모든 컬럼 → 이미 커버링 | INCLUDE로 명시 필요 |\n| Index-Only Scan | \"Using index\" (EXPLAIN) | \"Index Only Scan\" (EXPLAIN) |\n\n---\n\n#### 💡 커버링 인덱스(Covering Index)란?\n\n**🏠 비유: 카드 목록함에 이미 답이 적혀있는 경우**\n\n도서관에서 \"파이썬 책의 저자가 누구야?\"라는 질문에:\n- **일반 인덱스**: 카드 목록함에서 \"파이썬 책은 3층 A구역 12번 칸\" → 직접 가서 저자 확인 (2번 이동)\n- **커버링 인덱스**: 카드 목록함에 이미 \"파이썬 책, 저자: 홍길동\"이라고 적혀있음 → 가지 않아도 됨 (1번만 조회)\n\n**정의:** 쿼리가 필요로 하는 **모든 컬럼이 인덱스 안에 포함**되어 있어서, 테이블 본체에 접근할 필요가 없는 인덱스.\n\n```sql\n-- 쿼리\nSELECT name, email FROM users WHERE status = 'active';\n\n-- 인덱스에 (status, name, email)이 포함되어 있으면 → 커버링 인덱스\n-- 테이블 접근(Random I/O) 없이 인덱스만으로 응답 가능\n```\n\n**PostgreSQL 11+의 INCLUDE 구문:**\n```sql\nCREATE INDEX idx_users_covering \nON users (status) INCLUDE (name, email);\n-- status는 검색/정렬에 사용, name/email은 조회만 (트리 구조에 포함 안 됨)\n-- → 인덱스 크기를 줄이면서 커버링 효과\n```\n\n**INCLUDE vs 그냥 복합 인덱스 차이:**\n```sql\n-- 이것은: (status, name, email) 세 컬럼 모두 B-Tree 키\nCREATE INDEX idx1 ON users (status, name, email);\n-- → name, email로도 정렬/검색 가능하지만 인덱스가 더 큼\n\n-- 이것은: status만 B-Tree 키, name/email은 Leaf에만 저장\nCREATE INDEX idx2 ON users (status) INCLUDE (name, email);\n-- → name, email로 정렬/검색 불가, 하지만 인덱스가 더 작고 유지비 낮음\n```\n\n---\n\n## Q8. N+1 쿼리 문제 해결 방법\n\n### Q8-1: SQLAlchemy에서 N+1 해결 방법 5가지\n\n**🏠 비유: 학교에서 학생 30명의 성적표를 모으는 방법**\n\n- **방법 A**: 교실에 가서 \"1번 학생 성적표 줘\" → 다시 가서 \"2번 학생\" → ... → 30번 반복 (N+1)\n- **방법 B**: \"1반 전체 학생 성적표 한꺼번에 줘\" (한 번에 해결)\n\nSQLAlchemy에서 사용 가능한 N+1 해결 방법:\n\n| 방법 | SQLAlchemy 지원 | 설명 |\n|---|---|---|\n| **1. Eager Loading (joinedload)** | ✅ `joinedload()` | SQL JOIN으로 한 번에 로드 |\n| **2. Subquery Loading** | ✅ `subqueryload()` | 서브쿼리로 연관 데이터 별도 로드 |\n| **3. Select-In Loading** | ✅ `selectinload()` | `WHERE id IN (...)` 으로 로드 |\n| **4. Batch Loading (Lazy)** | ✅ `lazy='selectin'` 또는 relationship 설정 | 배치 단위로 로드 |\n| **5. DTO Projection** | ✅ `query(User.name, User.email)` 또는 `load_only()` | 필요한 컬럼만 조회 |\n\n**각각의 SQLAlchemy 코드:**\n\n```python\n# 1. Joined Load (= JPA의 Fetch Join과 유사)\nsession.query(User).options(joinedload(User.orders)).all()\n# → SELECT * FROM users LEFT JOIN orders ON ...\n\n# 2. Subquery Load (= JPA의 @Fetch(SUBSELECT)과 유사)\nsession.query(User).options(subqueryload(User.orders)).all()\n# → SELECT * FROM users;\n# → SELECT * FROM orders WHERE user_id IN (SELECT id FROM users);\n\n# 3. Select-In Load (SQLAlchemy 1.2+, 가장 권장)\nsession.query(User).options(selectinload(User.orders)).all()\n# → SELECT * FROM users;\n# → SELECT * FROM orders WHERE user_id IN (1, 2, 3, ...);\n\n# 4. Batch 크기 조절 (relationship 레벨)\nclass User(Base):\n    orders = relationship(\"Order\", lazy=\"selectin\")\n    # 또는 lazy=\"subquery\"\n\n# 5. 필요한 컬럼만 (Projection)\nsession.query(User.id, User.name).join(User.orders).all()\n# → 연관 엔티티 자체를 로드하지 않음 → N+1 발생하지 않음\n```\n\n💡 **SQLAlchemy에서 가장 권장되는 방법은 `selectinload()`이다.** `joinedload()`는 카테시안 곱 문제가 있을 수 있고, `subqueryload()`는 서브쿼리가 복잡해질 수 있다. `selectinload()`는 깔끔한 `IN` 절로 해결한다.\n\n---\n\n### Q8-2: Fetch Join에서 페이징은 정말 불가능한가?\n\n**짧은 답: \"불가능한 것이 아니라, 위험하다.\"**\n\n**🏠 비유로 이해하기:**\n\n책방에서 \"저자별로 책 목록\"을 만들 때:\n- 저자 A: 책 3권\n- 저자 B: 책 2권\n- 저자 C: 책 4권\n\nJOIN 결과는 이렇게 된다:\n```\nrow 1: 저자A - 책1\nrow 2: 저자A - 책2    ← 저자A인데 3행\nrow 3: 저자A - 책3\nrow 4: 저자B - 책1\nrow 5: 저자B - 책2    ← 저자B인데 2행\nrow 6: 저자C - 책1\n...\nrow 9: 저자C - 책4\n```\n\n여기서 `LIMIT 3 OFFSET 0`을 걸면?\n→ row 1~3 = 저자A의 책 3개만 나옴\n→ **저자 단위로 \"3명의 저자\"를 원했는데, 실제로는 1명만 나옴!**\n\n**JPA/Hibernate에서 이것이 문제인 이유:**\n\n```java\n// 이 코드를 실행하면:\n@Query(\"SELECT u FROM User u JOIN FETCH u.orders\")\nPage<User> findAll(Pageable pageable);\n```\n\nHibernate가 하는 일:\n1. SQL에 LIMIT/OFFSET을 **붙이지 않음**\n2. 전체 데이터를 **메모리로 다 가져옴**\n3. **Java 메모리 안에서** 페이징 처리\n4. 로그에 경고: `HHH000104: firstResult/maxResults specified with collection fetch; applying in memory!`\n\n→ 100만 건이면 100만 건이 메모리에 올라옴 → **OOM(Out Of Memory) 위험**\n\n**해결 방법들:**\n\n**방법 1: 2단계 쿼리 (가장 안전)**\n```sql\n-- 1단계: ID만 페이징해서 가져옴\nSELECT u.id FROM users u WHERE ... ORDER BY u.created_at LIMIT 20 OFFSET 40;\n\n-- 2단계: 해당 ID들에 대해 Fetch Join\nSELECT u FROM User u JOIN FETCH u.orders WHERE u.id IN (:ids)\n```\n\n**방법 2: @BatchSize + Lazy Loading**\n```java\n// 페이징은 User 엔티티만 대상\n// 연관 데이터는 BatchSize만큼 IN 절로 로드\n@BatchSize(size = 100)\n@OneToMany(mappedBy = \"user\")\nprivate List<Order> orders;\n```\n\n**방법 3: 질문의 \"offset 쿼리\"에 대해**\n직접 네이티브 SQL로 `OFFSET`을 적용할 수는 있지만, 위에서 설명한 것처럼 JOIN된 결과의 행 수가 원래 엔티티 수와 다르기 때문에 의도한 페이징이 되지 않는다. **부모 엔티티 기준이 아닌 JOIN 결과 행 기준으로 잘리기 때문이다.**\n\n---\n\n### Q8-3: EntityGraph 방식이란?\n\n**🏠 비유: 쇼핑 목록에 \"이것도 같이 가져와\"라고 체크 표시하기**\n\n마트에 갈 때 쇼핑 목록을 만든다. 기본적으로 \"우유\"만 사려고 했는데, 목록에 \"시리얼도 같이(체크)\" 표시하면 한 번에 가져온다.\n\nEntityGraph는 **\"이 엔티티를 조회할 때 어떤 연관 엔티티를 함께 로드할지\"를 선언적으로 지정**하는 JPA 2.1 표준 기능이다.\n\n```java\n// 방법 1: 어노테이션으로 정의\n@Entity\n@NamedEntityGraph(\n    name = \"User.withOrders\",\n    attributeNodes = @NamedAttributeNode(\"orders\")  // orders도 같이 가져와\n)\npublic class User {\n    @OneToMany(mappedBy = \"user\")\n    private List<Order> orders;  // 기본은 LAZY\n}\n\n// Repository에서 사용\n@EntityGraph(value = \"User.withOrders\", type = EntityGraphType.LOAD)\nList<User> findAll();\n\n// 방법 2: 동적으로 정의 (어노테이션 없이)\n@EntityGraph(attributePaths = {\"orders\", \"orders.orderItems\"})\nList<User> findByStatus(String status);\n```\n\n**Fetch Join과의 비교:**\n\n| 측면 | Fetch Join | EntityGraph |\n|---|---|---|\n| 정의 위치 | JPQL 쿼리 안 | 어노테이션 또는 별도 정의 |\n| 재사용성 | 쿼리마다 작성 | 한 번 정의, 여러 곳 재사용 |\n| JOIN 타입 | INNER JOIN (기본) | LEFT JOIN (기본) |\n| 유연성 | 쿼리마다 다르게 가능 | 같은 그래프를 여러 쿼리에 적용 |\n| 페이징 문제 | 동일하게 발생 | **동일하게 발생** |\n| 표준 | JPQL 문법 | JPA 2.1 표준 |\n\n**💡 핵심:** EntityGraph도 내부적으로 LEFT JOIN을 생성하므로, Fetch Join과 본질적으로 같은 메커니즘이다. 차이는 **선언 방식**이다. EntityGraph는 \"무엇을 로드할지\"를 쿼리와 분리해서 선언적으로 관리한다는 것이 장점.\n\n---\n\n### Q8-4: Batch Fetch Size와 Fetch Join의 관계\n\n**결론부터: Batch Fetch Size는 Fetch Join과 함께 사용하는 것이 아니다. 별개의 전략이다.**\n\n**🏠 비유:**\n\n- **Fetch Join** = \"한 트럭에 전부 싣고 한 번에 배달\" (JOIN으로 한 쿼리)\n- **Batch Fetch Size** = \"택배를 100개씩 묶어서 여러 번 배달\" (여러 쿼리지만, 1개씩이 아닌 배치 단위)\n\n**동작 방식:**\n\n```java\n// application.yml\nspring:\n  jpa:\n    properties:\n      hibernate:\n        default_batch_fetch_size: 100\n\n// 또는 엔티티에 직접\n@BatchSize(size = 100)\n@OneToMany(mappedBy = \"user\")\nprivate List<Order> orders;\n```\n\n```\n설정: batch_fetch_size = 100\nUser 300명 조회 시:\n\nFetch Join 없이 Lazy Loading + BatchSize:\n  쿼리 1: SELECT * FROM users;                              -- User 300명\n  쿼리 2: SELECT * FROM orders WHERE user_id IN (1,2,...,100);   -- 1~100번 User의 주문\n  쿼리 3: SELECT * FROM orders WHERE user_id IN (101,...,200);   -- 101~200번\n  쿼리 4: SELECT * FROM orders WHERE user_id IN (201,...,300);   -- 201~300번\n  \n  총 4개 쿼리 (1 + 300/100 = 4)\n\nBatchSize 없이 Lazy Loading:\n  쿼리 1: SELECT * FROM users;\n  쿼리 2: SELECT * FROM orders WHERE user_id = 1;\n  쿼리 3: SELECT * FROM orders WHERE user_id = 2;\n  ...\n  쿼리 301: SELECT * FROM orders WHERE user_id = 300;\n  \n  총 301개 쿼리 (1 + 300 = 301) ← 이것이 N+1\n```\n\n**핵심:** BatchSize는 **Lazy Loading이 발생할 때**, 1개씩 가져오는 대신 지정된 크기만큼 묶어서 `IN` 절로 가져온다. Fetch Join을 사용하면 BatchSize는 무시된다 (이미 JOIN으로 다 가져왔으므로).\n\n---\n\n### Q8-5: DTO Projection이 어떻게 N+1을 해결하는가?\n\n**좋은 지적이다. DTO Projection 자체가 N+1을 \"해결\"하는 것은 아니다.** 정확히 말하면 **N+1이 발생할 수 있는 구조를 아예 회피**하는 것이다.\n\n**🏠 비유:**\n\nN+1 문제는 \"연관된 물건\"을 따로따로 가지러 가는 문제이다.\n- **엔티티 조회**: \"사람 객체\"를 가져오면, 그 사람의 \"주문 목록\" \"배송 주소\" 등 연관 객체가 딸려 있음 → 접근 시 추가 쿼리\n- **DTO Projection**: \"이름, 이메일, 주문 개수\"만 **딱 필요한 값만** 가져옴 → 연관 엔티티 자체를 로드하지 않음 → Lazy Loading 트리거 자체가 없음\n\n**단계별 설명:**\n\n```java\n// ❌ 엔티티 조회 → N+1 위험\nList<User> users = userRepository.findAll();\nfor (User u : users) {\n    System.out.println(u.getOrders().size());  // 여기서 Lazy Loading 발생!\n}\n\n// ✅ DTO Projection → N+1 불가능\n@Query(\"SELECT new com.example.UserDTO(u.name, u.email, COUNT(o)) \" +\n       \"FROM User u LEFT JOIN u.orders o GROUP BY u.name, u.email\")\nList<UserDTO> findUserSummaries();\n// → 단일 쿼리, 엔티티가 아닌 순수 데이터 → Persistence Context에 들어가지 않음\n// → Lazy Loading 자체가 불가능 → N+1 불가능\n```\n\n**그래서 네 말이 맞다:** SELECT를 간소화하는 것이 맞다. 그런데 그 간소화의 결과로, **연관 엔티티를 아예 로드하지 않게 되어** N+1이 구조적으로 발생하지 않는 것이다. \"해결\"이라기보다 **\"회피\"**가 정확한 표현이다.\n\n---\n\n### Q8-6: @Fetch(SUBSELECT)란? Fetch Join과 무엇이 다른가?\n\n**🏠 비유:**\n\n- **Fetch Join** = \"학생 명단과 각 학생의 시험 점수를 하나의 큰 표(JOIN)로 합쳐서 가져오기\"\n- **@Fetch(SUBSELECT)** = \"학생 명단을 먼저 가져오고, 그 다음에 '이 학생들 전원의 시험 점수'를 한 번에 가져오기\"\n\n**동작 방식:**\n\n```java\n@Entity\npublic class User {\n    @OneToMany(mappedBy = \"user\", fetch = FetchType.LAZY)\n    @Fetch(FetchMode.SUBSELECT)  // Hibernate 전용 (JPA 표준 아님)\n    private List<Order> orders;\n}\n```\n\n```sql\n-- 1번째 쿼리: User 조회\nSELECT * FROM users WHERE department_id = 5;\n\n-- 2번째 쿼리: 위 쿼리의 결과에 해당하는 모든 orders를 서브쿼리로 한 번에\nSELECT * FROM orders \nWHERE user_id IN (\n    SELECT id FROM users WHERE department_id = 5  -- 원래 쿼리가 서브쿼리로 들어감\n);\n```\n\n**Fetch Join vs SUBSELECT vs BatchSize 비교:**\n\n| 측면 | Fetch Join | @Fetch(SUBSELECT) | @BatchSize |\n|---|---|---|---|\n| 쿼리 수 | **1개** | **2개** | **1 + N/size** |\n| JOIN 발생 | ✅ (카테시안 곱 위험) | ❌ | ❌ |\n| 페이징 문제 | ✅ 있음 | ❌ 없음 | ❌ 없음 |\n| 메모리 | JOIN 결과가 커질 수 있음 | 적절 | 적절 |\n| 설정 위치 | JPQL에서 명시 | 엔티티 어노테이션 | 엔티티/전역 설정 |\n| 표준 | JPA 표준 | Hibernate 전용 | Hibernate 전용 |\n| 연관 컬렉션 여러 개 | 1개만 가능 | 여러 개 가능 | 여러 개 가능 |\n\n**질문의 핵심에 대해: \"서브쿼리가 SELECT 절에 있으면 문제 아닌가?\"**\n\n정확한 지적이다. 만약 이런 쿼리라면:\n```sql\n-- ❌ 이건 N+1과 같은 문제 (Correlated Subquery)\nSELECT u.*, \n    (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.id) as order_count\nFROM users u;\n-- → users 행마다 서브쿼리 실행 가능\n```\n\n하지만 @Fetch(SUBSELECT)는 이렇게 동작하지 않는다:\n```sql\n-- ✅ 이건 독립적인 서브쿼리 (Non-correlated → 한 번만 실행)\nSELECT * FROM orders \nWHERE user_id IN (SELECT id FROM users WHERE department_id = 5);\n-- → 서브쿼리는 한 번 실행되고, 그 결과로 IN 절을 만들어 orders를 한 번에 가져옴\n```\n\n**핵심 차이:** SUBSELECT는 WHERE ... IN (서브쿼리) 형태로 **FROM/WHERE 절에서** 사용되며, 각 행마다 반복 실행되는 Correlated Subquery가 아니다. 원래 쿼리를 통째로 서브쿼리로 넣어서, 연관 데이터를 **한 방에** 가져오는 것이다.\n\n---\n\n## Q9. JPA에서 Lazy Loading과 Eager Loading\n\n### Q9-1: Lazy Loading이 N+1의 원인이 맞는가?\n\n**✅ 맞다. 정확한 이해다.**\n\n**단계별로 왜 그런지 설명:**\n\n**1단계: Lazy Loading의 동작**\n```java\nList<User> users = userRepository.findAll();\n// → SELECT * FROM users;  (1개 쿼리)\n// 이 시점에서 user.orders는 \"프록시 객체\" (빈 껍데기)\n```\n\n**2단계: N+1 발생**\n```java\nfor (User user : users) {  // users가 100명이라면\n    user.getOrders().size(); // ← 여기서 프록시가 초기화됨\n    // → SELECT * FROM orders WHERE user_id = 1;\n    // → SELECT * FROM orders WHERE user_id = 2;\n    // → ...100번 반복\n}\n// 총 쿼리: 1 (User 조회) + 100 (각 User의 Orders) = 101개 → N+1\n```\n\n**3단계: 그래서 Fetch Join으로 해결**\n```java\n@Query(\"SELECT u FROM User u JOIN FETCH u.orders\")\nList<User> findAllWithOrders();\n// → SELECT u.*, o.* FROM users u INNER JOIN orders o ON u.id = o.user_id;\n// → 1개 쿼리로 끝\n```\n\n**💡 정리:**\n```\nLazy Loading 자체는 \"나쁜 것\"이 아님\n  → 연관 데이터를 안 쓰면 로드 안 해서 효율적\n\n문제는 \"Lazy Loading + 반복 접근\" 조합\n  → N+1 발생\n  \n해결: 미리 알고 있으면 Fetch Join / Batch Size / SUBSELECT 등으로 한 번에 가져옴\n```\n\n그런데 **Eager Loading이 답이냐?** → ❌ 아니다.\n```\nEager Loading을 기본값으로 쓰면:\n  → 안 쓰는 연관 데이터도 무조건 로드\n  → 연관의 연관의 연관까지 줄줄이 로드될 수 있음\n  → 성능 대참사\n  \n업계 표준: \"모든 연관관계는 LAZY로, 필요할 때 Fetch Join\"\n```\n\n---\n\n### Q9-2: OneToMany, ManyToMany, OneToOne 실제 예시\n\n**쏘카인드 교육 도메인**을 사용하여 설명.\n\n#### 도메인 모델\n\n```\nCompany (회사: 쏘카인드)\n   │\n   ├── 1:N ── User (사용자: 직원들)\n   │            │\n   │            ├── N:M ── EducationSession (교육 세션)\n   │            │            │\n   │            │            └── 1:1 ── AIReport (AI 리포트)\n   │            │\n   │            └── 1:1 ── UserProfile (사용자 프로필)\n   │\n   └── 1:N ── EducationSession (교육 세션은 회사에 소속)\n```\n\n---\n\n#### 📌 @OneToMany: 회사 → 사용자들\n\n**테이블 구조:**\n```\ncompanies                    users\n+----+-----------+          +----+--------+------------+\n| id | name      |          | id | name   | company_id |  ← FK\n+----+-----------+          +----+--------+------------+\n| 1  | 쏘카인드  |          | 1  | 김철수 | 1          |\n+----+-----------+          | 2  | 이영희 | 1          |\n                            | 3  | 박민수 | 1          |\n                            +----+--------+------------+\n```\n\n**JPA 엔티티:**\n```java\n@Entity\npublic class Company {\n    @Id @GeneratedValue\n    private Long id;\n    private String name;\n    \n    @OneToMany(mappedBy = \"company\", fetch = FetchType.LAZY)  // 기본값이 LAZY\n    private List<User> users;\n}\n\n@Entity\npublic class User {\n    @Id @GeneratedValue\n    private Long id;\n    private String name;\n    \n    @ManyToOne(fetch = FetchType.LAZY)  // 주의: ManyToOne 기본값은 EAGER!\n    @JoinColumn(name = \"company_id\")\n    private Company company;\n}\n```\n\n**N+1 발생 시나리오:**\n```java\n// \"모든 회사와 각 회사의 직원 수를 조회\"\nList<Company> companies = companyRepository.findAll();\n// → SELECT * FROM companies;  (1쿼리)\n\nfor (Company c : companies) {\n    System.out.println(c.getName() + \": \" + c.getUsers().size());\n    // → SELECT * FROM users WHERE company_id = 1;  (회사마다 1쿼리)\n    // → SELECT * FROM users WHERE company_id = 2;\n    // → ...\n}\n```\n\n---\n\n#### 📌 @ManyToMany: 사용자 ↔ 교육 세션\n\n**비유:** 학교 수강신청. 한 학생이 여러 수업을 듣고, 한 수업에 여러 학생이 있음.\n\n**테이블 구조:**\n```\nusers                  user_sessions (중간 테이블)       education_sessions\n+----+--------+       +---------+------------+         +----+----------------+\n| id | name   |       | user_id | session_id |         | id | title          |\n+----+--------+       +---------+------------+         +----+----------------+\n| 1  | 김철수 |       | 1       | 1          |         | 1  | AI 기초        |\n| 2  | 이영희 |       | 1       | 2          |         | 2  | Python 심화    |\n| 3  | 박민수 |       | 2       | 1          |         | 3  | 리더십 교육    |\n+----+--------+       | 2       | 3          |         +----+----------------+\n                      | 3       | 1          |\n                      | 3       | 2          |\n                      | 3       | 3          |\n                      +---------+------------+\n```\n\n김철수: AI 기초, Python 심화 수강\n이영희: AI 기초, 리더십 교육 수강\n박민수: 전부 수강\n\n**JPA 엔티티:**\n```java\n@Entity\npublic class User {\n    @ManyToMany(fetch = FetchType.LAZY)  // 기본값이 LAZY\n    @JoinTable(\n        name = \"user_sessions\",\n        joinColumns = @JoinColumn(name = \"user_id\"),\n        inverseJoinColumns = @JoinColumn(name = \"session_id\")\n    )\n    private List<EducationSession> sessions;\n}\n\n@Entity\npublic class EducationSession {\n    @ManyToMany(mappedBy = \"sessions\", fetch = FetchType.LAZY)\n    private List<User> participants;\n}\n```\n\n**N+1 발생 시나리오:**\n```java\n// \"각 교육 세션에 참여한 사람 목록\"\nList<EducationSession> sessions = sessionRepository.findAll();\n// → SELECT * FROM education_sessions;  (1쿼리)\n\nfor (EducationSession s : sessions) {\n    System.out.println(s.getTitle() + \": \" + s.getParticipants().size());\n    // → SELECT u.* FROM users u \n    //   JOIN user_sessions us ON u.id = us.user_id \n    //   WHERE us.session_id = 1;   (세션마다 1쿼리)\n}\n```\n\n**⚠️ ManyToMany 추가 주의사항:**\n- 중간 테이블(`user_sessions`)에 추가 컬럼(수강일, 점수 등)이 필요하면 → ManyToMany 사용 불가 → **별도 엔티티로 분리해야 함** (`UserSession` 엔티티)\n- 실무에서는 거의 대부분 별도 엔티티로 분리한다\n\n---\n\n#### 📌 @OneToOne: 교육 세션 → AI 리포트\n\n**비유:** 한 세션이 끝나면 AI가 리포트를 **딱 하나** 생성. 세션과 리포트는 1:1 관계.\n\n**테이블 구조:**\n```\neducation_sessions              ai_reports\n+----+----------------+        +----+------------+--------+-------+\n| id | title          |        | id | session_id | score  | summary |\n+----+----------------+        +----+------------+--------+-------+\n| 1  | AI 기초        |        | 1  | 1          | 85.5   | \"참여도 높음\" |\n| 2  | Python 심화    |        | 2  | 2          | 72.0   | \"실습 필요\" |\n| 3  | 리더십 교육    |        +----+------------+--------+-------+\n+----+----------------+        (세션 3은 아직 리포트 없음)\n```\n\n**JPA 엔티티:**\n```java\n@Entity\npublic class EducationSession {\n    @Id @GeneratedValue\n    private Long id;\n    private String title;\n    \n    @OneToOne(mappedBy = \"session\", fetch = FetchType.LAZY)\n    private AIReport aiReport;\n}\n\n@Entity\npublic MySQL은 **type 컬럼을 먼저** 확인, PostgreSQL은 **최상위 노드의 cost**를 먼저 확인\n\n---\n\n## Q11. 대용량 테이블에서 페이징 성능 개선\n\n### Q11-1: 페이징 전략 실제 구현 코드 (Spring Boot/JPA)\n\n#### 방법 1: Offset 기반 (기본, 간단하지만 느림)\n\n```java\n// Controller\n@GetMapping(\"/posts\")\npublic Page<PostDto> getPosts(\n    @RequestParam(defaultValue = \"0\") int page,\n    @RequestParam(defaultValue = \"20\") int size\n) {\n    Pageable pageable = PageRequest.of(page, size, Sort.by(\"id\").descending());\n    return postService.getPosts(pageable);\n}\n\n// Repository\npublic interface PostRepository extends JpaRepository<Post, Long> {\n    // Spring Data JPA가 자동으로 OFFSET/LIMIT 쿼리 생성\n    Page<Post> findAll(Pageable pageable);\n}\n```\n\n생성되는 SQL:\n```sql\nSELECT * FROM posts ORDER BY id DESC LIMIT 20 OFFSET 1000000;\n-- + COUNT 쿼리도 추가로 실행됨 (총 개수 파악용)\n```\n\n#### 방법 2: Cursor(Keyset) 기반 (빠름, 무한 스크롤에 적합)\n\n```java\n// Controller\n@GetMapping(\"/posts\")\npublic List<PostDto> getPosts(\n    @RequestParam(required = false) Long lastId,\n    @RequestParam(defaultValue = \"20\") int size\n) {\n    return postService.getPostsAfter(lastId, size);\n}\n\n// Repository\npublic interface PostRepository extends JpaRepository<Post, Long> {\n    \n    // 첫 페이지 (lastId가 null일 때)\n    List<Post> findTopByOrderByIdDesc(Pageable pageable);\n    \n    // 다음 페이지 (lastId 기준)\n    @Query(\"SELECT p FROM Post p WHERE p.id < :lastId ORDER BY p.id DESC\")\n    List<Post> findByIdLessThan(@Param(\"lastId\") Long lastId, Pageable pageable);\n}\n\n// Service\n@Service\npublic class PostService {\n    public List<PostDto> getPostsAfter(Long lastId, int size) {\n        Pageable pageable = PageRequest.of(0, size); // offset은 항상 0!\n        \n        List<Post> posts;\n        if (lastId == null) {\n            posts = postRepository.findTopByOrderByIdDesc(pageable);\n        } else {\n            posts = postRepository.findByIdLessThan(lastId, pageable);\n        }\n        return posts.stream().map(PostDto::from).toList();\n    }\n}\n```\n\n생성되는 SQL:\n```sql\n-- 첫 페이지\nSELECT * FROM posts ORDER BY id DESC LIMIT 20;\n\n-- 다음 페이지 (lastId = 999980)\nSELECT * FROM posts WHERE id < 999980 ORDER BY id DESC LIMIT 20;\n```\n\n#### 방법 3: Covering Index + Deferred Join (중간 지점, offset이 필요할 때)\n\n```java\n// Repository - Native Query 사용\n@Query(value = \"\"\"\n    SELECT p.* FROM posts p\n    INNER JOIN (\n        SELECT id FROM posts ORDER BY id DESC LIMIT :size OFFSET :offset\n    ) AS sub ON p.id = sub.id\n    ORDER BY p.id DESC\n    \"\"\", nativeQuery = true)\nList<Post> findWithDeferredJoin(\n    @Param(\"offset\") int offset, \n    @Param(\"size\") int size\n);\n```\n\n핵심: 서브쿼리에서 **id만** 먼저 찾고 (인덱스만 스캔), 그 id로 본 테이블을 조인.\n\n| 전략 | 장점 | 단점 | 적합한 상황 |\n|------|------|------|------------|\n| Offset | 간단, 페이지 번호 지원 | 뒤로 갈수록 느림 | 관리자 화면, 소규모 데이터 |\n| Cursor | 일관된 성능, 데이터 정합성 | 페이지 번호 불가, \"5페이지로 건너뛰기\" 불가 | 무한 스크롤, 타임라인 |\n| Deferred Join | Offset보다 빠름, 페이지 번호 지원 | 복잡한 쿼리, DB 종속적 | Offset이 필요하지만 성능도 필요할 때 |\n\n---\n\n### Q11-2: 커서 기반이 offset보다 빠른 이유\n\n좋은 질문이야! 핵심을 정확히 짚어볼게.\n\n**현실 비유**: 도서관에서 책을 찾는 상황을 생각해봐.\n\n- **Offset 방식**: \"100만 번째 책부터 20권 주세요\" → 사서가 첫 번째 책부터 하나씩 세면서 100만 번째까지 감 → 그 다음 20권 뽑음\n- **Cursor 방식**: \"ISBN 번호가 999980보다 작은 것 중에서 20권 주세요\" → 사서가 ISBN 색인(인덱스)을 펼쳐서 999980 바로 앞 위치를 찾음 → 거기서 20권 뽑음\n\n#### 단계별로 보자\n\n**Offset 쿼리**:\n```sql\nSELECT * FROM posts ORDER BY id DESC LIMIT 20 OFFSET 1000000;\n```\n\nDB 내부 동작:\n1. `ORDER BY id DESC` → PK 인덱스를 역순으로 탐색 시작\n2. 1번째 행 읽음 → \"아직 offset 안 채움, 버림\"\n3. 2번째 행 읽음 → \"아직 offset 안 채움, 버림\"\n4. ... (100만 번 반복)\n5. 1,000,001번째 행 → \"이제 시작!\" → 결과에 포함\n6. 20개 모은 후 종료\n\n**Cursor 쿼리**:\n```sql\nSELECT * FROM posts WHERE id < 999980 ORDER BY id DESC LIMIT 20;\n```\n\nDB 내부 동작:\n1. `WHERE id < 999980` → PK 인덱스에서 **B-Tree 탐색** (O(log n))으로 999979 위치를 바로 찾음\n2. 거기서부터 역순으로 20개만 읽음\n3. 끝!\n\n#### 핵심 차이\n\n> **Offset은 인덱스를 사용하긴 하지만, \"몇 번째\"라는 위치를 찾기 위해 처음부터 세야 함.**\n> **Cursor는 WHERE 조건으로 B-Tree 인덱스를 직접 탐색(seek)하므로 바로 그 위치로 점프함.**\n\n너의 질문 \"offset 방식은 where절에서 index를 사용하지 않아서 그런가?\" → ✅ 정확해! Offset 쿼리에는 WHERE 절이 없어. `ORDER BY id DESC`로 인덱스를 타긴 하지만, OFFSET 1000000이라는 건 \"인덱스를 처음부터 100만 개 건너뛰어라\"는 뜻이고, B-Tree는 **\"몇 번째 노드\"를 직접 찾는 기능이 없어**. 하나씩 세야 해.\n\n반면 Cursor의 `WHERE id < 999980`은 B-Tree가 가장 잘하는 **값 기반 탐색(seek)**이야. 트리 높이만큼만 내려가면 바로 찾아 (보통 3~4단계).\n\n---\n\n### Q11-3: \"Offset이 100만행을 스캔하고 버린다\"는게 무슨 소리?\n\n**현실 비유**: 줄 서있는 100만 명 중에 \"100만 1번째부터 20명 나오세요\"라고 하면, 앞에 100만 명이 한 명씩 \"저 아니에요\"하고 지나가야 해. 100만 1번째 사람이 손들 수 있으려면.\n\n#### DB 엔진 내부 동작 (단계별)\n\n```sql\nSELECT * FROM posts ORDER BY id DESC LIMIT 20 OFFSET 1000000;\n```\n\n**Step 1: 실행 계획 수립**\n- 옵티마이저가 `ORDER BY id DESC`를 보고 PK 인덱스를 역순으로 읽기로 결정\n\n**Step 2: 인덱스 스캔 시작**\n- B-Tree 인덱스의 가장 마지막(가장 큰 id) 리프 노드로 이동\n- 여기서부터 역순으로 하나씩 읽기 시작\n\n**Step 3: Offset 카운팅 (이게 비용이 큰 부분!)**\n```\n행 1 읽음 (id=2000000) → offset 카운터: 1/1000000 → 결과에 안 넣음\n행 2 읽음 (id=1999999) → offset 카운터: 2/1000000 → 결과에 안 넣음\n행 3 읽음 (id=1999998) → offset 카운터: 3/1000000 → 결과에 안 넣음\n...\n행 1000000 읽음 (id=1000001) → offset 카운터: 1000000/1000000 → 결과에 안 넣음\n```\n\n여기서 **\"읽음\"이란 무엇인가?**\n- 인덱스 리프 노드를 하나씩 탐색 (인덱스 리프 노드 간 연결 리스트를 따라감)\n- 각 엔트리에서 해당 행의 실제 데이터가 필요하면 테이블 데이터 페이지까지 접근 (random I/O)\n- `SELECT *`이므로 **매 행마다 실제 데이터 페이지를 읽어야** 함 ⚠️\n\n💡 만약 `SELECT id`만 했다면 인덱스에 id가 있으니 테이블 접근 없이 인덱스만 스캔 (Covering Index). 그래도 100만 개를 세는 건 마찬가지.\n\n**Step 4: 실제 결과 수집**\n```\n행 1000001 읽음 (id=1000000) → LIMIT 카운터: 1/20 → ✅ 결과에 포함!\n행 1000002 읽음 (id=999999)  → LIMIT 카운터: 2/20 → ✅ 결과에 포함!\n...\n행 1000020 읽음 (id=999981)  → LIMIT 카운터: 20/20 → ✅ 결과에 포함!\n```\n\n**Step 5: 완료 → 결과 반환**\n\n#### 왜 \"100만번째에서 바로 시작\" 못하나?\n\n> B-Tree 인덱스는 **값(value) 기반** 자료구조이지, **위치(position) 기반**이 아니야.\n\nB-Tree는 \"id=999980인 노드 어디있어?\" → O(log n)으로 바로 찾을 수 있어.\n하지만 \"100만 번째 노드 어디있어?\" → 이건 **몰라**. 세봐야 알아.\n\n이건 배열(Array)과 연결 리스트(Linked List)의 차이와 비슷해:\n- 배열: `arr[1000000]` → 바로 접근 (O(1))\n- 연결 리스트: 1000000번째 → 처음부터 하나씩 따라가야 (O(n))\n\nB-Tree 리프 노드들은 **연결 리스트**처럼 연결되어 있어. 그래서 \"N번째\"를 찾으려면 처음부터 세야 해.\n\n---\n\n### Q11-4: Offset/Limit으로 무한 스크롤 못 만드나?\n\n✅ **네 말이 맞아. \"못 만든다\"가 아니라 \"성능 문제가 있다\"가 정확해.**\n\nOffset 기반으로 무한 스크롤 완벽하게 만들 수 있어:\n\n```javascript\n// 프론트엔드\nlet page = 0;\nconst size = 20;\n\nasync function loadMore() {\n    const response = await fetch(`/api/posts?page=${page}&size=${size}`);\n    const data = await response.json();\n    appendToList(data.content);\n    page++;\n}\n```\n\n이거 잘 동작해. 문제는 **성능**이야:\n\n| 스크롤 위치 | Offset 값 | 스캔하는 행 수 | 체감 속도 |\n|-----------|-----------|-------------|----------|\n| 1페이지 | 0 | 20 | ⚡ 즉시 |\n| 50페이지 | 1,000 | 1,020 | ⚡ 빠름 |\n| 500페이지 | 10,000 | 10,020 | ✅ 괜찮음 |\n| 5,000페이지 | 100,000 | 100,020 | ⚠️ 느려짐 |\n| 50,000페이지 | 1,000,000 | 1,000,020 | ❌ 매우 느림 |\n\n추가로 **데이터 정합성 문제**도 있어:\n\n사용자가 1페이지를 보는 동안 새 글이 5개 추가되면:\n- 2페이지 요청 시 `OFFSET 20`인데, 새 글 5개가 앞에 들어와서\n- 1페이지에서 봤던 글 5개가 2페이지에 또 나옴 (중복!)\n\nCursor 기반은 `WHERE id < lastId`라서 이 문제가 없어.\n\n#### 결론\n\n| 구분 | Offset 기반 | Cursor 기반 |\n|------|-----------|------------|\n| 무한 스크롤 구현 | ✅ 가능 | ✅ 가능 |\n| 초반 성능 | ✅ 빠름 | ✅ 빠름 |\n| 후반 성능 | ❌ 점점 느려짐 | ✅ 일정함 |\n| 데이터 중복/누락 | ⚠️ 발생 가능 | ✅ 없음 |\n| 페이지 번호 | ✅ 가능 | ❌ 불가 |\n| 특정 페이지 점프 | ✅ 가능 | ❌ 불가 |\n\n---\n\n## Q12. Spring Boot에서 트랜잭션 관리의 핵심 원칙\n\n### Q12-1: Service 계층에 @Transactional을 선언하는 이유\n\n**현실 비유**: 은행에서 \"계좌이체\"라는 업무를 생각해봐.\n\n- **Controller** = 은행 창구 직원. 고객의 요청을 접수하고 결과를 전달하는 역할\n- **Service** = 실제 업무 처리 담당자. \"A 계좌에서 빼고, B 계좌에 넣고, 이체 기록 남기기\"를 하나의 업무로 묶어서 처리\n- **Repository** = 금고 담당. 돈을 넣거나 빼는 단일 작업만 함\n\n#### Controller에 선언하면?\n\n```java\n@Transactional  // ❌ 여기?\n@PostMapping(\"/transfer\")\npublic ResponseEntity<?> transfer(@RequestBody TransferRequest req) {\n    transferService.withdraw(req.from(), req.amount());\n    transferService.deposit(req.to(), req.amount());\n    // HTTP 응답 만들기, 로깅 등도 트랜잭션 안에 포함됨\n    return ResponseEntity.ok(\"완료\");\n}\n```\n\n문제점:\n- ⚠️ 트랜잭션 범위가 너무 넓어짐 (HTTP 응답 직렬화까지 포함)\n- ⚠️ Connection을 오래 잡고 있음 → Connection Pool 고갈 위험\n- ⚠️ Controller는 웹 요청/응답 처리가 역할인데, 비즈니스 트랜잭션 경계까지 책임지면 **단일 책임 원칙(SRP) 위반**\n- ⚠️ 같은 Service를 다른 Controller, 스케줄러, 메시지 리스너에서 호출할 때 트랜잭션이 보장 안 됨\n\n#### Repository에 선언하면?\n\n```java\n@Repository\npublic class AccountRepository {\n    @Transactional  // ❌ 여기?\n    public void withdraw(Long accountId, BigDecimal amount) { ... }\n    \n    @Transactional  // ❌ 여기?\n    public void deposit(Long accountId, BigDecimal amount) { ... }\n}\n```\n\n문제점:\n- ⚠️ withdraw와 deposit이 **각각 별도의 트랜잭션**으로 실행됨\n- ⚠️ withdraw 성공 후 deposit에서 실패하면? → 돈이 사라짐! 💸\n- ⚠️ 비즈니스 로직의 원자성(Atomicity)을 보장할 수 없음\n\n#### Service에 선언하는 것이 맞는 이유 ✅\n\n```java\n@Service\npublic class TransferService {\n    @Transactional  // ✅ 여기!\n    public void transfer(Long fromId, Long toId, BigDecimal amount) {\n        accountRepository.withdraw(fromId, amount);   // 같은 트랜잭션\n        accountRepository.deposit(toId, amount);       // 같은 트랜잭션\n        transferLogRepository.save(new TransferLog(...)); // 같은 트랜잭션\n        // 하나라도 실패하면 전부 롤백!\n    }\n}\n```\n\n- ✅ 비즈니스 로직 단위로 트랜잭션 경계가 정해짐\n- ✅ 여러 Repository 작업을 하나의 트랜잭션으로 묶을 수 있음\n- ✅ Controller/스케줄러/메시지 리스너 어디서 호출해도 트랜잭션 보장\n\n---\n\n### Q12-2: 더티 체킹(Dirty Checking)이 뭐야?\n\n**현실 비유**: 시험 감독관이 학생들의 답안지를 감시하는 거야.\n\n1. 시험 시작할 때 각 학생의 답안지를 **복사**해둠 (원본 스냅샷)\n2. 시험 끝나면 원본 스냅샷과 현재 답안지를 **비교**\n3. 바뀐 부분이 있으면 → \"이 학생 답안 수정했음!\"으로 기록 → **UPDATE SQL 자동 생성**\n\n#### 단계별 설명\n\n```java\n@Transactional\npublic void updateUserName(Long userId, String newName) {\n    User user = userRepository.findById(userId).get(); // Step 1\n    user.setName(newName);                              // Step 2\n    // save() 안 해도 됨!                                // Step 3\n}\n```\n\n**Step 1: 엔티티 로딩**\n- DB에서 User를 가져옴\n- Hibernate가 이 User의 **스냅샷(사본)을 1차 캐시에 저장**\n- 이때 상태: `name = \"철수\"` (원본 스냅샷도 `name = \"철수\"`)\n\n**Step 2: 필드 변경**\n- `user.setName(\"영희\")` → 엔티티의 name이 \"영희\"로 바뀜\n- 하지만 스냅샷은 여전히 `name = \"철수\"`\n\n**Step 3: 트랜잭션 커밋 시점 (flush)**\n- Hibernate가 1차 캐시에 있는 모든 엔티티를 순회\n- 각 엔티티의 **현재 상태**와 **로딩 시 스냅샷**을 필드 하나하나 비교\n- `name: \"철수\" → \"영희\"` → 변경 감지! → **UPDATE SQL 자동 생성 및 실행**\n\n```sql\nUPDATE users SET name = '영희' WHERE id = 1;\n```\n\n#### readOnly = true면?\n\n```java\n@Transactional(readOnly = true)\npublic User getUser(Long userId) {\n    return userRepository.findById(userId).get();\n}\n```\n\n- Hibernate가 **스냅샷을 저장하지 않음** (어차피 수정 안 할 거니까)\n- 트랜잭션 커밋 시 **비교 작업(dirty checking)을 건너뜀**\n- 결과: **메모리 절약** (스냅샷 안 만듦) + **CPU 절약** (비교 안 함)\n\n💡 이건 마치 \"이 시험은 채점 안 해도 돼요~ 연습이에요~\"라고 감독관에게 말하는 것과 같아. 감독관이 답안지 복사도 안 하고, 비교도 안 하니까 편해지는 거지.\n\n---\n\n### Q12-3: Checked Exception 롤백\n\n#### rollbackFor에 정의된 예외가 아니면 커밋하는가?\n\nSpring의 기본 롤백 규칙:\n\n| 예외 타입 | 기본 동작 | 예시 |\n|----------|----------|------|\n| **Unchecked Exception** (RuntimeException, Error) | ✅ 자동 롤백 | `NullPointerException`, `IllegalArgumentException` |\n| **Checked Exception** (Exception) | ❌ 커밋됨 | `IOException`, `SQLException`, 커스텀 Checked Exception |\n\n그러니까 맞아. `rollbackFor`에 명시하지 않은 Checked Exception이 발생하면 **커밋해버려**. ⚠️\n\n#### 여러 Exception을 선언하고 싶으면?\n\n```java\n// 방법 1: 배열로 여러 개 지정\n@Transactional(rollbackFor = {IOException.class, CustomException.class, PaymentException.class})\npublic void doSomething() { ... }\n\n// 방법 2: 최상위 Exception으로 한방에 (모든 예외에서 롤백)\n@Transactional(rollbackFor = Exception.class)\npublic void doSomething() { ... }\n\n// 방법 3: 롤백하지 않을 것만 제외\n@Transactional(\n    rollbackFor = Exception.class,\n    noRollbackFor = BusinessWarningException.class  // 이것만 롤백 안 함\n)\npublic void doSomething() { ... }\n```\n\n💡 실무 팁: 많은 팀에서 `rollbackFor = Exception.class`를 기본으로 쓰고, 롤백하지 않을 것만 `noRollbackFor`로 빼는 전략을 사용해.\n\n#### 왜 Spring은 기본적으로 Checked Exception에서 롤백하지 않는가?\n\n이건 **Java의 예외 설계 철학** 때문이야.\n\n**Step 1: Java의 의도**\n- **Checked Exception** = \"예상 가능한, 복구 가능한 상황\" (파일 못 찾음, 네트워크 끊김)\n- **Unchecked Exception** = \"프로그래밍 오류, 복구 불가능한 상황\" (null 참조, 배열 범위 초과)\n\n**Step 2: Spring의 철학**\n- Checked Exception은 \"비즈니스적으로 예상된 상황\"이니까, 개발자가 **catch해서 적절히 처리**할 것으로 기대\n- 예: 결제 시 잔액 부족 → `InsufficientBalanceException` (Checked) → 롤백하지 말고, 사용자에게 \"잔액 부족\" 알림만 보낼 수도 있잖아?\n- Unchecked Exception은 \"예상 못 한 오류\"니까 → 무조건 롤백이 안전\n\n**Step 3: 현실과의 괴리**\n- 하지만 실제로는 Checked Exception도 롤백해야 하는 경우가 대부분\n- 그래서 Spring 커뮤니티에서도 `rollbackFor = Exception.class`를 권장하는 추세\n- 사실 EJB(Enterprise JavaBeans) 시절의 관례를 Spring이 따른 것이기도 해\n\n**비유**: \"교통사고(Unchecked)가 나면 보험처리(롤백) 자동이지만, 타이어 펑크(Checked)는 네가 알아서 수리하든 견인하든 결정해라\"라는 철학이야. 근데 현실에서는 펑크도 보험처리 하고 싶잖아? 그래서 `rollbackFor = Exception.class`로 설정하는 거지.\n\n---\n\n## Q13. Connection Pool과 HikariCP\n\n### Q13-1: maximum-pool-size 공식 (코어 × 2 + 디스크 수)의 이유\n\n**공식**: `pool size = (core_count * 2) + effective_spindle_count`\n\n이 공식은 PostgreSQL wiki에서 유래한 것으로, HikariCP 공식 문서에서도 인용하고 있어.\n\n#### 왜 코어 × 2인가?\n\n**현실 비유**: 레스토랑 주방에 요리사(코어)가 4명이야.\n\n- 요리사 1명이 스테이크를 굽고 있어 (DB I/O 대기 중)\n- 스테이크가 구워지는 동안 다른 주문(쿼리)을 처리할 수 있어\n- 그래서 요리사 1명당 최대 2개의 주문을 동시에 처리 가능\n- 4명 × 2 = 8개의 동시 주문\n\n기술적으로:\n- CPU 코어 1개는 한 번에 1개의 쓰레드만 실행\n- 하지만 **DB 작업은 디스크 I/O가 포함**되니까, I/O 대기 중에 다른 쓰레드를 처리할 수 있어\n- 그래서 코어당 2개의 Connection이 적절한 거야\n- 코어 × 2 = **CPU가 I/O 대기 시간에 다른 커넥션을 처리하는 것을 반영**\n\n#### \"더 많아도 되는거 아니야?\"에 대한 답\n\n❌ 아니야. 여기서 중요한 건 **DB 서버 측의 처리 능력**이야.\n\n- Connection이 아무리 많아도, **DB 서버의 CPU 코어**가 동시에 처리할 수 있는 쿼리 수는 한정적\n- Connection이 늘어나면:\n  1. DB 서버에서 더 많은 쓰레드가 동시에 동작 → **DB 서버의 컨텍스트 스위칭** 증가\n  2. 각 Connection은 DB 서버에서 **메모리를 소비** (MySQL: 커넥션당 약 ~10MB)\n  3. Lock contention이 증가 (더 많은 트랜잭션이 같은 행을 동시에 접근)\n\n💡 핵심: \"IO 작업이라 금방 쓰고 반환한다\"고 생각할 수 있지만, **DB 서버 입장에서는 그 커넥션들이 동시에 쿼리를 던지는 것**이야. DB 서버의 CPU도 한계가 있어!\n\n#### effective_spindle_count는 뭐야?\n\n**Spindle** = 하드 디스크의 **회전축(물리적 디스크)**을 말해.\n\n- HDD 1개 = spindle 1개 → `effective_spindle_count = 1`\n- RAID 구성으로 HDD 4개 → `effective_spindle_count ≈ 4`\n- SSD는 spindle이 없지만, 동시 I/O 처리 능력을 감안해서 보통 1로 둠 (SSD가 이미 충분히 빠르니까)\n\n디스크가 많으면 **동시에 여러 I/O를 처리**할 수 있으니까, 그만큼 더 많은 Connection이 유의미한 작업을 할 수 있어서 pool size에 더해주는 거야.\n\n⚠️ 이 공식은 **출발점(starting point)**이야. 실제로는 부하 테스트를 통해 조정해야 해. 현대의 NVMe SSD 환경에서는 이 공식보다 **DB 서버의 CPU 코어 수**가 더 중요한 기준이 되기도 해.\n\n---\n\n### Q13-2: Pool이 너무 크면 컨텍스트 스위칭 오버헤드 — 사용자 시나리오 분석\n\n사용자 시나리오를 정리하면:\n- DB: 동기 방식\n- Connection Pool: 4개\n- 코어: 4개\n- 스레드: 8개\n\n#### 사용자 분석이 맞는 부분 ✅\n\n> \"스레드 4개가 코어 4개에 할당돼서 connection pool에서 4개의 connection을 가져갔고, DB 처리가 되고 있는 동안 block이 되니 다른 스레드들이 코어에서 스케줄링이 되고 이때 context switching이 발생\"\n\n✅ 맞아. 애플리케이션 서버(Spring Boot)에서의 동작은 정확하게 이해하고 있어.\n\n> \"이건 connection pool과는 관계가 없지 않아?\"\n\n✅ **이것도 맞아!** 애플리케이션 서버 측의 컨텍스트 스위칭은 **스레드 수** 때문이지, connection pool 크기 때문이 아니야.\n\n#### 사용자가 놓친 핵심 포인트 ⚠️\n\n**\"Pool size가 크면 문제가 되는 곳은 애플리케이션 서버가 아니라 DB 서버야!\"**\n\n자, 시나리오를 바꿔볼게. Connection Pool을 **100개**로 늘렸다고 해보자.\n\n**애플리케이션 서버 (Spring Boot):**\n```\n스레드 200개 → 동시에 100개가 Connection을 획득 → DB에 쿼리 100개를 동시에 전송\n```\n\n**DB 서버 (MySQL/PostgreSQL):**\n```\n100개의 Connection이 동시에 쿼리 실행을 요청\n→ DB 서버의 CPU 코어가 8개라면?\n→ 8개만 동시 실행 가능, 나머지 92개는 대기\n→ DB 서버 내부에서 92개의 쓰레드가 스케줄링됨\n→ DB 서버에서 컨텍스트 스위칭 오버헤드 발생!\n→ Lock contention 증가 (같은 테이블/행을 더 많은 트랜잭션이 접근)\n→ 메모리 사용량 증가 (Connection당 메모리 할당)\n→ 결과: 모든 쿼리가 느려짐 📉\n```\n\n#### 비유로 설명\n\n**레스토랑 비유**:\n- 애플리케이션 서버 = 웨이터들 (주문 받는 사람)\n- Connection Pool = 주방에 전달하는 주문서 슬롯\n- DB 서버 = 주방 (요리사 = DB CPU 코어)\n\n웨이터가 아무리 많아도 주방 요리사가 4명이면:\n- 주문서 슬롯(pool)이 4개 → 요리사가 딱 맞게 일함 ✅\n- 주문서 슬롯(pool)이 100개 → 주문 100개가 동시에 주방에 쌓임 → 요리사들이 이 주문 저 주문 왔다갔다(컨텍스트 스위칭) → 모든 요리가 느려짐 ❌\n\n#### 정리\n\n| 구분 | Pool 크기 영향 |\n|------|-------------|\n| 앱 서버 컨텍스트 스위칭 | ❌ 관계 없음 (스레드 수에 의존) — 사용자 분석 맞음 |\n| DB 서버 컨텍스트 스위칭 | ✅ 직접적 영향 (동시 Connection = DB 동시 쿼리) |\n| DB 서버 메모리 | ✅ Connection당 메모리 소비 |\n| DB 서버 Lock contention | ✅ 동시 트랜잭션 증가 → 충돌 증가 |\n| 앱 서버 Connection 대기 | Pool 작으면 대기 증가, 크면 대기 감소 |\n\n---\n\n### Q13-3: HikariCP 이름의 의미\n\n#### Hikari = 光 (ひかり, 빛)\n\n✅ **Hikari는 일본어로 \"빛(光)\"을 의미해.** \"빛처럼 빠른 Connection Pool\"이라는 뜻이야.\n\nCP는 맞아, **Connection Pool**의 약자.\n\nHikariCP는 2012년경 Brett Wooldridge가 만든 Java Connection Pool 라이브러리로, 기존의 C3P0, DBCP, Tomcat Pool보다 **월등히 빠른 성능**을 자랑해서 Spring Boot 2.0부터 **기본 Connection Pool**로 채택되었어.\n\n#### Python의 uvicorn, gunicorn에 hikari 설정이 있나?\n\n❌ **아니야, 착각이 맞아.**\n\n| 도구 | 언어 | 역할 | Connection Pool |\n|------|------|------|-----------------|\n| HikariCP | Java/JVM | DB Connection Pool | 본인이 곧 CP |\n| uvicorn | Python | ASGI 웹 서버 | DB CP 아님 |\n| gunicorn | Python | WSGI 웹 서버 | DB CP 아님 |\n\nPython에서 DB Connection Pool을 쓰려면:\n- **SQLAlchemy**: 내장 Connection Pool (QueuePool이 기본)\n- **asyncpg**: PostgreSQL 전용 async pool\n- **aiomysql**: MySQL 전용 async pool\n\nuvicorn/gunicorn은 **웹 서버(HTTP 요청 처리)**이지 DB Connection Pool이 아니야. 너가 FastAPI/SQLAlchemy를 쓸 때 `create_engine(pool_size=5, max_overflow=10)` 같은 설정을 한 적 있을 수 있는데, 그게 SQLAlchemy 내장 Pool 설정이야.\n\n혹시 **Hikari**와 **worker** 설정을 혼동한 게 아닐까? gunicorn의 `--workers 4` 같은 설정이 HikariCP의 pool-size와 비슷한 \"리소스 개수 조절\"이라서 헷갈릴 수 있어.\n\n---\n\n## Q14. 동시성 문제 해결 방법\n\n### Q14-1: Optimistic Lock이 충돌이 적을 때 성능이 우수한 이유\n\n**현실 비유**: 구글 문서(Google Docs)에서 협업하는 상황이야.\n\n- **Optimistic Lock**: \"일단 다 같이 편집하고, 저장할 때 충돌 체크하자!\" → 충돌이 거의 없으면 모두가 자유롭게 작업 ✅\n- **Pessimistic Lock**: \"내가 편집하는 동안 아무도 건드리지 마!\" → 한 사람이 끝날 때까지 나머지가 대기 ⏳\n\n#### 내부 동작 단계별\n\n**Step 1: 엔티티에 version 필드 추가**\n```java\n@Entity\npublic class Product {\n    @Id\n    private Long id;\n    private String name;\n    private int stock;\n    \n    @Version  // Optimistic Lock용 버전 필드\n    private Long version;\n}\n```\n\n**Step 2: 데이터 읽기**\n```sql\nSELECT id, name, stock, version FROM products WHERE id = 1;\n-- 결과: id=1, name='키보드', stock=100, version=3\n```\n- 이때 **아무런 락을 걸지 않음** ✅\n- DB 자원을 점유하지 않음\n\n**Step 3: 비즈니스 로직 수행 (애플리케이션에서)**\n```java\nproduct.setStock(product.getStock() - 1); // 99로 변경\n```\n- 이 동안 다른 트랜잭션도 자유롭게 읽기/쓰기 가능\n\n**Step 4: 저장 시 버전 체크 (UPDATE)**\n```sql\nUPDATE products \nSET stock = 99, version = 4  -- version + 1\nWHERE id = 1 AND version = 3;  -- 읽었을 때의 version으로 조건!\n```\n\n**경우 1: 충돌 없음 (대부분의 경우)**\n- `WHERE version = 3` → 매칭됨 → UPDATE 성공 (affected rows = 1)\n- 완료! 락 없이 빠르게 처리됨 ⚡\n\n**경우 2: 충돌 발생 (다른 누군가가 먼저 업데이트)**\n- 다른 트랜잭션이 이미 version을 4로 올려버림\n- `WHERE version = 3` → 매칭 안 됨 → affected rows = 0\n- JPA가 `OptimisticLockException` 발생\n- 애플리케이션이 재시도 또는 에러 반환\n\n#### 왜 충돌이 적을 때 성능이 좋은가?\n\n| 단계 | Optimistic Lock | Pessimistic Lock |\n|------|----------------|-----------------|\n| 읽기 | 일반 SELECT (락 없음) | `SELECT ... FOR UPDATE` (행 잠금) |\n| 처리 중 | 다른 트랜잭션 자유 | 다른 트랜잭션 대기(block) |\n| 쓰기 | UPDATE with version check | UPDATE (락 해제) |\n| DB 자원 | 거의 안 씀 | 락 관리자가 메모리/CPU 사용 |\n| Connection 점유 시간 | 짧음 | 김 (락 유지 동안 점유) |\n\n충돌이 적으면 → Optimistic은 거의 항상 \"경우 1\"로 동작 → 락 오버헤드 제로에 가까움!\n\n---\n\n### Q14-2: Pessimistic Lock (FOR UPDATE) 심화\n\n#### 충돌이 잦은 상황에서 쓰나? ✅\n\n맞아. 동일한 데이터를 **여러 트랜잭션이 동시에 자주 수정**할 때 사용해.\n\n#### Optimistic Lock보다 빠른가?\n\n**상황에 따라 다르다!** 이게 바로 trade-off야.\n\n#### 데드락과 성능 저하 이유\n\n**데드락 시나리오**:\n```\n트랜잭션 A: 행 1 잠금 → 행 2 잠금 시도 (대기)\n트랜잭션 B: 행 2 잠금 → 행 1 잠금 시도 (대기)\n→ 둘 다 영원히 대기 → 데드락!\n```\n\n**성능 저하 이유**:\n1. 락을 잡는 동안 **다른 트랜잭션이 블로킹됨** → 처리량(throughput) 감소\n2. 블로킹 시간만큼 **Connection을 오래 점유** → Connection Pool 고갈 위험\n3. 락 관리 자체가 DB 서버의 **메모리와 CPU를 소비**\n\n#### 📌 Trade-off 정리 (핵심)\n\n이걸 **깨진 유리창 수리비** 비유로 생각해보자.\n\n- **Optimistic Lock** = \"유리창이 깨질 확률이 낮으니 보험 안 들어. 깨지면 그때 수리비(재시도 비용) 내자.\"\n- **Pessimistic Lock** = \"유리창이 자주 깨지니까 보험료(락 비용)를 매달 내자. 깨져도 수리비는 안 들어.\"\n\n| 상황 | Optimistic | Pessimistic | 승자 |\n|------|-----------|-------------|------|\n| 충돌률 1% | 재시도 거의 없음 ✅ | 매번 락 비용 ❌ | Optimistic |\n| 충돌률 5% | 가끔 재시도 | 매번 락 비용 | Optimistic (보통) |\n| 충돌률 20% | 재시도 빈번 ⚠️
