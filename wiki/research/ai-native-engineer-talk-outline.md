# AI Native Engineer 발표 골격 (개발자 청중 · 20~30분 · 교육)

> 근거 노트: [[ai-native-engineer-role]] · 보조: [[engineering-taste-concept]]
> 발표 톤: **중립 소개** — 하이프도 거품론도 각각 검증 대상으로 둔다. 개발자 청중에게는 이 균형이 곧 신뢰다.
> ⚠️ **벤더 수치 규칙**: SWE-bench 점수·"+20~30% 연봉" 등은 슬라이드에 올릴 때 반드시 "벤더 자체 보고 / 단일 출처" 라벨을 붙이고 METR 같은 비-벤더 데이터와 **대비**시킨다. (근거 노트에 1차 출처가 없는 수치는 발표에 쓰지 않는다.)

## ① 훅 (2분)

- **핵심 메시지**: "AI Native Engineer" 채용은 급증하는데, 정작 METR 실험은 "AI가 **경험 개발자 표본(N=16, 평균 5년 경력)**에서 작업을 19% *느리게* 했다"고 한다. 이 역설에서 시작하자. (⚠️ "숙련" 같은 강한 일반화 금지 — 작은 특정 표본임을 ①에서부터 정직하게.)
- **근거**: [[ai-native-engineer-role]] §⚖️ 비-벤더 검증(METR), §5 한국 시장(무신사·그루우)
- **출처**: METR arXiv 2507.09089 / 무신사 뉴스룸 2026-01-16
- **화면**: 좌우 대비 — "채용 공고 급증 📈" vs "METR 19% 슬로우다운 📉"
- **전달 팁**: 답을 주지 말고 질문만 남긴 채 ②로.

## ② 정의 (4분)

- **핵심 메시지**: AI Native Engineer = **AI 코딩 에이전트를 인프라로 오케스트레이션해 소프트웨어를 만드는 사람**. 모델을 *만드는* ML/AI Engineer도, 책임을 *위임하는* Vibe Coder도 아니다.
- **근거**: [[ai-native-engineer-role]] §1 정의 + 인접 용어 지도, §2 직군 비교표
- **출처**: Karpathy(Vibe Coder 2025-02), agentic-engineering-jobs.com, Howdy
- **화면**: 4형제 책임 구조 축 다이어그램 (Vibe Coder ↔ AI Native ↔ Agentic / AI Engineer)
- **전달 팁**: "ML Engineer는 엔진 설계, AI Engineer는 엔진 장착, AI Native는 로봇 부대 감독" 비유.

## ③ 실제 업무 — 하루가 어떻게 바뀌나 (6분) ★최우선

- **핵심 메시지**: 무게중심이 "코드 타이핑"에서 **"spec 정의 → 에이전트 실행 → 검증 → 리뷰"** 루프로 이동. **이건 벤더가 아니라 독립 실무자들이 수렴한 패턴**이다.
- **근거**: [[ai-native-engineer-role]] §4 Agent Wrangler + 🔄 워크플로 루프
- **출처**: Harper Reed(harper.blog), Addy Osmani(addyosmani.com, Always/Ask/Never), Sakasegawa(nyosegawa.com 4패턴) — **전원 비-벤더**
- **화면**: spec→agent→verify→review 루프 애니메이션 + `CLAUDE.md`/spec.md 실제 예시 코드
- **전달 팁**: 여기가 개발자 청중이 가장 몰입하는 구간. 실제 `CLAUDE.md` 한 장을 띄워 "이게 새 소스코드"라고. "70/30" 같은 숫자는 쓰지 말 것(측정치 아님) — "무게중심 이동"으로 정성 표현.

## ④ 자격요건 — 실제 JD가 말하는 것 (5분)

- **핵심 메시지**: 실제 공고(한국 그루우·와이큐, 미국 Libra·Ramp·Legion)가 공통으로 요구하는 건 ① AI 코딩 도구 실무 ② LLM API ③ **AI 생성 코드 검증 능력**.
- **근거**: [[ai-native-engineer-role]] §5 한국+해외 JD, §3 6대 역량(Augment)
- **출처**: 원티드(그루우·와이큐), Libra/Ramp/Legion 1차 공고 / Augment 6대 역량(벤더 — 라벨)
- **화면**: 실제 JD 캡처 3~4장 + 공통 요건 하이라이트
- **전달 팁**: ⚠️ "표본 6건, 직군명 미통일 → 비율(%)이 아니라 경향". Augment 6대 역량은 "벤더 채용 프레임"이라 명시. 그루우 *"컨텍스트·하네스 엔지니어링 경계"* 문구를 인용하면 임팩트.

## ⑤ 도구 지도 (3분)

- **핵심 메시지**: 도구는 경쟁이 아니라 **워크플로 레이어 분담**. Claude Code(터미널 에이전트)/Cursor(IDE)/Copilot(엔터프라이즈)/Codex(클라우드)/Augment(컨텍스트 엔진).
- **근거**: [[ai-native-engineer-role]] §7 도구 생태계 지도
- **출처**: 각 공식 문서 + dev.to 독립 비교 / MCP(Anthropic), 하네스 5계층
- **화면**: 도구 6종 포지션 매트릭스 (카테고리 × 강점)
- **전달 팁**: ⚠️ "SWE-bench 80.8%" 같은 점수는 띄우더라도 **"벤더 자체 발표, 버전 따라 출렁"** 라벨 필수. 도구 추천보다 "레이어 이해"에 집중(개별 도구 변동에 덜 취약). ⚠️ **신선도**: 도구 판도는 빠르게 변함(예: Windsurf 인수). 슬라이드에 "2026-05 기준 스냅샷" 명기하고 **발표 직전 6종 포지션 1회 재확인**.

## ⑥ 냉정한 현실 (4분)

- **핵심 메시지**: 벤더는 "생산성 폭증"을 말하지만 독립 데이터는 더 신중하다. **METR 19% 슬로우다운(조건부)**, Stanford 주니어 고용 위축(인과 미확정), MIT 95% zero-value(기업 ROI 차원).
- **근거**: [[ai-native-engineer-role]] §⚖️ 비-벤더 독립 검증 전체
- **출처**: METR, Stanford DEL, US BLS, MIT(2025-12), Gartner Hype Cycle — **전원 비-벤더**
- **화면**: 벤더 주장 vs 비-벤더 실증 좌우 대비표
- **전달 팁**: ⚠️ METR는 "N=16·대형 OSS·2025초 도구" 조건을 *먼저* 인정. "AI 무용론"이 아니라 "맥락 의존". Stanford -20%는 금리·팬데믹 교란 인정. **양방향 균형이 핵심.** ⚠️ **METR(개발자 개인 작업 속도)와 MIT 95%(기업 전사 ROI)는 측정 층위가 다름** — "개인 코딩은 METR, 기업 효과는 MIT"로 명확히 구분해 청중 혼동 방지.

## ⑦ 자가진단 — "나는/우리 팀은 AI Native인가?" (3분)

- **핵심 메시지**: 타이틀이 아니라 **행동**으로 판별. 아래 체크리스트로 자가평가.
- **근거**: [[ai-native-engineer-role]] §3 6대 역량 + §5 실제 JD 요건
- **화면**: 체크리스트 (예/아니오) + "시작하는 법" 3단계
- **전달 팁**: 청중이 스스로 체크하게 잠깐 멈춤. 마무리 임팩트.

### ✅ AI Native Engineer 자가진단 체크리스트

> 각 항목은 노트의 6대 역량 또는 실제 JD 요건에 근거. "예"가 많을수록 AI Native 성향.

| # | 체크 항목 | 근거 |
| - | --------- | ---- |
| 1 | AI 코딩 도구(Claude Code/Cursor/Codex 등)로 **실제 배포된 결과물**을 만들어 봤다 | 실제 JD 공통 1순위 |
| 2 | 에이전트에게 줄 **컨텍스트(`CLAUDE.md`·spec)를 의도적으로 설계**한다 | Context Curation / 그루우 JD |
| 3 | AI가 생성한 코드를 **정확성·보안·아키텍처 적합성** 기준으로 비판적으로 검증한다 | Agent Leverage / Libra JD |
| 4 | "그럴듯하지만 틀린(plausible-but-wrong)" 출력을 **감지하고 개입**한 경험이 있다 | Verification Ownership |
| 5 | 코드를 짜기 전에 **올바른 문제인지** 먼저 따진다 (Product/Outcome Taste) | 6대 역량 #1 / [[engineering-taste-concept]] |
| 6 | 단기 동작보다 **프로덕션 지속성·장기 트레이드오프**를 우려한다 | System Judgment |
| 7 | 작업을 **에이전트가 처리 가능한 단위로 분해**할 수 있다 | Task Decomposition |
| 8 | 검증 게이트(테스트·CI·Hooks 등) **하네스를 직접 설계**해 봤다 | Harness Design / 그루우 JD |
| 9 | 새 도구·워크플로가 나오면 **빠르게 실험하고 기존 방식을 버린다** | Learning Velocity |
| 10 | 내 코드 조각만이 아니라 **결과 전체를 소유**하고 장애물을 제거한다 | Ownership |

**시작하는 법 (3단계)**:
1. 사이드 프로젝트 1개를 Claude Code로 처음부터 끝까지 빌드 → `CLAUDE.md` 공개
2. 팀 워크플로 개선 1건 (개인 GitHub보다 강한 시그널)
3. 트레이드오프 판단을 매번 기록 (taste 훈련 → [[engineering-taste-concept]])

## ⑧ Q&A

- 예상 질문 대비:
  - "정말 새 직군인가, 시니어 재브랜딩인가?" → ⑥ 미결 논쟁([[ai-native-engineer-role]] §용어 회의)
  - "AI가 일자리를 줄이나?" → Stanford 교란변수(금리·팬데믹 공동요인)
  - "어떤 도구부터?" → ⑤ 레이어 분담
  - **"신입/주니어는 어떻게 진입하나?"** (Stanford 주니어 -16~20% 직후 필연적 질문) → §Edge Cases 신입 진입 변화 + 무신사 6개월 인턴 전환형 + §🎯 권장 단기·중기 단계
  - **"19% 느려진다면 이 직군을 왜 하나?"** (①훅·⑥에서 자극한 역설의 해소) → §⚔️ Contradictions 종합판정("빨리 짜기가 아니라 오류를 거르는 판단") + METR 조건 한정(N=16·구버전·대형 OSS)
