핵심 평결 · TL;DR
  1. 마크다운이 AI에 유리하다는 전제는 대체로 맞지만, 흔히 도는 수치(토큰 80% 절감 등)는 과장됨.
  2. Confluence/JIRA를 마크다운·git·AI 네이티브로 대체하는 도구는 이미 다수 존재 → 직접 빌드 불필요.
  3. 목적(Atlassian 탈출·도입·기여·학습)에 가장 정합적인 단일 경로는 Forgejo(또는 Gitea) 도입 + MCP 서버 기여.
  4. AGPL 계열(Plane·Docmost)은 회사 법무 확인 전까지 보류 — Google 등은 AGPL을 전면 금지.
01

전제 검증 — "마크다운이 AI에 유리한가?"

출발 질문: "AI 시대엔 마크다운이 AI가 이해하기 좋은 언어다. Confluence/JIRA를 마크다운으로 대체한 툴이 있나? 없으면 만들려고 한다." 전제부터 검증했다.

마크다운이 HTML보다 토큰 효율적 (Cloudflare "80% 감소") 🟡 LIKELY 수치는 실재하나 단일 블로그 1개 표본 + 기준선이 raw HTML(짚인형). 실무의 정제 텍스트와는 격차가 줄어든다.
토큰 절감 = AI가 더 잘 이해한다 ⚪ REFUTED 비약 — 비용(토큰)과 이해도는 다른 축이며 통제 실험이 없다. 상관과 인과의 혼동.
llms.txt 채택이 1년 새 1,835% 증가 ⚪ REFUTED 1차 출처 추적 실패 → 인용 금지.
💡

비유 — 마크다운은 "택배에서 포장재를 뺀 편지"라 AI가 싸고 빠르게 읽는다(맞음). 하지만 "싸게 읽는다 ≠ 더 정확히 이해한다". 표·의존성 같은 구조는 순수 텍스트에서 오히려 잃을 수 있어, 똑똑한 도구는 마크다운 + DB 하이브리드(예: Beads의 SQLite)를 쓴다. → "AI 친화"는 선택의 한 축일 뿐, 실무에선 MCP·에이전트 지원 여부로 구체화해야 한다.

03

핵심 결론 ① — 대체재는 이미 존재한다

원 조사는 "마크다운+AI 네이티브 통합 제품은 빈 공간"이라 결론냈으나, 검증에서 부분 반증됐다.

⚠️

"위키+이슈+마크다운 통합"은 이미 존재 — Fossil(VCS+위키+티켓을 단일 바이너리, 공통 MD 인터프리터), GitLab·Gitea·Forgejo(MD 위키 + MD 이슈 + git). 남은 좁은 틈은 "풀위키 + 풀이슈 + 순수 MD + AI 네이티브" 4박자 동시충족뿐이고, 그조차 Plane·Huly·Backlog.md가 빠르게 메우는 중.

✅

직접 빌드는 정당화되지 않는다. 통합의 90%는 같은 git repo에 위키 폴더 + 이슈 폴더로 공짜이고, 남는 빈틈은 해자가 없다.

04

목적 정렬 — 빌드가 아니라 "도입 + 기여"

실제 목적: "만드는 게 목적이 아니다. 대체할 게 있으면 학습해서 회사에 도입 + contributor로 활동. 없으면 그때 만든다." + 회사 현재 스택 = Atlassian(Confluence/JIRA), 탈출 희망.

대체할 도구가 있는가?
├─ YES (← 검증: 다수 존재) → 학습 + 회사 도입 + 기여  ✅ 여기
└─ NO → 직접 빌드 (발동 안 함)

06

라이선스 리스크 — 회사 도입 결정 요인

회사 AGPL 정책이 미확인이므로 이게 후보를 가른다. (1차 출처 검증)

🟢GitLab CE · GiteaMIT permissive — 사내·SaaS·수정 자유. 단 GitLab은 ee/ proprietary 혼재 → CE-only 빌드 확인.
🟢ForgejoGPL-3.0 사내 도입 자유.
🟡HulyEPL-2.0 network 조항 없음(SaaS 안전), file-level weak copyleft → AGPL보다 관대.
🔴Plane · DocmostAGPL-3.0 §13 network 조항 + Google 등 대기업 공식 전면 금지. Docmost는 +EE 유료 +CLA.
⚠️

AGPL §13: 트리거 = 수정 AND 네트워크 사용자 동시. 미수정 self-host는 미발동이나 "users" 범위가 회색지대 → 법무 검토 필수.

07

최종 추천 (정정본)

1순위 · 도입 = 기여 = 학습을 한 우물로

Forgejo (또는 Gitea)

위키(Confluence) + 이슈(JIRA) + git을 한 데이터모델에 담은 통합 단일 도구. 협동조합형 비영리 거버넌스라 기여가 실제 반영되고, AI 공백은 "Forgejo용 MCP 서버" 기여로 직접 채우면 AI-era 학습이 도입 도구로 전이된다.

통합위키+이슈+git 단일 데이터모델 → "통합" 욕구 충족
라이선스GPL/MIT → AGPL 법무 블로커 회피
기여=복리도입한 그 도구에 기여 = 운영지식 = 코드지식
경량단일 Go 바이너리 (Plane 10+/Huly 14서비스 대비)
⚖️

트레이드오프 — Forgejo 위키는 git+MD지만 Confluence만큼 화려하진 않다(매크로·템플릿 없음). AI는 "지금"이 아니라 "당신이 기여해 채우는" 형태.

08

액션 플랜 — 순서가 중요하다

  1. 법무에 AGPL 정책 확인 (병렬 진행) → 통과 시 AI 강한 Plane도 후보로 부활.
  2. Forgejo 파일럿 단일 바이너리 셀프호스트 → Confluence 위키 1개 + JIRA 이슈 일부를 이전해 감 잡기.
  3. 첫 기여 good first issue 1개로 워크플로우 익히기 + CONTRIBUTING 정독.
  4. 핵심 기여 (AI-era) Forgejo 코어에 MCP 서버 RFC/구현 제안 — 현재 서드파티만 존재(빈틈). "도입 도구 강화 + AI 학습 + OSS 기여"를 한 번에.
