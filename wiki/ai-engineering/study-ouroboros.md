# Ouroboros: 명세 우선(Specification-First) AI 개발 엔진 스터디

> "프롬프팅을 멈추고, 명세를 시작하라." — Stop prompting. Start specifying.

Ouroboros는 모호한 프로젝트 아이디어를 코드 작성 전에 검증된 명세로 변환하는 **Claude Code 플러그인**이다. 소크라테스식 질문법과 존재론적 분석으로 숨겨진 가정을 드러내고 요구사항을 명확화한다.

- 저장소: https://github.com/Q00/ouroboros
- 커뮤니티 프로젝트 (Q00) — Anthropic 공식 기능 아님
- 기술 스택: Python 3.14+, 이벤트 소싱(SQLite), MCP, Claude Agent SDK, LiteLLM

---

## 핵심 철학

> "AI는 무엇이든 만들 수 있다. 어려운 것은 **무엇을** 만들어야 하는지 아는 것이다."

- AI 보조 개발의 병목 = 기계 능력이 아니라 **인간의 명확성**
- 표면적 "어떻게" → 본질적 "이것이 무엇인가?"로 깊이 파고듦
- 두 가지 고대 방법론 결합:
  1. **소크라테스식 질문법** — 숨겨진 가정과 모순 드러냄
  2. **존재론적 분석** — 근본 문제를 찾고 존재 구조를 매핑

## ooo 워크플로 전체 흐름

```
ooo interview → ooo seed → ooo run → ooo evaluate → ooo evolve
     (수동)        (수동)   ╰────────────────────────────────╯
                                   반복 가능 구간
                            ╰════════════════════════════════╯
                              ooo ralph = 이 구간을 자동 반복
```

### Step 1: `ooo interview "아이디어"` — 소크라테스식 인터뷰

- 소크라테스식 인터뷰어 에이전트가 사용자에게 존재론적 질문
- "이것이 정말 무엇인가?", "숨겨진 가정은?", "근본 문제는?"
- 각 응답 후 **모호성 점수** 계산
- **모호성 ≤0.2** (80% 명확성)에 도달할 때까지 질문 반복
- 코드가 아니라 "무엇을 만들어야 하는지"를 명확히 하는 단계

### Step 2: `ooo seed` — 불변 시드(Seed) 명세 생성

- 시드 아키텍트 에이전트가 인터뷰 대화에서 구조화된 요구사항 추출
- YAML로 목표, 제약 조건, 수용 기준, 존재론 스키마, 평가 원칙, 종료 조건 생성
- 생성된 시드는 **불변(immutable)** — 이후 수정 불가, 워크플로의 "헌법"

```yaml
goal: "SQLite 저장소를 갖춘 개인 재무 추적기 구축"
constraints:
  - "데스크톱 애플리케이션만"
  - "카테고리 기반 예산 관리"
acceptance_criteria:
  - "수입과 지출 추적 가능"
  - "거래 자동 분류"
  - "월간 보고서 생성"
ontology_schema:
  name: "FinanceTracker"
  fields:
    - name: "transactions"
      type: "array"
metadata:
  ambiguity_score: 0.15
```

### Step 3: `ooo run` — AC 기반 병렬 실행 엔진

단순히 "프롬프트 → 코드 받기"가 아니라 정교한 오케스트레이션:

```
seed.yaml 입력
  → ① 시드 파싱 & 검증
  → ② PAL 라우터: AC별 복잡도 → 모델 티어 선택 (1x/10x/30x)
  → ③ 의존성 분석: AC 간 의존 관계 → 위상 정렬
  → ④ 병렬 실행: 독립 AC는 동시에, 의존 AC는 순서대로
  → ⑤ 충돌 해결: Coordinator가 같은 파일 수정 감지 & 머지
  → ⑥ 컨텍스트 전달: 이전 레벨 결과 + 경고를 다음 레벨에 주입
```

**의존성 분석 예시:**

```
AC 0: config.py, models.py 생성 (기반)
AC 1: 인증 기능 (config.py 필요)
AC 2: 로깅 기능 (config.py 필요)
AC 3: app.py 통합 (AC 1,2 필요)

→ 의존성: { "0":[], "1":[0], "2":[0], "3":[1,2] }
→ 실행 레벨:
   Level 1: [AC 0]       ← 혼자 먼저
   Level 2: [AC 1, AC 2] ← 병렬
   Level 3: [AC 3]       ← AC 1,2 완료 후
```

**AC 원자성 판단:**
- ATOMIC → Claude 세션 1개로 실행
- COMPLEX → 2~5개 Sub-AC로 분해 → Sub-AC 병렬 실행

**Coordinator 에이전트:**
- 같은 레벨의 AC가 같은 파일 수정 시 자동 감지
- 충돌 있으면 Claude 세션으로 diff 확인 후 머지
- 충돌 없으면 비용 $0

### Step 4: `ooo evaluate` — 3단계 점진적 평가

| Stage | 이름 | 비용 | 내용 |
|-------|------|------|------|
| 1 | 기계적 검증 | $0 | Lint, 빌드, 테스트, 정적분석, 70% 커버리지 |
| 2 | 의미적 평가 | $$ | AC 준수, 목표 정렬, 드리프트, 불확실성 점수 |
| 3 | 합의 (조건부) | $$$ | 다중 모델 투표 (GPT-4o, Claude Sonnet 4, Gemini 2.5 Pro) |

Stage 3 트리거 조건: 시드 수정, 존재론 진화, 목표 재해석, 드리프트 >0.3, 불확실성 >0.3, 측면 사고 채택

### Step 5: `ooo evolve` — 진화

- 평가 결과 기반으로 다음 세대 생성
- 존재론 스키마 유사도 **≥0.95** 수렴 시 종료
- 정체 감지 시 측면 사고 페르소나 투입 후 재시도

## Ralph — "바위는 멈추지 않는다"

### Ralph란?

`ooo ralph "요청"` = **실행→검증 루프를 검증 통과할 때까지 자동 반복하는 모드**

> interview/seed로 "뭘 만들지"를 확정하는 건 사람의 몫.
> Ralph는 그 확정된 명세를 "완성될 때까지 알아서 반복 실행"하는 역할.

### 동작 방식

```
while iteration < max_iterations (기본 10):
    1. 실행 (병렬 가능)
    2. 검증 (테스트 통과? 드리프트? 완성도?)
    3. 검증 통과 → 종료
    4. 실패 → 실패 분석 → 수정 → 체크포인트 저장 → 1로 복귀
```

### 수동 모드 vs Ralph 비교

| | 수동 (`run` + `evaluate` 반복) | Ralph |
|---|---|---|
| 실행 횟수 | 사용자가 매번 판단하고 재시작 | 검증 통과까지 자동 N회 반복 |
| 실패 시 | 사용자 개입 필요 | 자동 분석 → 수정 → 재시도 |
| 영속성 | 세션 단위 | 체크포인트로 세션 간 상태 유지 |
| 중단/재개 | 수동 | `ralph continue`로 이어서 실행 |

### 언제 사용하는가?

- "모든 테스트 통과시켜" — 완전히 성공할 때까지 반복
- "이 기능 끝까지 완성해" — 중간에 멈추지 않음
- 밤새 돌려놓고 내일 확인 — 체크포인트로 중단/재개
- 트리거 키워드: `ralph`, `don't stop`, `must complete`, `until it works`, `keep going`

### 어떤 Ralph인가?

**Geoffrey Huntley의 Ralph Loop에서 영감**받은 것. Claude Code 공식 기능 아님.

| 측면 | Huntley의 Ralph Loop | Ouroboros의 Ralph |
|------|---------------------|-------------------|
| 본질 | Claude Code 자율 반복 실행 패턴 | Ouroboros 진화 엔진 위의 반복 모드 |
| 범위 | 범용 — 어떤 작업이든 | 명세 기반 — 시드 AC 충족까지 |
| 검증 | 사용자 정의 또는 테스트 | 3단계 평가 (기계적/의미적/합의) |
| 정체 처리 | 수동/프롬프트 기반 | 자동 감지 + 5가지 측면 사고 페르소나 |
| 진화 | 없음 (같은 작업 반복) | 존재론 수렴 + 세대별 진화 |

Huntley의 Ralph Loop 개념을 가져와 존재론 수렴, PAL 라우팅, 이벤트 소싱 등과 결합한 확장 버전.

## 6단계 파이프라인 (아키텍처)

| Phase | 이름 | 역할 |
|-------|------|------|
| 0 | 빅뱅 (Big Bang) | 소크라테스식 인터뷰 → 모호성 ≤0.2 → 시드 생성 |
| 1 | PAL 라우터 | 복잡도 기반 모델 티어 선택 (절약 1x / 표준 10x / 최상위 30x) |
| 2 | 더블 다이아몬드 | 발견 → 정의 → 설계 → 전달, AC 분해 및 병렬 실행 |
| 3 | 회복력 (Resilience) | 정체 패턴 감지 + 측면 사고 페르소나 투입 |
| 4 | 평가 (Evaluation) | 3단계 점진적 평가 (기계적 → 의미적 → 합의) |
| 5 | 보조 루프 (Secondary) | 비핵심 TODO 지연 처리, 유휴 시간 일괄 실행 |

### PAL 라우터 상세

| 티어 | 비용 | 복잡도 범위 | 에스컬레이션 |
|------|------|-------------|-------------|
| Frugal (절약형) | 1x | < 0.4 | 2회 실패 → Standard |
| Standard (표준형) | 10x | < 0.7 | 2회 실패 → Frontier |
| Frontier (최상위) | 30x | ≥ 0.7 | 실패 → 정체 이벤트 |

복잡도 = 토큰 수(30%) + 도구 의존성(30%) + AC 중첩 깊이(40%)
5회 연속 성공 시 다운그레이드. 유사 작업은 자카드 유사도 ≥0.80으로 티어 상속.

### 정체 감지 & 측면 사고

**4가지 정체 패턴:**
- SPINNING — 동일 출력 해시 3회 반복
- OSCILLATION — A→B→A→B 패턴 2주기
- NO_DRIFT — 드리프트 변화 없음 (ε <0.01, 3회)
- DIMINISHING_RETURNS — 진행 개선 <0.01 (3회)

**5가지 페르소나:**
| 페르소나 | 역할 |
|----------|------|
| HACKER | 비정통적 우회 방법 — "불가능을 받아들이지 않는다" |
| RESEARCHER | 정보 탐색 — "추측 말고 답을 찾아라" |
| SIMPLIFIER | 복잡도 감소 — "작동하는 가장 단순한 것" |
| ARCHITECT | 근본 재구조화 — "아키텍처와 싸우면 아키텍처가 틀린 것" |
| CONTRARIAN | 가정 도전 — "당연한 것을 뒤집어라" |

## 9가지 에이전트

### 핵심 (Core)

| 에이전트 | 역할 |
|----------|------|
| 소크라테스식 인터뷰어 | 질문으로 모호한 아이디어 → 실행 가능한 요구사항. 오직 인터뷰만, 구현 금지 |
| 존재론자 | 본질/근본원인/전제조건/숨겨진 가정 4가지 핵심 탐구 |
| 시드 아키텍트 | 인터뷰 대화 → 불변 시드 YAML 변환 |
| 평가자 | 3단계 평가 수행 (기계적 → 의미적 → 합의) |
| 반론자 | 가정 뒤집기, 반대 전제 탐구, 무행동 분석 |

### 지원 (Support)

| 에이전트 | 역할 |
|----------|------|
| 해커 | 제약 우회, 엣지 케이스 탐색, 우회 경로 제안 |
| 단순화자 | YAGNI 적용, 불필요 구성요소 50% 제거 목표 |
| 연구자 | 조사 우선, 증거 기반, 추측 → 탐정 작업 |
| 아키텍트 | 구조적 증상 인식, 최소한의 재구조화 제안 |

## 설계 원칙

1. **절약 우선 (Frugal First)** — 가장 저렴하게 시작, 필요 시만 에스컬레이션
2. **불변 방향 (Immutable Direction)** — 시드 변경 불가, 실행 경로만 적응
3. **점진적 검증 (Progressive Verification)** — 저렴한 검사 먼저, 비싼 합의는 게이트에서만
4. **수직보다 수평 (Lateral Over Vertical)** — 막히면 강도 높이기보다 관점 전환
5. **이벤트 소싱 (Event-Sourced)** — 모든 상태 변경은 이벤트, 아무것도 손실 없음

## 모호성 점수 & 수렴 메커니즘

**모호성 점수 (Ambiguity Scoring):**
- 목표, 제약 조건, 성공 기준의 가중 차원
- ≤0.2 (80% 명확성)에 도달해야 시드 생성 가능
- "준비된 느낌"이 아니라 수학적 게이트

**존재론 수렴 (Ontology Convergence):**
- 연속 세대 존재론 스키마 유사도 ≥0.95 달성 시 수렴 판정
- 안전 메커니즘이 진동, 정체, 반복 패턴 감지

## 더블 다이아몬드(Double Diamond) 심화 설명

### 1. 더블 다이아몬드 디자인 방법론이란?

더블 다이아몬드는 원래 **영국 디자인 카운슬(British Design Council)**이 2005년에 제안한 디자인 프로세스 모델이다. 이름이 "더블 다이아몬드"인 이유는 **다이아몬드 모양이 두 번** 반복되기 때문이다.

```
    발산(넓히기)        발산(넓히기)
   ◇ 첫 번째 다이아몬드 ◇ 두 번째 다이아몬드
    수렴(좁히기)        수렴(좁히기)

  [Discover] → [Define] → [Design] → [Deliver]
   "문제 찾기"            "해결책 만들기"
```

핵심 원리는 간단하다:
- **발산(Diverge)** = 가능성을 최대한 넓게 탐색하는 것. "일단 많이 모아보자"
- **수렴(Converge)** = 그중에서 핵심만 골라내는 것. "이것만 남기자"

이걸 **두 번** 반복한다:
1. **첫 번째 다이아몬드**: "올바른 문제"를 찾는 과정 (문제 공간)
2. **두 번째 다이아몬드**: "올바른 해결책"을 만드는 과정 (해결 공간)

Ouroboros에서는 이것을 6단계 파이프라인의 **Phase 2**로 사용하며, 각 AC(수용 기준)가 이 4단계를 거쳐 실행된다.

### 2. 4단계를 소프트웨어 개발 예시로 설명

**예시 프로젝트: "우리 회사 직원들이 쓸 사내 식당 예약 시스템을 만들어주세요"**

#### Phase 1: Discover (발견) -- 발산적 탐색

> "일단 가능한 모든 것을 넓게 탐색한다"

이 단계에서는 판단하지 않고 최대한 많이 모은다:
- 직원들이 현재 식당을 어떻게 이용하는지 조사
- 비슷한 시스템(배민, 캐치테이블, 구글 캘린더 예약)은 어떻게 되어있는지 탐색
- 가능한 기술 스택을 나열 (웹앱? 모바일앱? 슬랙봇? 키오스크?)
- 이해관계자 인터뷰 ("식당 매니저는 뭘 원하나?", "직원은?", "경영진은?")
- 엣지 케이스 수집 ("단체 예약은?", "취소는?", "알러지 정보는?")

**발산 = 아이디어의 부채꼴.** 이 시점에서 "이건 안 돼"라고 자르지 않는다.

#### Phase 2: Define (정의) -- 수렴적 분석

> "수집한 것 중에서 핵심 문제를 정의한다"

넓게 탐색한 것을 분석하고 좁힌다:
- 핵심 문제 정의: "점심시간 혼잡 → 시간대별 예약으로 분산"
- 스코프 확정: "웹앱으로, 슬랙 연동, 모바일은 v2"
- 제약 조건 확정: "기존 SSO 인증 사용, 200명 동시 접속"
- 불필요한 것 제거: "키오스크는 과하다", "알러지는 별도 시스템에서 관리"
- AC(수용 기준) 작성: "시간대별 좌석 예약 가능", "예약 확인 슬랙 알림"

**수렴 = 깔때기로 걸러냄.** "이것만 만들면 된다"를 확정.

#### Phase 3: Design (설계) -- 발산적 설계

> "확정된 문제에 대해 다시 여러 해결 방안을 탐색한다"

문제는 확정됐지만 **만드는 방법**은 아직 열려있다:
- DB 설계안 A: 시간 슬롯 테이블 vs 설계안 B: 캘린더 이벤트 기반
- API 설계안: REST vs GraphQL
- 프론트엔드: Next.js SSR vs React SPA vs HTMX
- 예약 충돌 처리: 선착순 잠금 vs 낙관적 동시성 제어
- 알림 방식: 웹훅 vs 폴링 vs SSE

**다시 발산.** 해결책의 부채꼴을 펼친다.

#### Phase 4: Deliver (전달) -- 수렴적 구현

> "최적의 설계를 선택하고 실제로 구현한다"

여러 설계안 중 최적을 골라 실행:
- Next.js + PostgreSQL + Slack Webhook 선택
- AC별로 코드 구현 시작
- 테스트 작성 및 검증
- 코드 리뷰, 통합 테스트, 배포

**마지막 수렴.** "이 방법으로 만든다"를 확정하고 완성.

```
정리:
Discover(발산) → "뭐가 문제인지 넓게 탐색"
Define(수렴)   → "이게 진짜 문제다"로 좁힘
Design(발산)   → "어떻게 풀지 넓게 탐색"
Deliver(수렴)  → "이렇게 만든다"로 좁힘
```

### 3. "원자성 검사를 통한 재귀적 분해" 구체적 예시

"원자성(Atomicity)"이란 **"더 이상 쪼갤 필요 없이 한 번에 실행할 수 있는가?"**를 의미한다.

Ouroboros에서는 각 AC를 실행하기 전에 Claude에게 물어본다:
> "이 작업은 단일 세션(1~2개 파일 수정)으로 끝낼 수 있는가? 가능하면 ATOMIC, 불가능하면 COMPLEX라고 답하고 2~5개 하위 작업으로 분해해라."

**예시: "온라인 쇼핑몰 구축"의 재귀적 분해**

```
Level 0: "온라인 쇼핑몰 구축" ← COMPLEX (한 세션으로? 절대 불가능)
│
├─ Level 1: "상품 관리 시스템" ← COMPLEX (아직 크다)
│  ├─ Level 2: "상품 DB 모델 정의" ← ATOMIC (models.py 하나면 됨)
│  ├─ Level 2: "상품 목록 API" ← ATOMIC (routes.py + serializer.py)
│  ├─ Level 2: "상품 이미지 업로드" ← ATOMIC (upload.py + storage.py)
│  └─ Level 2: "상품 검색 기능" ← ATOMIC (search.py)
│
├─ Level 1: "장바구니" ← COMPLEX
│  ├─ Level 2: "장바구니 모델 & API" ← ATOMIC
│  ├─ Level 2: "수량 변경/삭제" ← ATOMIC
│  └─ Level 2: "장바구니 합계 계산" ← ATOMIC
│
├─ Level 1: "결제 시스템" ← COMPLEX
│  ├─ Level 2: "PG사 연동" ← COMPLEX (아직 크다!)
│  │  ├─ Level 3: "결제 요청 API" ← ATOMIC
│  │  ├─ Level 3: "결제 콜백 처리" ← ATOMIC
│  │  └─ Level 3: "결제 상태 관리" ← ATOMIC
│  ├─ Level 2: "주문 생성 로직" ← ATOMIC
│  └─ Level 2: "결제 완료 알림" ← ATOMIC
│
└─ Level 1: "사용자 인증" ← ATOMIC (auth.py + middleware.py로 충분)
```

**분해 과정을 말로 풀면:**

1. "쇼핑몰 구축" → 한 세션으로 되나? **아니요(COMPLEX)** → 4개로 쪼갬
2. "상품 관리 시스템" → 한 세션으로 되나? **아니요(COMPLEX)** → 4개로 쪼갬
3. "상품 DB 모델 정의" → 한 세션으로 되나? **네(ATOMIC)** → 바로 실행!
4. "결제 시스템" → 쪼개봤더니 "PG사 연동"이 또 크다 → 한 번 더 쪼갬
5. "사용자 인증" → 파일 2개면 되니까 **ATOMIC** → 바로 실행!

이것이 **재귀적** 분해다. "쪼개고 → 각각 검사하고 → 아직 크면 또 쪼개고"를 반복하는 것.

### 4. ATOMIC vs COMPLEX의 차이

| 기준 | ATOMIC | COMPLEX |
|------|--------|---------|
| 정의 | 단일 초점. 한 번에 끝남 | 여러 하위 작업 포함. 분해 필요 |
| 파일 수 | 1~2개 파일 수정 | 다수 파일, 다수 관심사 |
| 실행 | Claude 세션 1개로 바로 실행 | 2~5개 Sub-AC로 분해 후 각각 실행 |
| 예시 | "로그인 API 엔드포인트 작성" | "사용자 관리 시스템 구축" |
| 비유 | 레고 블록 1개 끼우기 | 레고 세트 전체 조립하기 |

**판단 기준 (Claude가 평가):**
- 여러 독립적인 단계가 있는가?
- 서로 다른 관심사(concern)가 섞여 있는가?
- 한 세션의 범위를 초과하는가?

하나라도 해당되면 **COMPLEX** → 분해. 아니면 **ATOMIC** → 바로 실행.

### 5. MAX_DEPTH=5와 COMPRESSION_DEPTH=3 쉬운 비유

#### MAX_DEPTH=5 -- "건물은 지하 5층까지만 파자"

재귀적 분해를 무한히 하면 어떻게 될까? "쇼핑몰" → "결제" → "PG 연동" → "API 호출" → "HTTP 헤더 설정" → "문자열 인코딩" → ... 끝없이 쪼개질 수 있다.

**MAX_DEPTH=5**는 이렇게 말하는 것이다:

> "아무리 복잡한 작업이라도 5단계 이상 쪼개지 마라."

```
깊이 0: 쇼핑몰 구축
깊이 1: └─ 결제 시스템
깊이 2:    └─ PG사 연동
깊이 3:       └─ 결제 콜백 처리
깊이 4:          └─ 웹훅 검증 로직
깊이 5:             └─ 여기가 한계. 더 이상 분해 금지!
```

비유하자면: **러시안 인형(마트료시카)**을 열면 안에 더 작은 인형이 있다. 하지만 5번째 인형에서 멈춘다. "여기서부터는 그냥 실행해."

왜 5인가? 너무 깊이 쪼개면:
- 컨텍스트가 폭발적으로 커지고 (각 레벨마다 2~5개씩이면 5단계에서 최대 3,125개)
- 오히려 전체 그림을 잃어버리고
- API 비용이 기하급수적으로 증가한다

#### COMPRESSION_DEPTH=3 -- "3층 아래부터는 요약본만 전달"

AI가 작업을 실행할 때 **이전 작업의 결과(컨텍스트)**를 다음 작업에 전달한다. 그런데 깊이가 깊어지면 전달할 컨텍스트가 너무 많아진다.

**COMPRESSION_DEPTH=3**은 이렇게 말하는 것이다:

> "깊이 3 이상부터는 컨텍스트를 500자로 압축해서 전달해라."

```
깊이 0: 전체 컨텍스트 전달 (제한 없음)
깊이 1: 전체 컨텍스트 전달 (제한 없음)
깊이 2: 전체 컨텍스트 전달 (제한 없음)
깊이 3: ✂️ 500자로 압축! "여기부터는 요약만"
깊이 4: ✂️ 500자로 압축!
깊이 5: ✂️ 500자로 압축! (MAX_DEPTH 도달)
```

비유하자면 **전화 돌리기(전언 게임)** 같은 것이다:
- 1~2번째 사람에게는 원본 메시지를 그대로 전달한다
- 3번째 사람부터는 "핵심만 요약해서" 전달한다
- 그래야 메시지가 왜곡되지 않고, 전달 비용(토큰)도 절약된다

**왜 이게 필요한가?**

Claude에는 컨텍스트 윈도우(한 번에 처리할 수 있는 텍스트 양) 제한이 있다. 깊이 4~5에 있는 아주 세부적인 작업("웹훅 검증 로직")에게 상위 모든 컨텍스트를 통째로 전달하면:
- 토큰 낭비 (비용 증가)
- 불필요한 정보에 주의가 분산됨
- 정작 중요한 지시사항이 묻힘

그래서 **"깊은 곳에서는 요약본만 보고 일하라"**는 규칙이다.

```
종합 정리:
MAX_DEPTH=5     → 분해의 최대 깊이 (더 이상 쪼개지 마!)
COMPRESSION_DEPTH=3 → 컨텍스트 압축 시작점 (여기부터 요약해서 전달!)

비유: 회사 조직도
- 사장 → 부장 → 과장 → 대리 → 사원 (5단계 = MAX_DEPTH)
- 사장의 지시를 대리/사원에게 전달할 때는 핵심만 요약 (3단계부터 = COMPRESSION_DEPTH)
- 사장이 10페이지 전략 문서를 썼다고 사원한테 그걸 다 읽히지 않는다
- "핵심: 이번 달 매출 10% 올리기" 이 한 줄이면 충분하다
```
