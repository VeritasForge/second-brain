## Feature Engineering 정리

---

### 1. Feature Engineering이란?

**Raw 데이터를 ML 모델이 이해할 수 있는 형태로 변환하는 과정**입니다.

비유: 요리에서 **재료 손질** 단계입니다.
- 마트에서 산 생닭(raw data) → 뼈를 발라내고, 양념하고, 적당한 크기로 잘라서(feature engineering) → 오븐에 넣음(모델 학습/추론)
- 같은 생닭이라도 손질 방법이 다르면 완전히 다른 요리가 됩니다.

```
Raw Data → [Feature Engineering] → Features → [ML Model] → 예측 결과
             ^^^^^^^^^^^^^^^^^^
             이 과정이 핵심
```

---

### 2. Feature Engineering의 구성 요소

넓은 범위에서 좁은 범위 순으로:

#### 2-1. Data Cleaning (데이터 정제)

이상치와 노이즈를 제거하는 단계

```
혈압: [120, 130, 9999, 125]
              ^^^^
              측정 오류 → 제거 또는 보정
```

- **Outlier 제거**: 물리적으로 불가능한 값 필터링
- **중복 제거**: 동일 데이터 중복 입력 처리
- **오류 수정**: 단위 오류, 입력 실수 보정

#### 2-2. Feature Transformation (값 변환)

모델이 잘 학습할 수 있도록 값의 스케일을 조정

```
정규화 전: 키 = 175cm, 몸무게 = 70kg, 나이 = 30세   ← 스케일이 다 다름
정규화 후: 키 = 0.75,  몸무게 = 0.58, 나이 = 0.17   ← 0~1 사이로 통일
```

| 기법             | 설명                       | 공식                      | 용도                       |
| ---------------- | -------------------------- | ------------------------- | -------------------------- |
| Min-Max 정규화   | 0~1 범위로 변환            | (x - min) / (max - min)   | 값의 범위가 알려진 경우    |
| Z-Score 표준화   | 평균 0, 표준편차 1로 변환  | (x - μ) / σ               | 정규분포에 가까운 데이터   |
| Log 변환         | 로그를 취함                | log(x)                    | 극단적으로 치우친 분포     |
| Binning          | 연속값을 구간으로 나눔     | 나이 → 10대/20대/30대     | 연속값을 범주로 바꿀 때    |

#### 2-3. Feature Construction (새 feature 생성)

원본 데이터에서 새로운 의미 있는 feature를 만들어냄

```
원본: 구매일 = 2026-01-15, 생년월일 = 1996-05-20
생성: 구매 시 나이 = 29세, 요일 = 목요일, 계절 = 겨울
```

- **시간 기반**: 날짜 → 요일, 월, 계절, 공휴일 여부 추출
- **시계열 리샘플링**: 불규칙 시간 데이터 → 고정 간격으로 양자화
- **교차 feature**: 키, 몸무게 → BMI 계산
- **통계 기반**: 최근 7일 평균, 이동 평균, 변화율
- **결측 마스크**: 데이터가 있었는지/없었는지 자체를 feature로 만듦
- **Delta**: 마지막 관측 이후 경과 시간

#### 2-4. Missing Value Handling (결측값 처리)

빈 데이터를 어떻게 채울지 결정

```
시간별 맥박: [80, ?, ?, 85, ?, 72]
```

| 방법        | 설명                         | 결과                                     |
| ----------- | ---------------------------- | ---------------------------------------- |
| Forward-fill | 앞의 값으로 채움            | [80, **80, 80**, 85, **85**, 72]         |
| Backward-fill | 뒤의 값으로 채움           | [80, **85, 85**, 85, **72**, 72]         |
| 기본값 대치 | 의학적/통계적 정상값 사용    | [80, **75, 75**, 85, **75**, 72]         |
| 보간법      | 앞뒤 값의 중간값             | [80, **82, 83**, 85, **79**, 72]         |
| 삭제        | 해당 행 자체를 제거          | [80, 85, 72]                             |

#### 2-5. Feature Encoding (인코딩)

문자/범주형 데이터를 숫자로 변환

```
성별: "남" → 1, "여" → 0                    (Label Encoding)
혈액형: "A" → [1,0,0,0], "B" → [0,1,0,0]   (One-Hot Encoding)
증상: "가슴이 아파요" → [101, 8834, 2345]    (Text Tokenization)
```

#### 2-6. Feature Selection (feature 선택)

모든 feature 중 모델에 실제로 넣을 것을 선택

```
후보: [키, 몸무게, 혈액형, 신발사이즈, 혈압, 맥박, ...]
선택: [키, 몸무게, 혈압, 맥박]  ← 예측에 의미 있는 것만
```

### 3. Training-Serving Skew (학습-추론 불일치)

#### 3-1. 왜 문제인가?

```
학습: 맥박 80 → [모듈 A: (80-0)/(300-0)] → 0.267 → 이 값으로 패턴 학습
추론: 맥박 80 → [모듈 B: (80-30)/(200-30)] → 0.294 → 다른 값 입력 → 예측 틀어짐
```

비유: 시험 공부할 때 "영한 번역기"로 공부했는데, 시험장에서 "영일 번역기"로 문제를 읽으면 — 같은 영어 원문이어도 답을 못 씁니다.

#### 3-2. Skew가 발생하는 주요 원인

| 원인                         | 예시                                                |
| ---------------------------- | --------------------------------------------------- |
| **코드 이중 구현**           | 학습은 Python, 추론은 Java로 따로 구현              |
| **정규화 파라미터 불일치**   | 학습 시 min=0, max=300 vs 추론 시 min=0, max=250    |
| **결측값 처리 차이**         | 학습은 0으로 채움 vs 추론은 평균으로 채움           |
| **feature 순서 변경**        | 학습 시 [맥박, 혈압, 체온] vs 추론 시 [체온, 맥박, 혈압] |
| **시간 윈도우 차이**         | 학습은 48시간 vs 추론은 24시간                      |
| **버전 드리프트**            | 학습 코드는 업데이트했는데 추론 코드는 안 함        |

### 4. Feature Engineering 관리 전략

#### 4-1. 같은 코드 공유 (가장 권장)

```
shared_feature_lib/        ← 하나의 라이브러리
├── transform.py
├── normalize.py
└── impute.py

training_pipeline/  → import shared_feature_lib   ← 동일 코드
serving_pipeline/   → import shared_feature_lib   ← 동일 코드
```

코드가 하나이므로 skew 자체가 불가능합니다.

#### 4-2. Feature Store 사용

```
                    ┌──────────────┐
raw data ──────→   │ Feature Store │ ←── feature 정의 + 변환 로직
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ↓                         ↓
         학습이 가져다 씀          추론이 가져다 씀
```

Feature를 **중앙 저장소에서 계산하고 저장**해두면, 학습/추론 모두 같은 곳에서 가져다 씁니다. 대표 도구: Feast, Tecton, Hopsworks.

#### 4-3. 모델에 전처리 내장

```
기존: raw data → [외부 전처리] → [모델] → 예측
개선: raw data → [모델 (전처리 포함)] → 예측
```

모델 파일 안에 전처리 로직을 포함시킴 (예: TensorFlow Transform, sklearn Pipeline). 모델을 배포하면 전처리도 함께 배포됩니다.

#### 4-4. 관리 체크리스트

| 항목                      | 확인 사항                                                                |
| ------------------------- | ------------------------------------------------------------------------ |
| **단일 소스**             | 학습/추론이 동일한 전처리 코드를 사용하는가?                             |
| **파라미터 버전 관리**    | 정규화의 min/max, 인코딩 매핑 등이 모델 버전과 함께 관리되는가?          |
| **테스트**                | 같은 입력에 대해 학습/추론 전처리의 출력이 동일한지 자동 검증하는가?     |
| **모니터링**              | 운영 중 feature 분포가 학습 시와 크게 달라지면 알림이 오는가?            |
| **문서화**                | 각 feature의 변환 방식, 범위, 결측 처리 방법이 문서화되어 있는가?        |

---

### 5. 한 줄 요약

> **Feature Engineering은 raw 데이터를 모델이 이해할 수 있는 형태로 변환하는 과정이며, 학습과 추론에서 반드시 동일한 변환 코드(모듈)를 사용해야 한다. 그렇지 않으면 Training-Serving Skew가 발생하여 모델 성능이 저하된다.**
