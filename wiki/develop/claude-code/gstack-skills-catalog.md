## gstack 스킬 전체 카탈로그

### 메인 워크플로우: 7단계 스프린트

gstack의 핵심은 **Think → Plan → Build → Review → Test → Ship → Reflect** 스프린트 사이클입니다. 각 단계에 "전문가 역할"이 배정되어 있습니다.

---

### Phase 1: Think (생각하기)

| 스킬             | 역할       | 설명                                                                                                                                                                                                                                             |
| ---------------- | ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `/office-hours`  | YC 파트너  | **진입점**. 두 가지 모드: (1) 스타트업 모드 — 수요 현실, 현 상태, 가장 좁은 쐐기 등 6가지 질문으로 아이디어 검증 (2) 빌더 모드 — 사이드 프로젝트/해커톤용 디자인 싱킹. 전제를 공격하고 대안을 생성. **설계 문서를 출력**하여 이후 모든 스킬에 자동 전달 |

비유: 선생님에게 "이거 하고 싶어요"라고 말하면, "정말? 왜? 누가 원해? 더 좋은 방법은?" 하고 질문 공세를 퍼붓는 단계.

---

### Phase 2: Plan (계획하기)

| 스킬                  | 역할              | 설명                                                                                                       |
| --------------------- | ----------------- | ---------------------------------------------------------------------------------------------------------- |
| `/plan-ceo-review`    | CEO/창업자        | 문제를 재정의. "10성급 제품"을 찾음. 4가지 모드: 확장/선택적 확장/유지/축소                                 |
| `/plan-eng-review`    | 엔지니어링 매니저 | 아키텍처 확정, 데이터 흐름 다이어그램, 엣지 케이스, 테스트 플랜 생성. 숨겨진 가정을 끄집어냄               |
| `/plan-design-review` | 시니어 디자이너   | 각 디자인 차원을 0-10 평가. "10점이면 어떤 모습인지" 설명 후 플랜 수정. AI Slop 탐지 포함                  |
| `/autoplan`           | 오케스트레이터    | **원커맨드**로 CEO→디자인→엔지니어링 리뷰를 자동 순차 실행. 취향 결정만 사용자에게 질문                    |

비유: 집을 짓기 전에 건축주(CEO), 건축사(디자이너), 구조기사(엔지니어)가 돌아가며 설계도를 검토하는 것.

---

### Phase 3: Build (만들기 - 디자인)

| 스킬                   | 역할           | 설명                                                                                                          |
| ---------------------- | -------------- | ------------------------------------------------------------------------------------------------------------- |
| `/design-consultation` | 디자인 파트너  | 경쟁사 조사 → 디자인 시스템 → 목업 생성. `DESIGN.md` 작성                                                    |
| `/design-shotgun`      | 디자인 탐색자  | **여러 디자인 변형**을 생성하고 브라우저에서 비교 보드를 열어 사용자가 선택. 취향이 시간이 지날수록 학습됨     |
| `/design-html`         | 디자인 엔지니어 | 승인된 목업을 **프로덕션 HTML**로 변환. React/Svelte/Vue 프레임워크 자동 감지                                 |

비유: 옷 가게에서 여러 벌 입어보고(shotgun) → 마음에 드는 걸 고르고(consultation) → 맞춤 제작(html).

---

### Phase 4: Review (검토하기)

| 스킬              | 역할            | 설명                                                                                                                                  |
| ----------------- | --------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| `/review`         | 스태프 엔지니어 | PR 사전 리뷰. CI는 통과하지만 프로덕션에서 터질 버그 탐지. 명확한 것은 자동 수정, 판단 필요한 것만 보고                                |
| `/investigate`    | 디버거          | 체계적 근본 원인 분석. **철의 법칙: 조사 없이 수정 금지**. 3회 실패 시 중단. 자동으로 `/freeze` 활성화                                |
| `/design-review`  | 디자이너+개발자 | 라이브 사이트 시각 감사 80개 항목 체크 → 문제 발견 시 수정 → before/after 스크린샷으로 커밋                                            |
| `/cso`            | 최고보안책임자  | OWASP Top 10 + STRIDE 위협 모델링. 8/10 이상 확신도 게이트. 각 발견에 **구체적 공격 시나리오** 포함                                    |

비유: 시험 답안지를 제출하기 전에 선생님 4명이 각자 관점에서 검토하는 것 — 코드 선생님, 탐정, 디자인 선생님, 보안 선생님.

---

### Phase 5: Test (테스트하기)

| 스킬         | 역할         | 설명                                                                                                          |
| ------------ | ------------ | ------------------------------------------------------------------------------------------------------------- |
| `/qa`        | QA 리드      | **실제 브라우저**를 열어 앱을 테스트 → 버그 발견 → 수정 → 재검증 → 회귀 테스트 자동 생성. URL을 주면 됨       |
| `/qa-only`   | QA 리포터    | `/qa`와 동일하지만 **보고서만** 생성 (코드 수정 안 함)                                                        |
| `/benchmark` | 성능 엔지니어 | 페이지 로드 시간, Core Web Vitals, 리소스 크기 측정. PR 전후 비교                                             |

비유: `/qa`는 게임 테스터가 직접 플레이하면서 버그를 찾고 고치는 것, `/qa-only`는 버그 리포트만 쓰는 것.

---

### Phase 6: Ship (출시하기)

| 스킬                 | 역할             | 설명                                                                                                              |
| -------------------- | ---------------- | ----------------------------------------------------------------------------------------------------------------- |
| `/ship`              | 릴리즈 엔지니어  | main 동기화 → 테스트 → 커버리지 감사 → push → PR 생성. **원커맨드**로 "코드 완성"에서 "PR 오픈"까지              |
| `/land-and-deploy`   | 릴리즈 엔지니어  | PR 머지 → CI 대기 → 배포 → 프로덕션 헬스 체크. "승인됨"에서 "프로덕션 검증"까지 원커맨드                         |
| `/canary`            | SRE              | 배포 후 모니터링 루프. 콘솔 에러, 성능 회귀, 페이지 실패 감시                                                    |
| `/document-release`  | 테크니컬 라이터  | 출시된 내용에 맞게 **모든 문서 자동 업데이트** (README, ARCHITECTURE, CONTRIBUTING, CLAUDE.md 등)                  |

비유: 택배로 비유하면 — 포장(ship) → 배송+수령 확인(land-and-deploy) → 배송 후 추적(canary) → 운송장 정리(document-release).

---

### Phase 7: Reflect (회고하기)

| 스킬     | 역할              | 설명                                                                                                                           |
| -------- | ----------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `/retro` | 엔지니어링 매니저 | 주간 회고. 인당 분석, 코드 라인 수, 커밋 수, 테스트 건강도, 성장 기회. `/retro global`로 모든 프로젝트+AI 도구 통합 회고        |

## 워크플로우 밖 스킬들 (Power Tools)

### 브라우저 도구

| 스킬                     | 용도           | 핵심 기능                                                                                                                                  |
| ------------------------ | -------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `/browse`                | **에이전트의 눈** | 상주 Chromium 데몬. `$B click @e1`, `$B screenshot` 등 ~100ms 명령. `$B handoff`로 CAPTCHA/MFA 시 사용자에게 제어권 넘김                   |
| `/connect-chrome`        | **라이브 관전**   | 실제 Chrome 창 + Side Panel 확장. 에이전트가 하는 작업을 실시간으로 볼 수 있음. 탭별 독립 에이전트                                          |
| `/setup-browser-cookies` | **세션 이식**     | Chrome/Arc/Brave/Edge의 쿠키를 헤드리스 브라우저에 임포트. 로그인 상태 유지                                                                |

비유: `/browse`는 로봇에게 눈을 달아주는 것, `/connect-chrome`은 로봇이 보는 화면을 모니터로 같이 보는 것, 쿠키 설정은 로봇에게 내 출입증을 빌려주는 것.

### 크로스 모델 검증

| 스킬     | 용도               | 핵심 기능                                                                                                                  |
| -------- | ------------------ | -------------------------------------------------------------------------------------------------------------------------- |
| `/codex` | **세컨드 오피니언** | OpenAI Codex CLI로 독립 리뷰. 3가지 모드: review(합/불), adversarial(적극적 공격), open(대화). Claude + OpenAI 교차 분석 리포트 생성 |

비유: 의사 한 명의 소견만 듣지 않고, 다른 병원 의사에게도 진단받는 "세컨드 오피니언".

### 안전 장치 (Guardrails)

| 스킬        | 용도              | 핵심 기능                                                              |
| ----------- | ----------------- | ---------------------------------------------------------------------- |
| `/careful`  | 파괴적 명령 경고  | `rm -rf`, `DROP TABLE`, force-push 등 실행 전 경고. 오버라이드 가능    |
| `/freeze`   | 편집 범위 잠금    | 특정 디렉토리만 수정 가능하게 **하드 블록**. 디버깅 중 다른 코드 건드리기 방지 |
| `/guard`    | 풀 세이프티       | `/careful` + `/freeze` 동시 활성화                                     |
| `/unfreeze` | 잠금 해제         | `/freeze` 해제                                                         |

비유: `/careful`은 "정말 지울 거야?" 팝업, `/freeze`는 특정 방만 들어갈 수 있는 출입 카드, `/guard`는 둘 다 켜는 비상 모드.

### 메모리 시스템

| 스킬     | 용도     | 핵심 기능                                                                        |
| -------- | -------- | -------------------------------------------------------------------------------- |
| `/learn` | 지식 축적 | 세션 간 학습 내용 관리. 프로젝트별 패턴, 함정, 선호도를 기억하고 점점 똑똑해짐   |

### 설정/유지보수

| 스킬              | 용도         | 핵심 기능                                                                    |
| ----------------- | ------------ | ---------------------------------------------------------------------------- |
| `/setup-deploy`   | 배포 설정    | `/land-and-deploy`를 위한 1회성 설정. 플랫폼, URL, 배포 명령어 감지          |
| `/gstack-upgrade` | 자체 업데이트 | gstack 최신 버전으로 업그레이드. 글로벌/벤더 설치 모두 지원                  |

## 전체 흐름도

```
                    ┌─ /careful, /freeze, /guard ─┐  (항시 활성화 가능)
                    │                              │
/office-hours ──→ /autoplan ──→ [코드 작성] ──→ /review ──→ /qa ──→ /ship ──→ /retro
   (Think)      ┌──────┐       ┌──────────┐     /codex      /benchmark  /land-and-deploy
                │CEO    │      │/design-*  │     /cso        /qa-only    /canary
                │Eng    │      │/design-html│    /investigate             /document-release
                │Design │      └──────────┘
                └──────┘
                                                  /browse ─── 모든 브라우저 작업의 기반
                                                  /learn ──── 세션 간 지식 축적
```

**핵심 포인트**: 스킬들이 독립적이 아니라 **파이프라인으로 연결**됩니다. `/office-hours`가 쓴 설계 문서를 `/plan-ceo-review`가 읽고, `/plan-eng-review`가 쓴 테스트 플랜을 `/qa`가 활용하는 식입니다. 이게 gstack이 단순 스킬 모음이 아닌 "소프트웨어 팩토리"인 이유입니다.
