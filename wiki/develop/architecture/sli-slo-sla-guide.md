# SLI, SLO, SLA — 서비스 신뢰성 지표 체계

## 정의

### SLI (Service Level Indicator) — 측정값

**"지금 우리 서비스가 얼마나 잘 하고 있는지"를 보여주는 숫자**

비유: 체온계로 잰 **체온 그 자체**. "36.5도"라는 측정값.

```
SLI = 좋은 이벤트 수 / 전체 이벤트 수
```

| SLI      | 계산                                       |
| -------- | ------------------------------------------ |
| 가용성   | 성공한 요청 수 / 전체 요청 수 = 99.95%     |
| 지연시간 | 200ms 이내 응답 수 / 전체 요청 수 = 98.2%  |
| 처리량   | 정상 처리된 메시지 수 / 전체 메시지 수 = 99.8% |

핵심: SLI는 **사실(fact)**. 판단이 없어. 그냥 숫자야.

### SLO (Service Level Objective) — 목표값

**"이 숫자가 최소 이만큼은 되어야 한다"는 팀 내부 목표**

비유: 의사가 정한 **정상 체온 기준**. "36~37도 사이여야 정상"이라는 기준선.

```
SLO = SLI가 달성해야 하는 목표 범위
```

| SLI                | SLO (목표) | 현재 실제값 | 판정     |
| ------------------ | ---------- | ----------- | -------- |
| 가용성             | >= 99.9%   | 99.95%      | Pass     |
| p99 지연시간       | <= 300ms   | 450ms       | **Fail** |
| 메시지 처리 성공률 | >= 99.5%   | 99.8%       | Pass     |

핵심: SLO는 **팀 내부의 약속**. 이걸 깨면 엔지니어링 대응이 필요하다는 신호.

### SLA (Service Level Agreement) — 계약

고객과 맺는 외부 약속. SLO보다 느슨하게 설정하며, 위반 시 환불/크레딧 등 페널티가 발생.

---

## 관계 정리

```
SLI (측정)  →  SLO (목표)  →  SLA (계약)
  체온          정상 범위       보험 약관
  숫자          내부 기준       외부 약속 (위반 시 페널티)
```

|            | SLI                      | SLO                    | SLA                        |
| ---------- | ------------------------ | ---------------------- | -------------------------- |
| **뭐냐**   | 메트릭                   | 목표                   | 계약                       |
| **누구를 위해** | 엔지니어            | 팀                     | 고객                       |
| **위반하면** | 아무 일 없음 (그냥 숫자) | 팀이 우선순위 조정     | 환불/크레딧 지급           |
| **예시**   | "가용성 99.95%"          | "가용성 >= 99.9% 유지" | "99.9% 미만 시 크레딧 제공" |

## P99 (백분위수)

### 개념

**P99 = 100명 중 99번째로 빠른 사람의 기록** = 99%의 요청은 이것보다 빠르다

비유: 학교에서 100명이 달리기 시험을 봤을 때, 모든 학생의 기록을 **빠른 순서대로** 줄 세운 것:

```
1번째: 10초  ← 제일 빠른 애
50번째: 15초  ← P50 (중간값, median)
95번째: 22초  ← P95
99번째: 35초  ← P99 (거의 제일 느린 애)
100번째: 60초 ← 제일 느린 애 (아마 걸어온 듯)
```

### 서버 응답시간 적용

요청 1000개가 들어왔을 때, 응답시간을 빠른 순서로 정렬:

```
P50 =  15ms  → 절반의 요청은 15ms 이내
P90 =  80ms  → 90%의 요청은 80ms 이내
P95 = 150ms  → 95%의 요청은 150ms 이내
P99 = 450ms  → 99%의 요청은 450ms 이내  ← 보통 SLO 기준
P999= 1200ms → 99.9%의 요청은 1.2s 이내
```

### 왜 평균 대신 P99를 쓰는가?

```
요청 10개의 응답시간:
10, 12, 11, 13, 10, 12, 11, 10, 12, 5000ms

평균 = 510ms   ← "평균적으로 꽤 느리네"
P50  = 12ms    ← "대부분은 빠른데..."
P99  = 5000ms  ← "한 명이 극단적으로 느리네!"
```

**평균은 거짓말쟁이야.** 한 명이 극단적으로 느리면 평균이 왜곡되지만, 대부분의 사용자 경험은 괜찮을 수 있어. 반대로, 평균이 좋아 보여도 P99가 나쁘면 **100명 중 1명은 끔찍한 경험**을 하고 있다는 뜻.

### SLO에서의 활용

| SLO 표현                    | 의미                                              |
| --------------------------- | ------------------------------------------------- |
| "P50 응답시간 <= 50ms"      | 절반의 요청은 50ms 이내여야 함 (너무 느슨)        |
| "P99 응답시간 <= 300ms"     | 99%의 요청은 300ms 이내여야 함 (일반적 기준)      |
| "P999 응답시간 <= 1s"       | 99.9%의 요청은 1초 이내 (엄격한 기준)             |

보통 **P99**를 SLO 기준으로 많이 쓰는 이유: **대다수 사용자의 경험을 보장하면서도**, 정말 극단적인 아웃라이어(네트워크 순간 끊김 등)는 제외해주니까 현실적.

## 비동기 아키텍처에서의 로드 테스트 적용

### 문제

`A → Queue → B → 외부 API` 구조에서 Locust로 A에 요청을 보내면 Queue에 메시지를 넣고 트래픽이 끝나버린다. E2E 응답시간을 직접 측정할 수 없다.

### 해결: 구간별 SLI 측정

E2E로 보지 않고, **각 구간이 자기 SLI를 지키는지** 독립적으로 측정:

```
┌─────────┐     ┌───────┐     ┌───────┐     ┌──────────┐
│ Locust  │────▶│   A   │────▶│ Queue │────▶│    B     │────▶ 외부 API
│         │     │ SLI①  │     │ SLI②  │     │  SLI③   │
└─────────┘     └───────┘     └───────┘     └──────────┘
```

| 구간             | SLI 예시                                              | 측정 방법                                        |
| ---------------- | ----------------------------------------------------- | ------------------------------------------------ |
| **SLI① A 서비스** | p99 응답시간 < 100ms, 에러율 < 0.1%                   | Locust로 직접 측정                               |
| **SLI② Queue**   | consumer lag < 1000, 메시지 체류시간 < 5s             | Queue 자체 메트릭 (CloudWatch, Kafka Exporter 등) |
| **SLI③ B 서비스** | 메시지 처리 p99 < 2s, 외부 API 호출 성공률 > 99.5%   | B의 애플리케이션 메트릭 (Prometheus 등)           |

### 논리

```
각 구간의 SLI가 모두 충족되면 → 전체 E2E도 자연스럽게 보장됨
어떤 구간이 SLI를 깨면 → 바로 어디가 문제인지 알 수 있음
```

E2E 측정은 "문제가 있다"는 건 알려주지만, **"어디가 문제다"**는 안 알려줘. 구간별 SLI는 둘 다 알려주니까 더 actionable.

### SLO 적용 예시

```
A 서비스의 SLI: p99 응답시간 = 85ms
A 서비스의 SLO: p99 응답시간 ≤ 100ms
→ 85ms < 100ms → SLO 충족 ✓

Queue의 SLI: 메시지 체류시간 = 12s
Queue의 SLO: 메시지 체류시간 ≤ 5s
→ 12s > 5s → SLO 위반 ✗ → B의 consumer를 스케일업 해야 함
```

### 보완 포인트

구간별 SLI로 충분하지만, "각 구간 SLI는 통과하는데 전체가 느린" 경우가 하나 있다: **큐에 메시지가 쌓여서 대기 시간이 길어지는 경우**. 이건 SLI②의 메시지 체류시간이 커버하므로, 이걸 SLI에 반드시 포함시켜야 한다.

---

## 핵심 정리

1. **SLI**는 측정값(숫자), **SLO**는 목표(기준선), **SLA**는 계약(페널티)
2. **P99**는 99%의 요청이 달성하는 응답시간 — 평균보다 사용자 경험을 정확히 반영
3. 비동기 아키텍처에서는 **E2E 대신 구간별 SLI**를 정의하고 각각 SLO를 설정하는 것이 더 실용적
4. SLI 없이 SLO는 존재할 수 없다 — 측정할 수 없으면 목표도 세울 수 없다
