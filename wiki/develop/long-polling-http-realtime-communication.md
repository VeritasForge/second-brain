# Long Polling: HTTP 실시간 통신의 이해

## 1. HTTP의 근본적 한계

HTTP는 클라이언트가 요청해야만 서버가 응답할 수 있는 구조다. 서버가 클라이언트에게 먼저 데이터를 보낼 수 없다.

```
HTTP의 규칙:
  클라이언트 -> 서버: 요청 가능
  서버 -> 클라이언트: 먼저 보내기 불가능

서버: "야 새 메시지 왔어!"  <- 이걸 할 수가 없음!
서버는 클라이언트가 물어봐야만 대답할 수 있음
```

> 비유: 식당에서 주방(서버)이 손님(클라이언트)한테 먼저 말을 걸 수 없음. 손님이 물어봐야만 대답할 수 있는 규칙.

이 한계를 극복하기 위한 여러 방식이 존재한다.

---

## 2. Short Polling과 그 한계

### 동작 방식

```
[클라이언트]                    [서버]
    |-- "새 메시지 있어?" ---->   |
    |   <-- "없음" -----------   |  즉시 응답
    |  (1초 대기)                 |
    |-- "새 메시지 있어?" ---->   |
    |   <-- "없음" -----------   |  즉시 응답
    |  (1초 대기)                 |
    |-- "새 메시지 있어?" ---->   |
    |   <-- "있어! 데이터!" ---   |  즉시 응답
```

> 비유: 1초마다 "엄마 밥 다 됐어?" "아직" "밥 다 됐어?" "아직" ... 반복

### Short Polling의 딜레마

```
실시간성 <-------- 트레이드오프 --------> 서버 부하
간격 짧게                                간격 길게
(0.1초 = 실시간적)                      (5초 = 부하 적음)
(서버 죽음)                             (실시간 아님)

-> 둘 다 만족시킬 수 없음!
```

## 3. Long Polling의 동작 원리

### 동작 방식

```
[클라이언트]                    [서버]
    |-- "새 메시지 있어?" ---->   |
    |                            |  데이터 없음...
    |   (서버가 응답을 안 보냄)    |  기다리는 중...
    |   (클라이언트도 기다림)       |  기다리는 중...
    |                            |  새 데이터 발생!
    |   <-- "있어! 데이터!" ---   |  이때서야 응답!
    |                            |
    |-- "새 메시지 있어?" ---->   |  <- 즉시 다음 요청
    |                            |  기다리는 중...
```

> 비유: "엄마 밥 다 되면 불러줘" -> 엄마가 다 되면 "다 됐어!" -> "OK, 다음에도 불러줘"

### 시간축 비교

```
시간 -->

Short Polling:
요청->응답 요청->응답 요청->응답 요청->응답 요청->응답(데이터!)
  빈응답     빈응답     빈응답     빈응답     데이터
  
Long Polling:
요청-------------------------------------->응답(데이터!)
     서버가 데이터 생길 때까지 홀딩     

-> 불필요한 요청/응답이 없음!
```

---

## 4. Long Polling의 진짜 존재 이유

Long Polling의 목적은 핸드셰이크 오버헤드 절감이 **아니다**. HTTP/1.1에서는 Keep-Alive가 기본이므로 Connection Pool을 사용하면 TCP/TLS 핸드셰이크 비용은 이미 해결된다.

Long Polling의 진짜 목적은 **"서버에서 새 데이터가 생긴 걸 클라이언트가 즉시 알 수 있게 하는 것"** -- 즉 **실시간성**이다.

### Keep-Alive + Short Polling으로 해결 안 되는 것

```
Keep-Alive + Short Polling (1초 간격):

[클라이언트]              [서버]
    |-- GET /messages -->  |  "없음" (즉시 응답)
    |<-- 200 [] ---------- |
    |  (1초 대기)           |
    |-- GET /messages -->  |  "없음" (즉시 응답)
    |<-- 200 [] ---------- |
    |                      |  <- 0.3초 후 새 메시지 도착!
    |  (1초 대기)           |  <- 클라이언트는 아직 모름! 0.7초 동안!
    |-- GET /messages -->  |  "있음!"
    |<-- 200 [msg] ------- |

TCP 비용: Keep-Alive로 해결됨
서버 처리 비용: 매초 쿼리 실행 (빈 결과여도!)
실시간성: 최대 1초 지연
```

```
Long Polling:

[클라이언트]              [서버]
    |-- GET /messages -->  |
    |   (서버 홀딩 중...)    |
    |                      |  <- 새 메시지 도착!
    |<-- 200 [msg] ------- |  <- 즉시 응답! (지연 약 0)
    |-- GET /messages -->  |  <- 즉시 다음 요청
    |   (서버 홀딩 중...)    |

TCP 비용: 어차피 하나의 요청이 오래 유지
서버 처리 비용: 데이터 있을 때만 쿼리
실시간성: 거의 0초 지연
```

| 항목                     | Keep-Alive + Short Polling | Long Polling     |
| ------------------------ | -------------------------- | ---------------- |
| TCP 핸드셰이크           | 없음                       | 없음             |
| 빈 요청/응답             | 매 간격마다 발생           | 없음             |
| 서버 쿼리 비용           | 매번 DB 조회               | 이벤트 기반      |
| **실시간성**             | **최대 N초 지연**          | **거의 0초**     |

## 5. Long Polling과 Keep-Alive의 관계

Long Polling은 Keep-Alive와는 다른 메커니즘이다. 둘 다 "연결을 오래 유지한다"는 점에서 비슷해 보이지만, 작동 레벨과 목적이 완전히 다르다.

| 항목       | Keep-Alive                       | Long Polling                         |
| ---------- | -------------------------------- | ------------------------------------ |
| 레벨       | TCP 연결 관리                    | HTTP 요청/응답 패턴                  |
| 목적       | 연결 재사용 (성능 최적화)        | 실시간 데이터 수신 (기능)            |
| 누가       | urllib3, 브라우저가 자동 처리    | 개발자가 의도적으로 설계             |
| 언제       | 응답 후에도 TCP를 안 끊음        | 응답 자체를 늦게 보냄                |

```
Keep-Alive 중인 idle 연결:
  클라이언트 ================ 서버
              아무 데이터 없음
              그냥 TCP가 열려있을 뿐

Long Polling 중인 연결:
  클라이언트 --GET /events--> 서버
              서버가 요청을 "처리 중"
              데이터 생기면 응답 보낼 예정
```

Long Polling은 Keep-Alive를 사용하지 않아도 동작한다. 다만 Long Polling + Keep-Alive 조합이면, Long Poll 응답 후 다음 요청 시 TCP 연결을 재사용할 수 있어 더 효율적이다.

```
             Keep-Alive OFF        Keep-Alive ON
            +------------------+------------------+
 Short      | 매번 새 연결 +    | 연결 재사용 +     |
 Polling    | 빈 응답 반복      | 빈 응답 반복      |
            +------------------+------------------+
 Long       | 매번 새 연결 +    | 연결 재사용 +     |
 Polling    | 응답 홀딩         | 응답 홀딩         |
            +------------------+------------------+
```

## 6. Long Polling의 단점

Long Polling은 요청을 자주 보내지는 않지만, 서버가 연결을 물고 있으므로 서버 리소스를 점유한다. 이는 가용성 저하로 이어질 수 있다.

```
서버 리소스 점유:

Short Polling:                    Long Polling:
요청 -> 즉시응답 -> 연결 반환     요청 -> 30초간 물고 있음 -> 응답
     (0.01초 점유)                      (30초 점유!)

클라이언트 1000명이면:
Short: 순간 동시 연결 = 수십 개    Long: 동시 연결 = 1000개!
```

| Long Polling 단점        | 설명                                          |
| ------------------------ | --------------------------------------------- |
| **동시 연결 수**         | 클라이언트 수 = 물고 있는 연결 수             |
| **서버 메모리**          | 연결마다 메모리 점유                          |
| **Worker 고갈**          | 동기 서버(Gunicorn sync)면 worker가 다 물림   |
| **로드밸런서 부담**      | 오래 열린 연결 관리                           |

## 7. Long Polling 서버 구현 예시

```python
import asyncio
from fastapi import FastAPI

app = FastAPI()

new_message_event = asyncio.Event()
latest_message = None

@app.get("/poll")
async def long_poll():
    """클라이언트가 호출하면, 새 메시지가 올 때까지 응답을 안 보냄"""
    await asyncio.wait_for(
        new_message_event.wait(),
        timeout=30  # 최대 30초 대기 후 빈 응답
    )
    new_message_event.clear()
    return {"message": latest_message}


@app.post("/send")
async def send_message(msg: str):
    """누군가 메시지를 보내면, 대기 중인 long poll에 알림"""
    global latest_message
    latest_message = msg
    new_message_event.set()  # 대기 중인 /poll 요청을 깨움!
    return {"status": "sent"}
```

```
흐름:
1. 클라이언트 A: GET /poll -> 서버에서 대기 중... (응답 안 옴)
2. 클라이언트 B: POST /send "안녕!" -> event.set()
3. 서버: 대기 중이던 A의 요청에 {"message": "안녕!"} 응답!
4. 클라이언트 A: 응답 받음 -> 즉시 다시 GET /poll -> 대기...
```

## 8. 실시간 통신 방식 전체 비교

```
진화 순서:

Short Polling -> Long Polling -> SSE -> WebSocket
   (원시적)       (개선)      (단방향)   (양방향)
```

| 방식              | 연결                          | 방향              | 실시간성           | 서버 부담          | 사용 예              |
| ----------------- | ----------------------------- | ----------------- | ------------------ | ------------------ | -------------------- |
| **Short Polling** | 매번 새로 (또는 Keep-Alive)   | 클->서 반복       | 낮음 (간격 의존)   | 높음 (빈 요청)     | 단순 상태 확인       |
| **Long Polling**  | 요청마다 홀딩                 | 클->서 (홀딩)     | 중간               | 중간 (홀딩 비용)   | 채팅, 알림           |
| **SSE**           | 1개 연결 유지                 | 서->클 단방향     | 높음               | 낮음               | 실시간 피드, AI 스트리밍 |
| **WebSocket**     | 1개 연결 유지                 | 양방향            | 매우 높음          | 낮음               | 게임, 실시간 협업    |

> 참고: ChatGPT/Claude가 답변을 한 글자씩 보여주는 것은 **SSE (Server-Sent Events)** 를 사용한다.

### 언제 뭘 쓰나

```
"서버 -> 클라이언트 실시간 전달"이 필요한가?

  아니오 -> Short Polling + Keep-Alive 충분

  예 -> 어떤 환경?
    +-- WebSocket 사용 가능 -> WebSocket (양방향, 가장 효율적)
    +-- 서버->클라이언트 단방향만 -> SSE (ChatGPT 스트리밍 같은)
    +-- 둘 다 안 되는 환경 -> Long Polling (레거시 호환)
```

### 역사적 흐름

```
2005~2010: Long Polling 전성기 (Gmail, Facebook 채팅)
           -> WebSocket이 없던 시절의 최선의 방법

2011~현재: WebSocket 표준화 (RFC 6455)
           -> Long Polling은 폴백(fallback)으로 격하

현재:      SSE + WebSocket이 주류
           Long Polling은 레거시 호환용
```

## 핵심 정리

1. **Long Polling의 존재 이유**는 핸드셰이크 비용 절감이 아니라, HTTP의 "서버가 먼저 말할 수 없는" 한계를 극복하여 **실시간성**을 확보하는 것이다.
2. **Keep-Alive + Short Polling**으로는 TCP 비용은 해결되지만, 빈 요청의 서버 쿼리 비용과 polling 간격에 의한 실시간성 한계는 해결되지 않는다.
3. **Long Polling은 Keep-Alive와 독립적인 개념**이다. Keep-Alive는 TCP 연결 재사용(전송 계층), Long Polling은 응답 지연 패턴(애플리케이션 계층)이다.
4. **Long Polling의 단점**: 연결을 물고 있어 서버 리소스를 점유하며, 클라이언트 수 = 동시 연결 수로 확장성이 제한된다.
5. **현재는 WebSocket과 SSE가 주류**이며, Long Polling은 WebSocket이 없던 시절의 차선책으로 레거시 호환용으로 남아있다.
