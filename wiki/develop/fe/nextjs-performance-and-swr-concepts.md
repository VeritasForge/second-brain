# Next.js 페이지가 느린 이유와 핵심 개념 정리

> Next.js App Router(React 19) 환경에서 `"use client"`, 마운트 후 fetch waterfall, SWR, Server Component의 동작 원리와 상호관계를 정리한 문서.

---

## 1. 핵심 개념 3가지 — 왜 페이지가 느린가?

### 1.1 `"use client"` — 누가 화면을 만드는가? 👨‍🍳

**🍔 비유: 햄버거 가게 두 가지 스타일**

```
[Server Component (서버 컴포넌트)] = "완성 햄버거"
  주문 → 주방에서 완성 → 손님은 받자마자 바로 먹음  ⚡

[Client Component = "use client"]   = "셀프 조립 키트"
  주문 → 빈 그릇 + 재료 봉지만 받음
       → 손님이 직접 펴고, 익히고, 조립
       → 그제야 먹을 수 있음                          🐢
```

**실제로 일어나는 일:**

| 방식             | 서버가 보내는 것          | 브라우저가 하는 일                                       |
| ---------------- | ------------------------- | -------------------------------------------------------- |
| Server Component | 완성된 HTML (데이터 포함) | 그냥 화면에 표시                                         |
| `"use client"`   | JavaScript 코드 + 빈 HTML | JS 다운로드 → 실행 → fetch → 데이터 받기 → 화면 그리기 |

### 1.2 "마운트 후 fetch Waterfall" 🌊

**용어 분해:**

- **마운트(mount)** = 컴포넌트가 화면에 처음 그려지는 순간 (= "셀프 조립 완료된 시점")
- **fetch** = 서버에 데이터 요청 (예: `/api/reviews` 호출)
- **Waterfall(폭포)** = 한 단계가 끝나야 다음이 시작되는 순차 흐름

**"통계" 메뉴 클릭 시 일어나는 일 — 시간 순서:**

```
시간       단계                                          사용자가 보는 것
─────────────────────────────────────────────────────────────────
0ms       👆 "통계" 클릭
  ↓
50ms      📨 HTML 도착 (거의 비어있음)                   빈 화면
  ↓
300ms     📦 JavaScript 번들 다운로드                    빈 화면
  ↓
500ms     ⚙️  React 실행 + 컴포넌트 마운트              "로딩 중..."
  ↓
501ms     🔍 useEffect 발동 → /api/statistics fetch 시작 "로딩 중..."
  ↓
1000ms    📊 응답 도착 (DB 쿼리가 무거워 500ms 소요)     "로딩 중..."
  ↓
1050ms    🎨 화면에 차트/숫자 그리기                     ✅ 통계 보임!
```

→ **총 ~1초** 동안 사용자는 빈 화면 또는 "로딩 중..." 만 봅니다.

**🆚 만약 Server Component였다면:**

```
0ms       👆 "통계" 클릭
  ↓
500ms     서버가 DB 조회 + 데이터 박힌 HTML 생성 후 전송
          ✅ 즉시 통계 보임!
```

같은 작업인데 사용자 체감이 **2배 빨라집니다**. JS 다운로드/실행/fetch 단계가 통째로 사라지기 때문.

### 1.3 "SWR 미사용" — 캐싱이 없어 매번 새로 요청 📦

**SWR (Stale-While-Revalidate)** = "오래된 거 일단 보여주고, 백그라운드에서 새 거 확인"이라는 뜻의 캐싱 라이브러리.

**☕ 비유: 단골 카페**

```
[캐싱 없음 — 현재 상태]
첫 방문:  "아메리카노 주세요" → 주문서 작성 → 원두 갈기 → 만들기 → 받기 (5분)
재방문:   "아메리카노 주세요" → 주문서 작성 → 원두 갈기 → 만들기 → 받기 (5분)
재방문:   "아메리카노 주세요" → 주문서 작성 → 원두 갈기 → 만들기 → 받기 (5분)
😵 매번 처음부터 다시!

[SWR 캐싱 — 개선 후]
첫 방문:  주문 → 만들기 → 받기 (5분)  + 카운터에 메모 📝 "이 손님 = 아메리카노"
재방문:   "늘 마시던 거" → 즉시 제공! (10초)
                          + 백그라운드에서 "오늘 원두 신선한가?" 확인
재방문:   즉시 제공! (10초)
⚡⚡⚡ 두 번째부터는 거의 즉시
```

### 1.4 세 가지 문제가 합쳐지면

```
사용자: "통계 메뉴 클릭"
  │
  ├─ [문제①] "use client"라서 빈 페이지 + JS부터 다운로드해야 함  (~300ms)
  ├─ [문제②] 마운트 끝나야 fetch 시작 (폭포처럼 순차)              (~100ms)
  ├─ [문제③-a] /api/statistics 가 무거움
  │            (streak 30일 = DB를 30번 왕복!)                     (~500ms)
  ├─ [문제③-b] 캐시 없어 다음에 다시 와도 동일 비용                (재방문도 1초)
  ↓
  총 ~1000ms 대기, 데이터가 늘면 점점 더 느려짐 📈
```

## 2. SWR 심화

### 2.1 SWR이란? — 라이브러리 + 전략 (두 가지 의미)

| 구분            | 내용                                                                                                                                                                  |
| --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 📚 **라이브러리** | [SWR (swr)](https://swr.vercel.app) — Vercel(Next.js 만든 회사)이 만든 React용 데이터 fetching 라이브러리                                                              |
| 🔧 **전략 이름**  | **S**tale-**W**hile-**R**evalidate의 약자. "오래된 캐시 일단 보여주고, 백그라운드에서 새 거 확인"이라는 캐싱 패턴                                                      |
| 🌐 **어원**       | 원래는 HTTP 캐시 헤더 표준 ([RFC 5861](https://datatracker.ietf.org/doc/html/rfc5861))에 정의된 지시어. 브라우저/CDN이 캐시를 다룰 때 쓰는 용어를 React 라이브러리 이름으로 가져온 것 |

### 2.2 SWR의 동작 사이클

```
1️⃣ 캐시에 있는 데이터를 즉시 화면에 표시  (stale일 수도 있음)
       ↓
2️⃣ 동시에 백그라운드에서 fetch 시작  (revalidate)
       ↓
3️⃣ 응답 받으면 캐시와 비교
       ↓
4️⃣ 다르면 캐시 갱신 + React가 자동 리렌더
   같으면 아무 일도 안 일어남
```

이게 바로 라이브러리 이름의 어원:

- **S**tale (오래된 거)
- **W**hile (먼저 보여주는 동안)
- **R**evalidate (뒤에서 검증)

**🥛 비유로 한 번 더 — "우유 사오는 두 가지 방식"**

**일반 HTTP 캐시 (브라우저 기본 동작)**

```
냉장고 확인
  ├─ 우유 있고 신선함  → 그거 마심 ✅
  ├─ 우유 있는데 상함  → 버리고 마트 다녀옴 (그동안 못 마심) ⏳
  └─ 우유 없음        → 마트 다녀옴 (그동안 못 마심) ⏳
```

**SWR**

```
냉장고 확인
  ├─ 우유 있음 → 일단 그거 따라 마심 ⚡
  │              └─ 동시에 누가 마트로 뜀 🏃
  │                    └─ 새 우유 도착 → 컵에 부어줌 🥛
  └─ 우유 없음 → 마트 다녀옴 (이때만 기다림)
```

**📊 시각화로 보는 차이**

```
시간 →

[일반 캐시 (stale 데이터)]
클릭 ───── 서버 요청 ───── 응답 ─── 화면
0ms                              500ms ✅

[SWR]
클릭 ─── 캐시 표시 ✅        ← 0ms부터 사용자가 봄
         ↓
         (백그라운드) ───── 응답 ───── 화면 갱신
                                   500ms (보이지 않는 작업)
```

### 2.3 SWR이 자동으로 해주는 추가 기능들

`useSWR` 한 줄을 쓰면 따라오는 기본 동작들:

| 기능                       | 동작                                                          |
| -------------------------- | ------------------------------------------------------------- |
| **dedupe (중복 제거)**     | 같은 키로 동시에 여러 컴포넌트가 요청해도 실제 fetch는 1회    |
| **revalidateOnFocus**      | 사용자가 다른 탭 보다가 돌아오면 자동으로 새 데이터 확인      |
| **revalidateOnReconnect**  | 네트워크 끊겼다 복구되면 자동 갱신                            |
| **자동 재시도**            | 실패 시 지수 백오프(exponential backoff)로 재시도             |
| **mutate (수동 갱신)**     | "지금 당장 다시 가져와!" 명령 가능 (예: 저장 직후)            |

### 2.4 `useEffect + fetch` vs `useSWR` 코드 비교

```tsx
// 😩 useEffect + fetch — 매번 새로 fetch
useEffect(() => {
  fetch("/api/statistics").then(...);
}, []);

// 😎 SWR — 캐시 우선 + 자동 재검증
const { data } = useSWR("/api/statistics", fetcher);
```

같은 키(`"/api/statistics"`)로 두 번째 호출하면 **즉시 캐시 반환** + 백그라운드에서 갱신 여부 확인. 코드는 거의 한 줄로 짧아지고 캐싱은 자동.

## 3. SWR이 어디서, 어떻게 동작하나

### 3.1 캐시는 어디 있는가? 🗄️

**결론: 브라우저 메모리 안 (JavaScript 변수)** — 서버는 전혀 관여하지 않음.

좀 더 정확히: SWR은 내부에 **JavaScript `Map` 객체** 하나를 만들어 두고 거기에 데이터를 쌓아둠. 그냥 변수.

```
[브라우저 탭이 열려있는 동안만 살아있는 메모리]

      ┌─────────────────────────────────┐
      │  SWR Cache (자바스크립트 Map)     │
      ├─────────────────────────────────┤
      │  "/api/reviews"    → { ... }   │
      │  "/api/statistics" → { ... }   │
      │  "/api/categories" → { ... }   │
      └─────────────────────────────────┘
```

**🚨 함의:**

- 페이지를 **새로고침(F5)** 하면 캐시 다 사라짐
- 탭을 **닫으면** 사라짐
- 같은 사용자라도 **다른 탭/다른 브라우저**면 캐시 공유 안 됨
- 서버는 캐시 존재 여부조차 모름 (그냥 평소처럼 API 요청 받을 뿐)

**캐시 종류 총정리**

| 레이어              | 위치                          | 영속성                | SWR과의 관계                |
| ------------------- | ----------------------------- | --------------------- | --------------------------- |
| **SWR 캐시**        | 브라우저 **메모리** (JS Map)  | 탭 닫으면 사라짐      | 이게 우리가 얘기한 것 ✅    |
| HTTP 캐시           | 브라우저 **디스크**           | 만료 시까지 영구      | 무관 (브라우저가 자체 관리) |
| Service Worker      | 브라우저 **디스크**           | 영구                  | 무관 (별도 기술)            |
| Redis/Memcached     | **서버** 메모리               | 서버 재시작 전까지    | 서버 측 캐시 (다른 얘기)    |
| CDN                 | **엣지 노드**                 | TTL 기반              | 서버 앞단 캐시              |

**🥡 비유:**

- SWR 캐시 = 책상 위에 펼쳐둔 메모 📝 (자리 뜨면 사라짐)
- HTTP 캐시 = 책장 📚 (전원 꺼져도 남아있음)
- 서버 캐시 = 사서가 외운 정보 🧑‍🏫 (모든 손님이 공유)

### 3.2 Single-Threaded인데 어떻게 Background Fetch? 🧵

**핵심 깨달음: JavaScript는 single-threaded지만, 브라우저는 multi-threaded다!**

```
┌─────────────────────────────────────────────────────────┐
│                    브라우저 (Chrome 등)                  │
│                                                          │
│  ┌──────────────────┐    ┌────────────────────────────┐ │
│  │  JS 엔진 (V8)    │    │  브라우저 백그라운드 영역    │ │
│  │  메인 스레드 1개  │    │  (C++로 구현)              │ │
│  │  ← 여기만 1개!   │    │                            │ │
│  │                  │    │  • 네트워크 스레드 (fetch)  │ │
│  │  내 코드 실행     │    │  • 타이머 스레드           │ │
│  │                  │    │  • 렌더링 스레드           │ │
│  │                  │    │  • Disk I/O 스레드 ...     │ │
│  └──────────────────┘    └────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

내가 작성하는 **JavaScript 코드는 1개 스레드에서만 돌지만**, `fetch()` 같은 함수는 사실 **"브라우저에게 일을 시키는 부탁"** 일 뿐. 진짜 네트워크 작업은 브라우저(C++)가 별도 스레드에서 처리.

**🍕 비유: 1인 피자집**

```
주인공: 메인 셰프 한 명 (= JavaScript 메인 스레드)
도구:   오븐, 반죽기 (= 브라우저 백그라운드)

[혼자서 어떻게 여러 피자를 동시에 만들까?]

1. 셰프가 도우 1번을 만들어 오븐에 넣음 → 오븐이 굽는 중 🔥
2. 셰프는 안 기다리고 즉시 도우 2번 만들기 시작 ✍️
3. "띵!" 오븐이 알림 → 셰프가 "할 일 목록"에 "1번 꺼내기" 추가
4. 셰프가 지금 일 끝나면 → "1번 꺼내기" 실행
```

여기서:

- **오븐 = 브라우저의 네트워크 스레드** (실제 일을 함)
- **"띵!" 소리 = Callback** (응답 도착 알림)
- **"할 일 목록" = Task Queue**
- **셰프가 일 끝났는지 확인하는 행동 = Event Loop**

**실제 코드 흐름을 시간순으로**

```
[JavaScript 메인 스레드]              [브라우저 백그라운드]
─────────────────────────────────────────────────────────
0ms   fetch("/api/x") 호출   ────→   네트워크 스레드: "맡았다!"
                                     (서버에 요청 전송 중...)
0ms   console.log("다음 줄")
0ms   다른 코드 계속 실행            (서버 응답 기다리는 중...)
0ms   사용자 클릭 처리                    ↓
50ms  렌더링                              ↓
...                                        ↓
300ms (한가해짐)                     응답 도착!
                                     "Task Queue에 콜백 추가"
                                            ↓
301ms ┌─ Event Loop가         ◄──────────┘
      │  Task Queue에서 콜백 꺼냄
      ↓
302ms .then(data => setState(data)) 실행
```

핵심은: **JS는 한 줄도 멈춰서 기다린 적이 없음.** 그냥 부탁만 하고 다음 일 한 것. 이걸 **"non-blocking I/O"** 라고 부름.

**`async/await`도 결국 같은 메커니즘**

```js
const data = await fetch("/api/x");
//           ^^^^^ 여기서 "멈추는 것처럼 보이지만",
//                 실제로는 함수를 잠시 일시정지하고
//                 메인 스레드는 다른 일을 함.
//                 응답 오면 이어서 실행.
```

`await`는 "스레드를 막는 것"이 아니라 **"이 함수만 일시정지하고 메인 스레드는 풀어준다"** 는 뜻. syntax sugar일 뿐.

**🎁 보너스 — 진짜 멀티스레드를 쓰고 싶다면?**

| 기술                  | 용도                                          |
| --------------------- | --------------------------------------------- |
| **Web Worker**        | CPU 무거운 작업 (이미지 처리, 큰 계산) 분리   |
| **Service Worker**    | 백그라운드 동기화, 오프라인 캐싱 (PWA)        |
| **SharedArrayBuffer** | 워커 간 메모리 공유                           |

다만 SWR은 이런 거 안 씀. 그냥 표준 `fetch + 이벤트 루프` 로 충분.

## 4. Server Component와 SWR의 관계

> 핵심 질문: "Server Component를 쓰면 SWR이 필요없는 거 아니야?"
> 답: **절반은 맞고, 절반은 아님.**

### 4.1 Server Component만으로 충분한 경우 — "첫 페이지 로드"

처음 페이지에 진입할 때는 100% 충분.

```
사용자: "통계" 클릭
       ↓
Server Component:
  1. 서버에서 DB 조회
  2. 결과를 HTML에 박아서 전송
       ↓
브라우저: 받자마자 화면 표시 ✅
```

이 시점에서 SWR은 **할 일이 없음**. 데이터는 이미 화면에 있고, fetch도 필요 없음.

### 4.2 그럼에도 SWR(또는 클라이언트 캐싱)이 필요한 5가지 상황

문제는 **"첫 로드 이후"**. 페이지는 살아있는 동안 계속 사용자와 상호작용.

#### 상황 1: 데이터를 사용자가 바꿨을 때 🔄

```
사용자가 "완료" 버튼 누름
  ↓
서버: DB 업데이트
  ↓
❓ 화면의 숫자/리스트는 어떻게 갱신하지?
```

Server Component는 한 번 그려진 뒤로는 다시 안 그려짐. 누군가 갱신 신호를 줘야 함.

| 방법                          | 동작                                          |
| ----------------------------- | --------------------------------------------- |
| `router.refresh()` (Next.js)  | 페이지 전체를 서버에서 다시 가져와 리렌더     |
| `useSWR + mutate()`           | 해당 데이터만 다시 fetch + 캐시 갱신          |

#### 상황 2: 같은 데이터를 여러 컴포넌트가 쓸 때 👯

```
대시보드 한 페이지에:
  ├─ 상단 헤더: "오늘 완료 5개" 표시
  ├─ 가운데 카드: 완료된 항목 리스트
  └─ 사이드: 미니 차트

세 곳 다 /api/reviews 데이터 필요 → fetch 3번?
```

Server Component는 prop으로 내려보내면 해결되지만, **클라이언트에서 다시 fetch가 필요한 시점이 오면** SWR의 dedupe(중복 제거)가 빛을 발함. 같은 키로 부르면 한 번만 요청.

#### 상황 3: 백그라운드 자동 갱신이 필요할 때 🔁

- 사용자가 다른 탭 갔다가 돌아왔을 때 → 신선한 데이터로 갱신
- 네트워크 끊겼다가 복구됐을 때 → 자동 재검증
- 5분마다 자동 폴링

Server Component만으로는 이런 자동 갱신이 어려움. **새로고침(F5)을 누르기 전까지는 그대로**.

#### 상황 4: 낙관적 UI(Optimistic Update) 🎭

```
"좋아요" 버튼 누름
  ↓
[방법 A — Server 왕복]
  서버 요청 → 응답 대기 (300ms) → 숫자 +1 표시
  사용자는 300ms 동안 변화 없음 🥱

[방법 B — Optimistic UI]
  즉시 화면에 +1 표시 ⚡
  동시에 백그라운드로 서버 요청
  실패하면 되돌리기
```

방법 B를 하려면 **클라이언트 측 캐시를 즉시 수정할 수 있어야** 함. Server Component는 서버 다녀와야 하니 불가능. SWR이나 React Query가 잘하는 영역.

#### 상황 5: 페이지 간 이동 후 재방문 ↩️

Next.js App Router에 **Router Cache**라는 게 있어서 어느 정도 해결되긴 함. 하지만:

- 데이터의 **신선도**(언제 갱신할지)를 세밀하게 제어하기 어려움
- 캐시 시간을 5초로 줄여놓으면 그 이상 머무르면 또 서버 왕복

SWR이라면 "포커스 시 재검증", "30초마다 폴링" 같은 정책을 컴포넌트별로 다르게 줄 수 있음.

### 4.3 비유로 정리: 식당 + 미니바

```
[Server Component]
  = 셰프가 직접 만들어 가져다주는 코스 요리 🍽️
  → 처음 식탁에 차려진 음식은 완벽함

[그 후 일어나는 일들]
  • 음료 한 잔 더?  → 셰프 다시 부르기 (router.refresh) 🔔
  • 옆 친구도 같은 음료 → 두 번 만들기? (중복 fetch)
  • 5분 후 얼음 녹음 → 자동으로 새 음료? (background revalidate)

[SWR/클라이언트 캐시]
  = 테이블 옆에 둔 미니바 🍷
  → 셰프 안 부르고도 즉시 마실 수 있음
  → 떨어지면 자동으로 채워짐 (백그라운드 fetch)
```

**Server Component**는 "**첫 상차림**" 을 잘하고,
**SWR**은 "**식사 중 운영**" 을 잘함. 둘은 경쟁 관계가 아니라 **역할 분담**.

### 4.4 매트릭스로 한눈에 보기

| 시나리오                       | Server Component  | SWR    |
| ------------------------------ | :---------------: | :----: |
| 첫 페이지 로드                 | ✅ 최강           | ❌ 불필요 |
| 페이지 내 인터랙션             | ❌ 불가능         | ✅ 강력 |
| 같은 데이터 여러 곳 사용       | 🟡 prop 전달 가능 | ✅ 자동 dedupe |
| 백그라운드 자동 갱신           | ❌ 어려움         | ✅ 기본 제공 |
| Optimistic UI                  | ❌ 불가능         | ✅ 기본 제공 |
| 인증 정보로 보호된 데이터      | ✅ 안전 (서버 검증) | 🟡 가능하나 주의 |
| SEO/초기 표시 속도             | ✅ 최강           | ❌ 불리 |

### 4.5 실전 패턴 두 가지

#### Pattern A — Server-First (Next.js 공식 권장)

```
Server Component로 첫 로드
       ↓
변경 시 Server Action 호출
       ↓
revalidatePath() 또는 router.refresh()
       ↓
서버가 다시 렌더해서 내려줌
```

장점: 단순함, 항상 서버가 진실의 원천
단점: 매 변경마다 서버 왕복, optimistic UI 어려움

#### Pattern B — Hybrid (현실에서 자주 쓰임)

```
Server Component로 첫 로드 (초기 HTML에 데이터 박힘)
       ↓
클라이언트는 SWR로 같은 키 사용 (initialData 전달)
       ↓
이후 인터랙션은 SWR이 처리
```

장점: 초기 빠름 + 이후 인터랙션도 매끄러움
단점: 약간 더 복잡

### 4.6 한 줄 요약

> Server Component는 **"문서를 인쇄해서 주는 것"**,
> SWR은 **"손에 든 노트에 계속 받아 적는 것"**.
> 정적인 정보는 인쇄가 빠르고, 살아있는 정보는 노트가 편하다.

## 5. 처방 매트릭스 — 어떤 도구가 어떤 문제를 고치는가

| 처방                                                    | 고치는 문제                            | 효과                                                 |
| ------------------------------------------------------- | -------------------------------------- | ---------------------------------------------------- |
| **Streak SQL 단일 쿼리화** (예: `async-parallel`)       | API 내부 N+1 쿼리                      | API 응답 ~30배 빨라짐                                |
| **SWR 도입** (`client-swr-dedup`)                       | 캐시 부재 + useEffect waterfall        | 재방문 시 거의 즉시                                  |
| **Server Component 전환** (`server-parallel-fetching`)  | "use client" + 마운트 후 fetch         | 첫 방문도 즉시                                       |
| **Suspense 경계** (`async-suspense-boundaries`)         | 빈 화면 → "로딩 중..." 대기            | 레이아웃 즉시 표시 + 데이터 부분만 스트림            |

## 부록 — Next.js 성능 룰북 (Vercel 공식)

Vercel(Next.js 만든 회사)이 공식 배포하는 `vercel-react-best-practices` 스킬에는 **69개 룰이 8개 카테고리**로 정리되어 있다.

### 카테고리 개요

```
1. async-*     (CRITICAL)   Waterfall 제거, Promise.all, Suspense
2. bundle-*    (CRITICAL)   번들 크기, dynamic import, barrel imports
3. server-*    (HIGH)       RSC 캐싱, 병렬 fetch, 직렬화
4. client-*    (MED-HIGH)   SWR dedup, 이벤트 리스너
5. rerender-*  (MEDIUM)     리렌더 최적화, memo, transitions
6. rendering-* (MEDIUM)     SVG, hydration, content-visibility
7. js-*        (LOW-MED)    JS 미세 최적화
8. advanced-*  (LOW)        고급 패턴
```

### 본 문서에 등장한 개념 ↔ 룰 매핑

| 개념                            | 해당 Vercel 룰                                                       | 등급         |
| ------------------------------- | -------------------------------------------------------------------- | ------------ |
| `useEffect + fetch` 패턴        | `client-swr-dedup` "Use SWR for Automatic Deduplication"             | MEDIUM-HIGH  |
| `"use client"` 남용             | `server-parallel-fetching` (Server Component + 컴포지션)             | CRITICAL     |
| API 내부 순차 await (N+1)       | `async-parallel` "Use Promise.all() for independent operations"      | CRITICAL     |
| 로딩 중 빈 화면                 | `async-suspense-boundaries` "Faster initial paint"                   | HIGH         |
| 무거운 컴포넌트 항상 번들 포함  | `bundle-dynamic-imports`                                             | CRITICAL     |

### `client-swr-dedup` 룰의 "Incorrect 예시" (원문)

```tsx
function UserList() {
  const [users, setUsers] = useState([])
  useEffect(() => {
    fetch('/api/users')
      .then(r => r.json())
      .then(setUsers)
  }, [])
}
```

→ 룰북이 "이러지 마세요" 라고 콕 집어 말한 패턴.

### `client-swr-dedup` 룰의 "Correct 예시" (원문)

```tsx
import useSWR from 'swr'

function UserList() {
  const { data: users } = useSWR('/api/users', fetcher)
}
```
